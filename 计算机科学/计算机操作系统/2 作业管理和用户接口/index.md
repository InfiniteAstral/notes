# 第 2 章 作业管理和用户接口

第 1 章已经回答了“操作系统是什么、做什么、有哪些共性特征”。从本章开始，我们把视角进一步拉近，去看一个更贴近“使用过程”的问题：

- 用户究竟是怎样和操作系统打交道的
- 一项计算任务进入系统后，系统怎样把它当成“作业”来管理
- 应用程序为什么不能直接操作硬件，而要通过系统调用
- 图形用户界面为什么能让操作系统更容易使用

如果说第 1 章建立的是“全景图”，那么第 2 章关注的就是“入口”和“控制方式”。学完这一章后，你应当能把“命令行”“Shell”“JCL”“系统调用”“用户态/核心态”“GUI”这些概念串成一条完整链路。

::: info 学习提示

本章最容易混淆的，是三个“接口层次”：

- **作业级接口**：用户如何控制一个作业从提交到运行结束
- **命令级接口**：用户如何通过命令与系统交互
- **程序级接口**：程序如何通过系统调用请求操作系统服务

看起来它们都像“接口”，但服务对象和使用方式并不相同。

:::

## 2.1 用户与操作系统的接口

操作系统并不只是“在后台管理资源”，它还必须向用户和程序提供使用系统的入口。否则，系统即使功能再强，也难以真正被使用。

用户与操作系统的接口主要可以分为两大类：

- **作业级接口**
- **程序级接口**

其中，作业级接口又可以继续分为脱机和联机两类。

### 接口总览

#### 作业级接口

**作业级接口** 是操作系统为了让用户能够控制作业运行全过程而提供的功能。

这里的“全过程控制”包括：

- 如何提交作业
- 如何说明作业要做什么
- 如何指定所需资源
- 如何决定作业完成或出错后的处理方式

作业级接口按使用方式，通常可分为两类：

- **脱机用户接口**：主要对应批处理方式
- **联机用户接口**：主要对应交互式方式

#### 程序级接口

**程序级接口** 是操作系统为程序一级提供服务而设置的接口，通常由一组 **系统调用** 构成。

它回答的问题是：

**应用程序在运行过程中，怎样请求操作系统代为完成文件访问、设备控制、进程管理、内存申请等操作？**

这类接口主要面向程序员，而不是普通终端用户。

::: tip

可以先这样区分：

- 想“控制一个作业怎么运行”，看作业级接口
- 想“在终端里输入命令”，看命令级接口
- 想“让程序请求系统服务”，看程序级接口

这三者并不是互相替代，而是面向不同使用场景。

:::

### 作业级接口

#### 脱机用户接口

脱机用户接口主要用于 **批处理系统**。

其特点是：

- 用户不需要一直守在终端前
- 用户事先把程序、数据、控制要求一起提交给系统
- 系统按说明自动执行作业

这种方式适合：

- 大批量、重复性强的任务
- 不要求即时交互的计算任务
- 需要系统自动顺序处理的作业集合

#### 联机用户接口

联机用户接口主要用于 **交互式系统**。

其特点是：

- 用户通过终端与系统实时对话
- 用户发出命令后可立即看到系统反馈
- 用户可以根据反馈继续输入下一步命令

这种方式适合：

- 开发、调试程序
- 日常文件和目录操作
- 需要频繁试错与调整的任务

### 命令级接口

联机用户接口中的核心形式之一是 **命令级接口（Command Line Interface, CLI）**，同时还存在图形化方向的接口。

#### 命令级接口的本质

命令级接口的基本工作方式是：

1. 用户输入命令
2. 系统接收命令
3. 命令处理程序解释命令含义
4. 操作系统或相关程序执行命令
5. 系统把结果返回给用户

这种方式的优点是：

- 控制灵活
- 表达能力强
- 适合组合操作
- 便于脚本化和自动化

缺点则是：

- 命令多、格式复杂
- 初学者需要记忆较多语法
- 使用门槛高于图形界面

#### 命令驱动方式

在交互式系统中，最经典的一种联机控制方式就是 **命令驱动方式**。

用户通过控制台终端输入操作系统提供的命令来控制作业运行，系统在解释并执行命令后，再通过屏幕把结果反馈给用户。

这种方式的基本任务包括：

- **解释命令**
- **传送命令以便执行**
- **接受系统信息并反馈给用户**

#### 命令驱动方式需要哪些组成部分

命令接口通常包括：

- **一组联机命令**
- **终端处理程序**
- **命令解释程序**

它们可以分别理解为：

- **联机命令**：用户可以输入的指令集合
- **终端处理程序**：负责管理用户和终端设备之间的交互
- **命令解释程序**：负责分析命令含义，并启动相应处理过程

### 终端处理程序与命令解释程序

用户与操作系统的通信，通常要借助某种输入输出装置，例如键盘和显示器。为了使终端交互顺利进行，系统需要专门的终端处理程序。

#### 终端处理程序要处理什么

有几个典型问题需要处理：

- **回送显示（Echoing）**：用户输入的字符是否立即回显到屏幕上
- **提前打入（Type ahead）**：系统忙时，用户是否可以先输入后续字符并暂存
- **字符变换**：输入输出过程中字符是否需要特殊转换
- **中断字符（Break Characters）**：遇到特殊字符时，是否打断当前处理流程

这些问题看起来很细碎，但它们直接影响用户的交互体验。

#### 命令解释程序的两种处理方式

命令解释程序通常有两种处理方式：

- **由终端命令解释程序直接处理**
- **由子进程代为处理**

这两种方式的区别，本质上是在于：

- 命令解释器自己是否直接完成命令处理
- 还是把具体命令交给新的执行实体去完成

后者在现代系统中更加常见，因为它更利于保持命令解释器本身的稳定与独立。

### 命令接口的结构

命令接口并不是一条简单的“读取输入 -> 执行命令”的直线过程，而是一个可组织、可替换、可扩展的系统模块。

这里中给出的几种实现思路包括：

- 作为操作系统的一部分存在
- 作为操作系统的独立模块存在
- 作为容易被替换的普通程序存在

这说明命令接口并不是绝对固定的。

例如在 UNIX 类系统中：

- 不同用户可以使用不同的 Shell
- 同一操作系统可以同时支持多种命令接口风格

这正体现了操作系统接口层的灵活性。

### 命令语言

命令级接口之所以能表达丰富操作，是因为它背后有一套 **命令语言**。

#### 命令语言是什么

命令语言规定了由操作系统执行的一系列操作，以及命令应如何书写、如何带参数、如何组合。

多数命令行中的一条命令，本质上都可以看成：

**带有参数的一次过程调用。**

#### 命令语言通常包含哪些命令

常见的典型命令可分为五类：

1. **系统访问命令**
2. **文件管理命令**
3. **编辑、编译和执行命令**
4. **询问命令**
5. **操作员专用命令**

此外，命令语言还涉及：

- 命令行的格式
- 命令参数的组织方式
- 命令分类
- 命令简化方法

### 图形用户界面与命令行的关系

除了命令行方式外，还有 **图形用户接口（GUI）**。

两者可以先作如下对比：

| 对比维度   | 命令行方式           | 图形用户界面                         |
| ---------- | -------------------- | ------------------------------------ |
| 使用方式   | 依赖输入命令         | 依赖图标、窗口、菜单、鼠标           |
| 学习成本   | 需要记忆命令和格式   | 更直观，记忆负担较轻                 |
| 灵活性     | 高，适合复杂组合操作 | 易学易用，但复杂控制常不如命令行方便 |
| 自动化能力 | 强，便于脚本化       | 一般较弱                             |
| 效率       | 熟练后很高           | 对大量重复操作不一定更快             |

这里还特别指出：

**图形用户界面虽然更友好，但效率未必总比命令行更高。**

这提醒我们：界面是否“先进”，不能只看视觉效果，还要看使用场景。

## 2.2 作业管理

作业管理是本章的核心内容之一。它回答的是：

**一个用户提交的任务，在操作系统眼里到底是什么？系统又如何对它进行描述、控制和组织？**

### 基本概念

#### 什么是作业

这里中给出的定义是：

**作业（Job）** 是用户在一次计算过程或一次事务处理过程中，要求计算机系统所做工作的总称。

可以把“作业”理解为：

- 从用户角度出发的一项完整任务
- 这项任务包含程序、数据以及运行要求
- 系统需要把它当作一个整体进行管理

例如：

- 一次批量编译任务
- 一次数据处理任务
- 一次科学计算任务

都可以看作作业。

#### 什么是作业步

一个作业往往不是单一步骤完成的，因此这里引入了 **作业步（Job Step）** 的概念。

**作业步** 指处理作业时相对独立的子任务。

例如，一个作业可以包括：

- 编译源程序
- 链接目标文件
- 执行可执行程序
- 输出处理结果

这些都可以看成不同的作业步。

#### 什么是作业流

**作业流** 指把若干作业依次放在输入设备上，并在操作系统控制下依次运行所形成的作业序列。

作业流这个概念在批处理系统中尤其重要，因为批处理的基本思想就是：

- 先形成一批作业
- 再由系统自动顺序处理

### 作业由哪些部分组成

可以看到，一个作业通常由三部分组成：

- **程序**
- **数据**
- **作业说明书**

这三部分各自承担不同作用：

- **程序**：告诉系统“要做什么处理”
- **数据**：给出“处理对象是什么”
- **作业说明书**：告诉系统“怎样组织和控制这项处理”

::: info

很多初学者会把“程序”和“作业”混为一谈。其实程序只是作业的一部分。

一个作业是更完整的概念，它不仅包括程序本身，还包括数据以及控制要求。

:::

### 作业处理过程与作业类型

可以把作业类型与用户接口方式联系起来：

- **脱机作业 -> 脱机用户接口 -> 批处理方式**
- **联机作业 -> 联机用户接口 -> 交互式方式**

这意味着：

- 不是所有作业都必须由用户实时参与控制
- 也不是所有任务都适合完全脱机批处理

系统需要根据任务性质提供不同的管理方式。

### 批处理系统中的作业处理

#### JCL 与作业说明书

在批处理系统中，用户通常无法在运行过程中不断插手控制，因此必须事先把控制意图一并告诉系统。

这就引出了 **作业控制语言（JCL, Job Control Language）**。

其含义可以概括为：

- JCL 是书写作业说明书的语言
- 用于描述批处理作业的处理过程和控制意图
- 其中可以包含 I/O 命令、编译命令、操作命令和条件命令等

换句话说，JCL 的作用就是：

**把“用户希望系统如何处理该作业”明确写下来。**

#### 脱机用户接口的两种典型形式

常见的两种经典形式是：

- **作业控制卡**
- **作业说明书**

它们都服务于同一个目的：表达用户对作业的控制意图。

其中通常要描述的内容包括：

- **作业的基本描述**
- **作业控制描述**
- **资源要求描述**

也就是说，系统不仅要知道“运行什么”，还要知道：

- 作业叫什么
- 需要什么资源
- 按什么顺序执行
- 出现问题时如何处理

#### 作业控制块 JCB

系统内部不能只靠一张“说明书”管理作业，它还需要一种正式的数据结构来记录作业信息，这就是 **作业控制块（JCB, Job Control Block）**。

需要强调的是：

- JCB 是对作业进行描述的数据结构
- 是批处理作业存在的标志
- 保存系统管理该作业所需要的全部信息
- 位于磁盘区域中，与作业一一对应

#### JCB 有什么用

可以把 JCB 理解成“操作系统给作业建立的档案卡”。

有了这张“档案卡”，系统才能知道：

- 这个作业是谁提交的
- 它目前处于什么状态
- 它需要哪些资源
- 它对应的数据和程序在哪里
- 它应该如何继续调度和处理

#### JCB 的生命周期

JCB 的生命周期包括三个阶段：

- **建立**
- **使用**
- **撤销**

这说明 JCB 不是永久存在的，而是随着作业的进入、运行和结束而建立和释放。

### 批处理作业的状态与转换

结合前后文，可以把批处理作业的状态转换理解为：

- 作业先进入系统
- 等待被调度和装入
- 进入运行或处理阶段
- 完成后退出系统

在这个过程中，操作系统必须跟踪作业当前所处阶段，并根据资源和调度条件推动其状态变化。

这也是为什么需要 JCB 这类数据结构。

### 交互式系统中的作业管理

与批处理不同，交互式系统中的作业管理更强调“人机对话式控制”。

可以看到，交互式系统作业管理通常由一组操作系统命令实现，用户通过对话方式控制作业运行。

#### 交互式系统中的三种常见方式

常见的典型方式有三种：

- **命令驱动方式**
- **窗口系统与菜单驱动方式**
- **命令文件方式**

这三种方式并不是互斥的，很多现代系统会同时支持。

### 命令驱动方式

命令驱动方式是最经典的交互式控制方式之一。

其流程是：

1. 用户输入命令
2. 系统解释命令
3. 系统执行命令
4. 屏幕显示结果
5. 用户根据结果决定下一步操作

这种模式最突出的特点是：

- **交互性强**
- **控制灵活**
- **适合调试和逐步操作**

但它的缺点也很明显：

- 命令格式复杂
- 学习成本较高
- 需要用户具备较强的命令记忆能力

### 窗口系统与菜单驱动方式

窗口系统与菜单驱动方式，是比纯命令行更直观的一类交互方式。

与命令行相比：

- 用户不必记住大量命令格式
- 可以借助图标、窗口、菜单和鼠标完成操作
- 更适合普通用户和初学者

但这里也提醒：

**图形方式虽然方便，但在效率上不一定优于命令行方式。**

这尤其体现在：

- 大批量重复操作
- 复杂命令组合
- 自动化控制

这些场景中，命令行往往更强。

### 命令文件方式

命令文件方式可以看成是：

**把多条命令组织成一个文件，再把这个文件当作命令来执行。**

这使得交互式系统也能在一定程度上实现“联机状态下的批处理”。

#### 命令文件方式的意义

这里中强调，它主要面向：

- 大量顺序操作
- 可重复执行的任务
- 需要一定自动化能力的工作

它的本质作用，是把重复劳动从“手工逐条输入命令”变成“写一次，反复执行”。

#### 命令文件通常具备哪些能力

这里中提到的能力包括：

- 接受参数和变量
- 支持宏变量替换
- 允许嵌套调用其他命令文件
- 允许输出到终端、文件或打印机
- 允许加入注释
- 允许在出错时进行用户干预
- 支持循环、分支、转移等控制结构

可以看出，命令文件已经不只是“命令列表”，而是具备了相当强的脚本语言特征。

### UNIX 中的 Shell 实例

以 UNIX 为例，交互式作业管理中一个很典型的实现就是 **Shell**。这一部分虽然属于交互式作业管理的延伸，但内容很重要，单独整理会更清楚。

#### Shell 是什么

##### Shell 的定义

一个常见定义是：

**Shell 是 UNIX 操作系统内核和用户之间的接口。**

在 UNIX 类系统中：

- **Kernel（内核）** 在系统启动时载入内存
- **Shell** 存放在磁盘上，由内核负责装入内存
- 用户登录后，Shell 启动
- 用户通过 Shell 与内核交互

所以 Shell 的本质角色就是：

**命令解释器 + 用户接口层。**

##### Shell 为什么重要

Shell 负责解释用户输入的命令，并组织这些命令与内核、程序和文件系统之间的交互。

它不仅让用户能够：

- 运行命令
- 管理文件
- 启动程序
- 重定向输入输出

还让用户可以把命令写成脚本，实现自动化处理。

#### Shell 如何解释命令

这里中给出了一个典型流程：

1. 用户登录，Shell 启动
2. Shell 提示用户输入命令
3. 用户键入命令
4. Shell 分析命令行
5. Shell 处理通配符、重定向、管道、任务控制等
6. Shell 查找命令并执行

这说明 Shell 不是简单“把字符串原样交给系统”，而是做了大量预处理工作。

#### Shell 的类型

常见 Shell 包括：

- **Bourne shell（sh）**
- **C shell（csh）**
- **Korn shell（ksh）**
- **Bourne Again shell（bash）**
- 以及其他扩展 Shell，如 `tcsh`、`pdksh`、`zsh`

##### 几类常见 Shell 的特点

- **sh**：最早期、最基础、兼容性强，Shell 编程能力较好。
- **csh**：更注重交互友好性，语法风格更接近 C 语言。
- **ksh**：综合了 sh 与 csh 的优点，并保持与 Bourne shell 的兼容。
- **bash**：在兼容 sh 的基础上做了大量增强，兼顾编程能力与交互友好性，是 Linux 中极常见的一种 Shell。

#### 初始化文件

Shell 在启动时，往往会读取一些初始化文件，用来完成环境设置。

这些初始化文件主要用来设置：

- `PATH`
- `USER`
- 文件权限相关默认属性
- 程序异常终止时的 core 文件限制

不同 Shell 对应的初始化文件不同，例如：

- `sh` 典型执行 `.profile`
- `csh` 典型执行 `.cshrc` 和 `.login`
- `ksh` 常执行 `.profile` 和 `$ENV`

#### Shell 的基本特征

##### 内部命令与外部命令

先看两类常见命令：

- **内部命令**：如 `cd`、`exit`
- **外部命令**：如 `cp`、`ls`、`mkdir`

两者的区别可以先理解为：

- **内部命令** 由 Shell 自己直接处理
- **外部命令** 通常对应独立程序，由 Shell 启动执行

##### 选项与参数

命令通常可以带选项，例如：

```bash
ls -a -l
```

其中：

- `-a` 表示列出所有文件
- `-l` 表示列出详细信息

这说明命令行语言并不是“只有命令词”，而是包含命令、选项、参数的完整结构。

##### 通配符

这里特别指出，通配符是由 Shell 处理后，再传递给外部命令的。

例如：

```bash
cat *.c
```

这里并不是 `cat` 自己识别 `*.c`，而是 Shell 先把它展开成具体文件名列表，再把结果传给 `cat`。

#### 输入输出重定向

Shell 启动后，通常会继承三个标准文件：

- `stdin`：标准输入
- `stdout`：标准输出
- `stderr`：标准错误输出

默认情况下：

- `stdin` 来自键盘
- `stdout` 和 `stderr` 输出到屏幕

但在很多情况下，我们希望把输入输出改向其他文件，这就需要 **重定向**。

##### 常见重定向符号

- `<`：标准输入重定向
- `>`：标准输出重定向
- `>>`：标准输出追加重定向
- `2>`：标准错误输出重定向
- `2>>`：标准错误输出追加重定向

还有一种典型写法：

```bash
2>&1
```

它表示把标准错误输出重定向到标准输出当前所指向的位置。

##### 重定向的意义

重定向让命令行操作不再局限于“键盘输入、屏幕输出”。

它使得我们可以：

- 从文件中读取输入
- 把结果写入文件
- 把错误信息单独保存
- 把多条命令连接成更复杂的数据处理流程

#### 管道

**管道（Pipe）** 用来把一个命令的输出作为下一个命令的输入。

典型例子如：

```bash
ls -l | wc -l
```

它表示：

- `ls -l` 先输出目录内容
- 这个输出不直接显示，而是通过管道送给 `wc -l`
- `wc -l` 再统计行数

另一个典型例子是：

```bash
cat sample.text | grep "High" | wc -l
```

这展示了管道的重要思想：

**把多个简单命令连接起来，组合完成较复杂的功能。**

#### 后台执行

有些命令执行时间较长，如果一直占用终端，会影响后续操作。为此 Shell 提供了 **后台执行** 功能。

这里中给出的基本形式是：

```bash
cmd &
```

在命令后加上 `&`，就表示命令在后台运行，终端可以继续接收新的输入。

#### 环境变量与内部变量

这里需要区分：

- **内部变量**
- **环境变量**

并强调一个关键点：

**继承只对环境变量有效。**

也就是说：

- 内部变量通常只在当前 Shell 中有效
- 环境变量会被子进程继承

这也是为什么像 `PATH` 这种变量必须作为环境变量存在。

#### 别名

Shell 还支持 **别名（alias）**，用来给复杂或冗长命令起一个更短、更顺手的名字。

例如：

```bash
alias dir='ls -a -l'
```

这样以后输入 `dir`，就等价于输入 `ls -a -l`。

别名的价值在于：

- 提高常用操作效率
- 降低复杂命令的记忆负担
- 允许用户定制自己的命令环境

#### 常用外部命令

这里列举了一批常见命令，大致可分为几类：

- **帮助与文本处理**：`man`、`echo`、`wc`、`grep`、`sed`、`awk`、`sort`、`cut`
- **文件与目录操作**：`pwd`、`ls`、`mkdir`、`rmdir`、`cp`、`rm`、`mv`、`ln`
- **权限与属主管理**：`chmod`、`chown`、`chgrp`
- **文件查看**：`cat`、`more`、`tail`
- **测试与计算**：`test`、`expr`
- **编辑器**：`vi`

#### Shell 脚本与批处理

命令文件进一步发展后，就形成了 **Shell Script（Shell 脚本）**。

##### Shell 脚本是什么

Shell 脚本本质上是：

**把一系列 Shell 命令按一定规则写进文件，再交给 Shell 解释执行。**

需要注意：

- 注释一般用 `#`
- 续行可用反斜杠 `\`
- 解释执行，效率通常不如编译型程序
- 常用 `#!/bin/sh` 这类首行指定解释器

##### 为什么脚本很重要

脚本让命令接口不再只是“手工交互工具”，还变成了：

- 自动化工具
- 批处理工具
- 快速搭建工作流的工具

#### 脚本中的几个典型结构

##### 变量替换

变量赋值的一般形式是：

```bash
temp="test message"
```

变量引用时常写作：

```bash
$temp
${temp}
"$temp"
```

其中单双引号的区别很重要：

- **双引号**：通常仍进行变量替换
- **单引号**：通常不进行变量替换

##### 条件转移

脚本中可以用 `if ... then ... else ... fi` 结构进行条件判断。

它的作用与高级语言中的 `if-else` 非常相似，本质上是根据条件决定执行哪一组命令。

##### 循环

这里给出了 `while ... do ... done` 的形式。

循环的作用是：

- 在某条件成立时重复执行一组命令
- 常用于等待资源、轮询状态、批量处理文件等场景

##### 分支

Shell 也支持 `case ... in ... esac` 这类多分支结构。

它适合：

- 根据参数类型进行不同处理
- 按文件后缀、命令参数等进行分类判断

##### 函数

Shell 允许把一组命令组织成 **函数**，从而提高脚本的复用性和结构性。

函数通常由两部分组成：

- **函数标题**
- **函数体**

有了函数以后：

- 脚本更容易组织
- 重复逻辑可以封装
- 参数传递也更加自然

::: details 小结：Shell 在本章中的位置

Shell 不是“顺手补充”的小知识，而是联机用户接口最典型、最有代表性的实现之一。

它把以下能力连接了起来：

- 用户输入命令
- 系统解释命令
- 命令组合、重定向和管道
- 自动化脚本
- 联机状态下的批处理

所以学懂 Shell，实际上是在理解“交互式作业管理”是如何落地的。

:::

## 2.3 系统调用

系统调用是本章另一条非常重要的主线。它把“用户程序”和“操作系统内核”联系起来。

### 为什么需要系统调用

用户程序在运行时，常常需要做很多自己无法直接完成的事情，例如：

- 读写文件
- 访问设备
- 创建和撤销进程
- 申请和释放内存
- 获取系统时间

这些操作要么直接涉及硬件，要么会影响整个系统状态。如果允许普通用户程序随意直接操作，系统很容易失去保护和控制。

因此，操作系统必须提供一种受控方式，让用户程序“请求”系统代劳，这就是 **系统调用（System Call）**。

::: warning 一个非常关键的结论

系统调用是操作系统提供给软件开发人员的唯一正式接口。

也就是说，应用程序不能随意直接操作受保护的系统资源，而要通过系统调用进入内核，请求操作系统完成相关工作。

:::

### 程序状态类型：用户态与核心态

为了理解系统调用，必须先理解 **程序状态类型**。

#### 两种基本状态

这一部分常用的术语是：

- **算态 / 用户态**
- **管态 / 系统态**

它们通常可对应为：

- **用户态**：用户程序运行的状态
- **核心态**：操作系统内核运行的状态

#### 为什么要区分两种状态

因为有些操作具有很高权限，例如：

- 直接控制设备
- 改变内存管理状态
- 执行特权指令

如果这些操作在用户态就可以随意执行，系统安全性和稳定性将无法保证。

因此：

- 用户程序在用户态运行，只能使用普通指令
- 操作系统在核心态运行，可以使用特权指令

#### 特权指令与访管指令

两个重要概念：

- **特权指令**
- **访管指令**

其中：

- **特权指令**：只允许在核心态下执行的指令
- **访管指令**：用于引发从用户态到核心态的切换，让操作系统接管处理

系统调用正是通过访管指令触发的。

### 系统调用的定义

系统调用可以定义为：

**至少包含一条访管指令在内的、由操作系统代替用户完成某些特定功能的程序段。**

这个定义里要抓住三个关键词：

- **访管指令**
- **代替用户完成**
- **特定功能**

这说明系统调用不是一般的函数调用，而是：

**用户程序借助硬件和内核机制，请求操作系统执行受保护服务的过程。**

### 系统调用的主要功能

系统调用的功能可归纳为多个方面：

- **设备管理**：设备的读写和控制
- **文件管理**：文件读写和文件控制
- **进程控制**：创建、中止、暂停进程
- **进程通信**：建立和使用消息队列、共享存储区、Socket 等通信渠道
- **存储管理**：内存申请和释放
- **系统管理**：读取时间、读取主机和用户标识等

可以看出，系统调用几乎覆盖了操作系统向程序提供的核心服务。

### 系统调用的实现过程

#### 基本思路

为了保护操作系统不被用户程序破坏，系统不能允许用户程序直接访问操作系统内部程序和数据。

因此，系统调用需要借助一个类似“中断处理”的机制来完成：

1. 用户程序执行系统调用指令
2. 处理机在执行到该指令时，产生相应中断或陷入
3. 控制权转交给系统中的处理机构
4. 系统在核心态下完成所请求功能
5. 执行结束后，再返回用户程序继续运行

#### 陷入与异常处理机构

- 控制系统调用服务的机构称为 **陷入（Trap）** 或 **异常处理机构**
- 引起这种转变的指令称为 **陷入指令**、**异常指令** 或 **访管指令**

这说明系统调用并不是“直接跳转到某个系统函数”，而是要经过专门的受控入口。

#### 功能号与入口地址表

在实际系统中，每个系统调用通常会有一个预先规定的 **功能号**。

系统通常还会准备一张 **入口地址表**：

- 功能号对应表项
- 表项对应某个系统程序入口
- 陷入处理程序根据功能号找到对应服务程序

于是，系统调用的实际处理流程就是：

1. 用户程序给出功能号和必要参数
2. 陷入处理机构接管
3. 查入口地址表
4. 启动相应系统子程序
5. 返回结果

#### 现场保护与恢复

系统调用执行前后，还必须处理一个重要问题：**保护现场**。

原因很简单：

- 用户程序本来正在执行
- 一旦进入系统调用，处理机上下文会变化
- 如果不保存现场，就无法正确返回原程序继续执行

因此系统需要：

- 在进入核心态前保存处理机现场
- 在系统调用结束后恢复现场

这样用户程序才能像“调用了一个服务后回来继续执行”那样正常运行。

### 参数传递问题

系统调用不仅要告诉内核“做什么”，还要告诉内核“用什么数据做”。这就涉及参数传递。

这里给出了三类常见做法：

#### 由陷入指令自带参数

这种方法简单直接，但缺点也明显：

- 陷入指令长度有限
- 还要携带功能号
- 因而只能附带很少量参数

#### 通过通用寄存器传递参数

这种方式速度较快，但也有限制：

- 可用寄存器数量有限
- 单个寄存器容量有限
- 不适合传递大量或复杂参数

#### 通过内存中的专用区域传递参数

可以看到，实际系统中常在内存中开辟专门区域或堆栈区来传递参数。

这种方式更灵活，因为它可以：

- 传递更多参数
- 支持更复杂的数据结构
- 便于系统调用接口扩展

### 系统调用与普通过程调用的相同点和不同点

这一部分是典型易错点。

#### 相同点

两者相同之处在于：

- 都会改变指令执行流程
- 都可能重复执行和公用某段功能
- 调用完成后都需要返回原处

也就是说，从“程序流程控制”角度看，它们有相似性。

#### 不同点

两者的本质差异在于：

| 对比维度   | 系统调用                           | 普通过程调用                   |
| ---------- | ---------------------------------- | ------------------------------ |
| 调用对象   | 操作系统提供的受保护服务           | 同一程序中的普通过程或函数     |
| 运行状态   | 通常涉及用户态到核心态切换         | 一般不改变运行特权级           |
| 调用依据   | 功能号、陷入机制、系统入口表       | 静态调用地址                   |
| 权限级别   | 可请求受保护的系统服务             | 只能执行本程序权限范围内的逻辑 |
| 实现复杂度 | 更高，需要陷入、现场保护、返回处理 | 相对简单                       |

这里还特别指出：

- **系统调用是动态调用**
- **CALL 方式的普通过程调用更偏向静态调用**

并且：

- 系统调用指令中一般不直接包含被调用代码地址
- 它更多只包含功能号，由系统再去分派处理

### 系统调用举例

典型例子主要围绕：

- 向打印机输出字符
- 读写硬盘文件内容
- 在高级语言中通过调用实现输出

这些例子的核心目的不是让你记住某条汇编指令，而是让你理解：

**同样一个“输出”动作，可以通过系统调用实现，也可以直接试图操作硬件；而操作系统之所以强调系统调用，是为了在保证安全和正确性的前提下统一管理这些资源。**

## 2.4 图形用户接口

前面已经在用户接口部分多次提到 GUI，这里再单独系统整理。

### GUI 的基本思想

**图形用户接口（GUI, Graphical User Interface）** 的核心思想，是用图形元素来表达功能，让用户通过更直观的方式与操作系统交互。

与命令行相比，GUI 的优势主要在于：

- 操作更直观
- 记忆负担更小
- 更适合一般用户
- 能把多个任务并行地展示在屏幕上

### 窗口系统的特点

窗口系统有几个突出特点：

- **利用图形元素表示功能**
- **支持同屏多窗口**
- **多窗口常与并发进程相对应**

这说明 GUI 不只是“好看”，它还提供了一种新的任务组织方式。

通过窗口系统，用户可以更容易感知：

- 哪些程序正在运行
- 哪个窗口当前在前台
- 多个任务如何并行存在

### 常见图形元素

窗口系统中的几个典型图形元素包括：

- **窗口（Window）**
- **图标（Icon）**
- **鼠标指针（Mouse Pointer）**
- **按钮（Button）**
- **菜单（Menu）**
- **对话框（Dialog Box）**

这些元素共同组成了 GUI 的基本表达语言。

### 窗口管理器

GUI 并不是应用程序各自随意画窗口就行，还需要一个统一协调者，这就是 **窗口管理器（Window Manager）**。

窗口管理器主要负责：

- 处理窗口的普遍特性
- 协调各窗口之间的相互关系

例如：

- 哪个窗口在前面
- 窗口如何移动、缩放、关闭
- 窗口焦点属于谁
- 多窗口之间如何避免混乱

### GUI 的优点与局限

#### 优点

窗口系统有几个明显优势：

- 图形元素风格一致
- 操作直观
- 有利于并发交互

对于初学者和普通用户来说，这些优点尤其重要。

#### 局限

但 GUI 也并非在所有方面都优于命令行：

- 对批量重复任务不一定高效
- 自动化能力通常不如脚本和命令行
- 对专业用户来说，复杂操作未必方便

因此现代操作系统常常同时提供：

- GUI
- CLI
- 程序级系统调用接口

它们共同构成完整的接口体系。

### Windows 作为 GUI 示例

这里以 `MS Windows` 为例说明 GUI 系统的典型结构。

其结构可粗略分为三部分：

- **OS 系统服务（Kernel）**：内存管理、程序加载、任务调度、文件管理等
- **用户接口（USER）**：窗口和消息管理，以及菜单、控制、对话框、定时器等
- **图形设备接口（GDI）**：为 USER 和应用程序提供与图形硬件相对独立的接口

这个例子说明：

GUI 并不是“附加在系统表面的一层皮肤”，而是与内核、消息机制、图形设备支持紧密结合的系统组成部分。

8. 为什么用户程序不能直接操作硬件，而必须通过系统调用？
9. 用户态和核心态有什么区别？访管指令、陷入和系统调用之间是什么关系？
10. 系统调用与普通过程调用有哪些相同点和不同点？
11. GUI 的主要图形元素有哪些？窗口管理器负责什么？

::: details 参考答案提纲

1. **作业级接口**用于**控制作业运行全过程**，**命令级接口**用于**用户通过命令与系统交互**，**程序级接口**用于**程序通过系统调用请求操作系统服务**。
2. **脱机接口面向批处理**，强调**事先提交和自动执行**；**联机接口面向交互式使用**，强调**实时对话和即时反馈**。
3. **作业是用户要求系统完成的一项完整工作**；**作业步是作业中的独立子任务**；**作业流是若干作业按顺序组成的作业序列**；作业通常由**程序**、**数据**和**作业说明书**构成。
4. **JCL 是书写作业说明书的语言**，**作业说明书表达用户对作业处理的控制意图**，**JCB 是系统内部用于描述和管理作业的数据结构**。
5. **命令驱动方式灵活但要求记忆命令**；**窗口菜单方式直观易用**；**命令文件方式适合把多条命令组织成自动化处理过程**。
6. Shell 是**内核与用户之间的接口**，也是**命令解释器**；它负责**解释命令、处理通配符、重定向、管道、任务控制，并支持脚本**。
7. **重定向解决输入输出改向问题**，**管道解决命令组合问题**，**后台执行解决长任务不阻塞终端的问题**，**变量替换则提高脚本编写灵活性**。
8. 因为**直接访问硬件会破坏系统安全性和稳定性**，所以必须由操作系统通过受控方式提供服务。
9. **用户态权限较低，核心态权限较高**；**访管指令会触发陷入**，使**程序从用户态进入核心态**，由**操作系统执行相应系统调用服务**。
10. 系统调用和普通过程调用**都会改变控制流程并在结束后返回**，但**系统调用涉及特权级切换和内核服务**，而**普通过程调用通常只是在同一程序内部转移执行**。
11. GUI 的典型元素包括**窗口、图标、鼠标指针、按钮、菜单和对话框**；**窗口管理器负责统一管理窗口特性并协调窗口之间的关系**。
