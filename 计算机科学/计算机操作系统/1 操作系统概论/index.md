# 第 1 章 操作系统概论

本章是整门操作系统课程的起点。学习这一章的目标，不是立刻去记忆大量实现细节，而是先建立一张完整的“认知地图”：

- 操作系统到底是什么
- 为什么计算机必须有操作系统
- 操作系统经历了怎样的发展过程
- 操作系统负责做哪些事情
- 操作系统有哪些最核心的共性特征
- 常见操作系统可以如何分类
- 操作系统内部通常按什么结构组织

当这些问题有了清晰答案后，再去学习进程管理、存储管理、文件系统和设备管理时，就不会只见树木、不见森林。

::: info 学习提示

这一章最重要的，不是死记名词，而是把几个视角统一起来：

- 从**用户视角**看，操作系统是“方便使用计算机的工具”
- 从**资源视角**看，操作系统是“统一管理软硬件资源的管理者”
- 从**实现视角**看，操作系统是“控制程序执行、组织系统运行的一组核心程序”

后续很多概念，例如并发、共享、虚拟、系统调用、微内核，本章都会先给出直观理解。

:::

## 1.1 什么是操作系统

### 基本认识

#### 先从直觉出发

一台裸机只有硬件时，用户直接使用它会非常困难。程序如何装入内存、CPU 先运行谁、磁盘数据如何组织、多个程序如何避免相互破坏，这些问题都不能靠应用程序各自“随便处理”。

于是，计算机系统需要一层专门的软件，负责：

- 控制和协调硬件工作
- 管理各种系统资源
- 为用户和应用程序提供统一、方便的使用方式

这层软件就是 **操作系统（Operating System, OS）**。

#### 操作系统的几种典型理解

从课堂内容来看，操作系统至少可以从下面几个角度来理解：

- **资源管理者**：管理处理机、存储器、外设、文件等软硬件资源。
- **用户接口**：为用户使用计算机提供命令接口、图形接口、程序接口等。
- **扩展机器 / 虚拟机**：把复杂、零散、不易直接使用的硬件，包装成一个更容易使用的“逻辑机器”。
- **系统程序集合**：它是最重要、最基本、最复杂的一类系统软件，控制应用程序的执行。

这几个说法并不矛盾，而是分别强调了操作系统的不同侧面。

#### 一个较完整的定义

可以把操作系统定义为：

**操作系统是控制和管理计算机软硬件资源，合理组织计算机工作流程，并为用户和应用程序提供方便使用环境的程序集合。**

这个定义里有几个关键词，初学时一定要读懂：

- **控制**：让系统按照规定方式运行，而不是混乱运行。
- **管理**：对有限资源进行分配、回收、保护、调度和协调。
- **合理组织工作流程**：让 CPU、内存、外设、文件等协同工作，提高整体效率。
- **方便使用**：用户和程序不需要直接面对底层硬件细节。

### 角色与定位

#### 为什么说操作系统是“扩展机器”

“扩展机器”或“虚拟机”这个说法，初学者第一次看到时往往会疑惑。

它的意思并不是“操作系统凭空创造了新硬件”，而是说：

- 底层真实硬件通常很难直接使用
- 操作系统在其上提供了更高级、更友好的抽象
- 用户实际面对的是“经过包装后的机器”

::: info 例如

- 物理磁盘本来只是扇区和块，操作系统把它组织成**文件和目录**
- 真实内存容量有限，操作系统通过虚拟存储技术让程序感觉自己拥有更大的地址空间
- 一个 CPU 在某一瞬间只能真正执行一个指令流，但多道程序与分时技术让用户感觉多个程序像是在“同时运行”

:::

::: tip 直观理解

操作系统就像“中间层翻译器 + 总调度员”：

- 对下，它理解硬件细节
- 对上，它提供更容易使用的抽象

所以用户通常不是直接操纵硬件，而是在使用操作系统提供的能力。

:::

#### 操作系统在计算机系统中的位置

从层次上看，一个典型计算机系统可以粗略分成：

1. **硬件**：CPU、主存、磁盘、键盘、显示器、网卡等
2. **操作系统**：管理和控制硬件资源
3. **应用程序**：编译器、浏览器、数据库、编辑器、游戏等
4. **用户**：人或其他外部系统

也就是说，操作系统位于**硬件和应用之间**，既不是硬件本身，也不是普通应用软件。

### 引入操作系统的目标

引入操作系统主要有几个目标：

- **方便性**：从用户角度看，要让计算机更易用。
- **有效性**：从系统管理角度看，要提高资源利用率和系统吞吐量。
- **可扩充性**：从升级演化角度看，系统应能较容易增加或替换功能。
- **开放性**：从兼容互联角度看，系统应尽量遵循标准，便于与其他系统协作。

这几个目标在现实中往往不是完全一致的，而是需要做权衡。

例如：

- 更强的保护机制通常更安全，但可能带来额外开销
- 更灵活的结构通常更容易扩展，但实现可能更复杂
- 更高的性能有时会牺牲一定的通用性

这也是为什么操作系统课程里经常会提到 **trade-off（折中、权衡）**。

## 1.2 操作系统的发展历史

操作系统不是一开始就存在的，它是随着硬件能力提升和应用需求变化逐步发展出来的。

整体来看，操作系统的发展目标始终围绕两件事：

- **充分利用硬件资源**
- **为用户提供更好的服务**

### 主要发展阶段

#### 第一代：手工操作

早期计算机大约处于 `1945 ~ 1955` 年间，当时的特点是：

- 计算机体积大、成本高、硬件原始
- 主要使用机器语言
- 输入输出方式原始，如插件板、纸带、卡片
- **没有真正意义上的操作系统**

那时计算机的使用方式带有明显的“手工操作”特征：

- 程序员准备好程序和数据
- 通过人工方式装入机器
- 人工控制作业运行
- 等待结果输出

这种方式的核心问题是：**人工操作太慢，机器等待太多，昂贵的计算能力被大量浪费。**

#### 第二代：单道批处理系统

随着硬件发展，进入 `1955 ~ 1965` 年左右，出现了 **单道批处理系统（Simple Batch Processing System）**。

“批处理”的关键思想是：

- 用户不再直接坐在机器前交互操作
- 而是把一个个作业提交给操作员
- 操作员把许多作业组成一批，交给系统顺序执行

这类系统的代表有 `FMS`、`IBSYS` 等。

##### 单道批处理是怎样工作的

在单道批处理环境中，系统一次只装入并运行一个作业。一个作业完成后，系统再自动装入下一个作业。

它相比手工操作的进步在于：

- 用户不需要频繁直接控制机器
- 同一批作业可以自动依次执行
- 减少了作业切换中的人工等待时间

但它也有明显局限：

- 某时刻内存中通常只有一个作业
- 如果该作业在等待 I/O，CPU 就可能闲着
- 如果该作业计算很多，外设又可能闲着

换句话说，**CPU 和 I/O 设备的忙闲不均衡问题仍然很突出**。

##### 通道和中断技术

为了改善 CPU 与外设不能高效并行的问题，系统逐渐引入了 **通道** 和 **中断** 技术。

- **通道（Channel）**：专门负责控制 I/O 设备与内存之间的数据传输。启动后可以相对独立于 CPU 工作。
- **中断（Interrupt）**：当外部事件发生时，CPU 可以暂停当前工作，转去处理该事件，处理完再返回断点继续执行。

它们的重要意义在于：

- CPU 不必亲自盯着整个 I/O 过程
- I/O 和 CPU 的工作可以更好地重叠
- 系统并行工作能力增强

::: info 为什么中断这么重要

没有中断时，CPU 往往只能不断轮询设备状态，效率很低；有了中断后，设备“准备好了再通知 CPU”，系统效率会明显提高。

后续很多操作系统机制，本质上都离不开中断。

:::

#### 第三代：多道批处理系统

进入 `1965 ~ 1980` 年左右，随着集成电路出现，操作系统进入 **多道批处理系统（Multiprogramming System）** 阶段。

“多道”的核心思想是：

**内存中同时存放多个作业，当某个作业因 I/O 等原因不能继续占用 CPU 时，系统可以让另一个作业运行。**

这就是 **多道程序设计（Multiprogramming）** 的基本思想。

##### 多道程序设计为什么能提高效率

假设两个程序的资源需求不同：

- 程序 A 更偏向计算
- 程序 B 更偏向 I/O

如果顺序执行，A 等 I/O 时 CPU 可能空闲，B 等 I/O 时 CPU 也可能空闲；但若把它们放在内存中交替推进，就能让一个程序等待时，另一个程序补上，从而提高 CPU 利用率。

::: info 例如

- 顺序环境下，两个程序总耗时 `80s`，CPU 利用率只有 `50%`
- 多道程序环境下，总耗时缩短到 `45s`，CPU 利用率提高到约 `88.89%`

:::

这说明多道程序设计的本质价值是：

- **提高资源利用率**
- **提高系统吞吐量**

::: info 资源利用率和吞吐量

- **资源利用率**：CPU、内存、I/O 设备等资源被有效使用的程度
- **吞吐量**：单位时间内系统完成的工作总量

:::

##### 多道批处理系统的优缺点

优点：

- 资源利用率高
- 作业吞吐量大

缺点：

- 用户交互性差
- 调试程序困难
- 作业平均周转时间可能较长

  ::: info 周转时间

  从作业提交给系统开始，到作业完成并得到结果为止的总时间。

  :::

##### SPOOLing 技术

在批处理系统发展过程中，还出现了一项很重要的技术：**SPOOLing**。

SPOOLing 是 Simultaneous Peripheral Operation On-Line 的缩写，常译为 **假脱机技术**。

它的基本思想是：

- 利用磁盘作为大缓冲区
- 把输入、计算、输出组织成彼此相对独立的任务流
- 让 I/O 和计算更充分地并行

SPOOLing 的典型作用包括：

- **提高 I/O 速度**
- **把独占设备改造成共享设备**
- **实现虚拟设备功能**

例如，打印机本来一次只能真正为一个作业服务；有了假脱机后，多个作业的打印请求可以先进入磁盘缓冲区排队，用户会感觉像是在“同时提交打印”。

::: tip

“假脱机”中的“假”可以理解为“逻辑上的、模拟出来的”。

它是通过缓冲、排队和调度，让多个用户都像在独占使用设备。

:::

#### 第四代：分时系统

随着交互需求增强，操作系统进一步发展出 **分时系统（Time-Sharing System）**。

分时的基本思想是：

- 把 CPU 时间划分为许多很短的时间片
- 多个用户或多个程序轮流占用 CPU
- 切换速度足够快时，用户会感觉自己在“独占使用计算机”

分时系统解决的重点问题，不再只是“提高吞吐量”，而是更强调：

- **及时响应**
- **支持交互**
- **让多个用户共享一台主机**

##### 分时系统的四个典型特征

这里总结了分时系统的四个重要特点：

- **同时性 / 多路性**：多个用户看起来在同时工作。
- **交互性**：用户可以边输入命令、边观察结果、边调整操作。
- **独立性**：各用户彼此独立、互不干扰。
- **及时性**：系统要在较短时间内响应用户请求。

::: tip 响应时间

响应时间是**从用户发出命令，到系统处理完命令并给出回答**所经历的时间。

影响响应时间的因素包括：

- 机器处理能力
- 请求本身的复杂程度
- 在线终端数量
- 请求分布情况
- 调度算法
- 时间片大小

:::

##### 分时系统面临的问题

分时系统虽然交互性好，但也带来了新挑战：

- 如何及时接收多个终端输入
- 如何在多用户之间公平分配 CPU
- 如何缩短响应时间
- 如何降低频繁切换带来的开销

一些典型思路有：

- 设置多路缓冲区
- 提高对换速度
- 合理限制用户数目
- 缩短时间片，但避免过短导致切换开销过大
- 使用可重入代码
- 采用请求页式存储管理，只换入程序的一部分

### 分化与扩展

#### UNIX、个人计算机与操作系统分化发展

分时思想成熟之后，操作系统开始进一步分化。

一条重要历史线索是 `CTSS -> MULTICS -> UNIX`：

- **CTSS**：较早的分时系统代表
- **MULTICS**：提出了许多先进思想，对后来的操作系统影响巨大
- **UNIX**：在实践中取得巨大成功，并深刻影响了之后的 Linux、BSD、POSIX 体系

之后，随着个人计算机兴起，操作系统又发展出多个方向：

- **个人操作系统**：如 `CP/M`、`DOS`、`Windows`、`Mac OS`
- **实时操作系统**
- **网络操作系统**
- **分布式操作系统**
- **嵌入式操作系统**

#### 一些有代表性的系统

下面列出一些重要系统。记住它们的意义，比单纯记年份更重要：

- **OS/360**：多道批处理时代的代表系统之一，也让人们深刻认识到大型操作系统会变得非常庞大而复杂。
- **CTSS**：较早的分时系统代表，体现了多用户交互式计算的方向。
- **MULTICS**：虽然研发过程艰难，但提出了许多后来被证明非常有价值的思想，对 UNIX 影响深远。
- **UNIX**：以简洁、实用和良好的可移植性著称，后来发展出 `BSD`、`System V` 等谱系，并通过 `POSIX` 标准影响深远。
- **CP/M、DOS、Windows、Mac OS**：代表个人计算机操作系统走向普及，图形界面也逐步成为主流。

#### 国内操作系统研制概况

国内在操作系统研制方面也有持续探索，例如：

- 早期面向专用计算和控制场景的操作系统
- 国产 UNIX 类系统
- 面向嵌入式和 Linux 生态的国产系统

这一部分在本章不要求记忆具体型号，但应知道：

- 操作系统发展并不只发生在少数国家或少数公司
- 不同应用需求会推动不同类型操作系统持续演化
- “国产化、自主可控、面向专用场景优化”也是操作系统发展中的重要现实主题

### 历史总结

#### 发展历史的主线要抓住什么

这一段历史细节很多，但初学者最需要抓住的是演化逻辑：

1. **手工操作**：机器贵，人慢，浪费严重
2. **单道批处理**：减少人工切换，提高自动化程度
3. **多道批处理**：让 CPU 与 I/O 更充分并行，提高利用率和吞吐量
4. **分时系统**：强调多用户交互与及时响应
5. **进一步分化**：面向个人、实时、网络、分布式、嵌入式等不同需求发展

::: warning 容易混淆

“多道批处理”和“分时系统”都可能表现出多个程序轮流使用 CPU，但它们的核心目标不同：

- 多道批处理更关注**提高资源利用率和吞吐量**
- 分时系统更关注**交互性和响应时间**

不要把两者简单看成同一个东西。

:::

## 1.3 操作系统的功能

操作系统的功能通常可概括为五大类：

- **处理机管理**
- **存储管理**
- **设备管理**
- **文件管理**
- **用户接口**

这五类功能构成了操作系统的基本职责框架。

### 功能总览

### 资源管理功能

#### 处理机管理

处理机管理面对的核心问题是：

**CPU 很宝贵，而程序很多，应该让谁先运行、谁后运行、运行多久、彼此如何配合？**

处理机管理的主要内容包括：

- **进程控制**
- **进程同步**
- **进程通信**
- **进程调度**

::: info 名词解释

**进程**：一个正在执行的程序，可以把它理解为“程序的一次运行过程”。

**进程控制**：创建进程、撤销进程、阻塞进程、唤醒进程等。

**进程同步**：协调多个进程的执行次序，保证它们在正确时机配合。

**进程通信**：让多个进程交换信息。

**进程调度**：决定当前哪个进程占用 CPU。

:::

::: info

后续“[3 进程管理](../3%20进程管理/index.html)”章节会系统展开这些概念。本章只需要先明白：操作系统必须负责安排 CPU 的使用，否则多个程序一起运行时系统会陷入混乱。

:::

#### 存储管理

存储管理主要针对 **内存（主存）** 的使用与组织。

核心内容有：

- **内存分配**
- **内存保护**
- **地址映射**
- **内存扩充**

它们分别对应什么问题？

- **内存分配**：哪个程序分到多少内存，什么时候分配，什么时候回收。
- **内存保护**：一个程序不能随意破坏另一个程序或操作系统本身的内存区域。
- **地址映射**：程序里使用的地址，如何转换成真实可访问的物理地址。
- **内存扩充**：通过虚拟存储等技术，让程序可使用的逻辑空间大于实际物理内存。

存储管理的重要性在于：如果没有它，多个程序装入内存后很容易互相覆盖，系统的稳定性和安全性都无法保证。

#### 设备管理

设备管理负责组织和控制各类输入输出设备，如键盘、显示器、磁盘、打印机、网卡等。

设备管理内容包括：

- **缓冲管理**
- **设备分配**
- **设备处理**

可以这样理解：

- **缓冲管理**：利用缓冲区平衡 CPU 与设备速度不匹配的问题。
- **设备分配**：决定当前哪个进程可以使用某台设备。
- **设备处理**：通过驱动程序和中断机制控制设备完成实际 I/O 操作。

设备管理的一个典型难点是：**不同设备速度差异极大、使用方式不同、共享方式也不同。**

例如：

- 打印机通常是慢速且顺序工作的
- 磁盘可随机访问，但需要考虑调度效率
- 键盘和鼠标常常以中断方式报告输入事件

#### 文件管理

文件管理解决的问题是：

**数据如何长期保存、如何组织、如何命名、如何查找、如何保护？**

文件管理内容包括：

- **文件存储空间管理**
- **目录管理**
- **文件的读 / 写管理**
- **文件保护**

::: tip 为什么文件管理重要？

- 内存中的数据通常不能长期保存
- 用户更希望以“文件”和“文件夹”的方式组织信息
- 程序也需要统一的数据访问接口

于是，操作系统把底层磁盘块抽象成用户易于理解的文件系统结构。

:::

### 用户接口功能

#### 用户接口

如果说前四类功能主要偏向“内部管理”，那么 **用户接口** 则更多回答“用户和程序怎样使用操作系统”。

用户接口通常可分为三类：

- **命令接口**（又可分为**联机用户接口**和**脱机用户接口**）
- **程序接口**
- **图形接口**

##### 命令接口

- **联机用户接口**：用户在终端上直接输入命令，由命令解释程序解释执行。典型例子是 Shell。
- **脱机用户接口**：主要面向批处理作业用户，常通过作业控制语言 `JCL` 等方式描述作业执行要求。

##### 程序接口

程序接口是应用程序在执行过程中请求操作系统服务的通道，需要强调的是：

**程序接口是用户程序取得操作系统服务的唯一途径，由一组系统调用组成。**

::: info 系统调用（System Call）

- 应用程序通常不能直接执行特权操作
- 当它需要文件读写、进程创建、设备访问等服务时
- 就必须通过系统调用，请求内核代为完成

这也是应用程序和操作系统之间最重要的正式接口之一。

:::

##### 图形接口

图形接口用更直观的图形化方式展示系统功能、应用程序和文件，降低了普通用户的使用门槛。

例如：

- 窗口
- 图标
- 菜单
- 鼠标操作

图形界面的出现，是个人计算机普及的重要因素之一。

::: tip 从功能角度记忆操作系统

可以把操作系统想成“五位总管”：

- 管 CPU 的，是处理机管理
- 管内存的，是存储管理
- 管外设的，是设备管理
- 管数据长期保存的，是文件管理
- 管人与系统交互方式的，是用户接口

这套框架后续会不断重复出现。

:::

## 1.4 操作系统的特征

操作系统最核心的共性特征可概括为四个词：

- **并发**
- **共享**
- **虚拟**
- **异步**

这四个概念既是考试高频点，也是理解操作系统本质的重要抓手。

### 四大特征

#### 并发

**并发（Concurrency）** 指的是两个或多个事件在**同一时间间隔内**发生。

这与 **并行（Parallelism）** 不完全相同：

- **并发**：强调“在一段时间内都在推进”
- **并行**：强调“在同一时刻真正同时执行”

在单处理器系统中：

- 微观上，同一时刻 CPU 仍只能执行一个指令流
- 但多个程序可以交替推进
- 因而在宏观上表现为“同时运行”

这就是操作系统中最常说的并发。

::: warning 并发不等于并行

这是最常见的混淆点之一：

- 单核 CPU 上也可以有并发
- 但单核 CPU 上通常不能实现真正的同时并行执行多个指令流

所以“多个程序看起来一起运行”不一定意味着“它们在同一时刻真的同时运行”。

:::

#### 共享

**共享（Sharing）** 指的是系统中的资源可供多个并发执行的进程共同使用。

常见共享方式可以粗略理解为两类：

- **互斥共享**：某一时刻只允许一个进程使用资源
- **同时访问**：某些资源允许多个进程在一定条件下共同使用

例如：

- 打印机通常更接近互斥共享
- 只读文件在某些场景下可被多个进程同时访问

这里特别强调：

**并发和共享是操作系统最基本的两个特征，它们互为存在条件。**

为什么这么说？

- 没有并发，就谈不上多个执行活动争用和共享资源
- 没有共享，并发程序共同使用系统的意义会大幅下降

#### 虚拟

**虚拟（Virtual）** 指的是通过某种技术，把一个物理实体变成若干个逻辑上的对应物，或者把有限资源在逻辑上扩展成更“丰富”的形式。

几种典型虚拟技术包括：

- **虚拟处理机技术**：多道程序和分时技术让多个程序像是各自拥有处理机。
- **虚拟存储器技术**：让逻辑地址空间看起来大于实际物理内存。
- **虚拟设备技术**：把一台物理 I/O 设备抽象为多台逻辑设备供多个用户使用。
- **虚拟信道**：把一条物理信道通过分时等方式虚拟成多条逻辑信道。

虚拟技术的价值在于：它让有限、具体、难直接共享的物理资源，变得更灵活、更易用。

#### 异步

**异步性（Asynchronism）** 也常被称为 **不确定性（Nondeterminacy）**。

它强调的是：

- 多个进程推进的先后顺序不完全固定
- 每个进程运行快慢不可预知
- 系统在某一时刻的状态往往不容易事先精确预测

在并发系统中，进程可能出现“时走时停”的现象：

- 等待 I/O 时暂停
- 等待资源时阻塞
- 获得 CPU 后再继续执行

因此，系统执行过程通常不是简单、线性的。

##### 异步为什么会带来困难

异步性的存在，使得并发程序设计和系统调试变得困难：

- 同样的程序每次运行，内部执行次序可能不同
- 某些错误只在特定时序下出现，难以重现
- 如果缺乏同步与互斥机制，结果可能不正确

因此有一个重要判据：

**无论进程执行快慢如何变化，只要同步和互斥做得正确，结果都应当保持正确。**

::: details 四个特征之间的关系

- 并发：说明系统中有多个活动在推进
- 共享：说明这些活动会共同使用资源
- 虚拟：说明操作系统会对资源进行抽象和逻辑扩展
- 异步：说明这些活动推进的时机和速度常常不确定

可以把这四个特征看成对同一类系统现象的不同描述，而不是彼此独立、毫无关联的四个词。

:::

## 1.5 操作系统的分类

按照使用目标和应用场景的不同，操作系统可以分为多种类型。常见的有以下几类：

- **批处理操作系统**
- **分时操作系统**
- **实时操作系统**
- **网络操作系统**
- **分布式操作系统**
- **多处理机操作系统**
- **嵌入式操作系统**

### 按应用场景分类

#### 批处理操作系统

批处理操作系统的工作方式是：

- 用户把作业交给操作员
- 操作员把许多作业组成一批
- 系统自动、依次执行这些作业
- 执行结果再交还给用户

它又可进一步分为：

- **单道批处理系统**
- **多道批处理系统**

批处理系统的主要优点是：

- 作业流程自动化
- 资源利用率较高
- 吞吐量大

主要缺点是：

- 用户交互性差
- 调试不方便
- 平均周转时间可能较长

#### 分时操作系统

分时操作系统面向多用户交互场景，通过时间片轮转方式让多个用户依次使用系统资源。

它的关键特征是：

- 多路性
- 独立性
- 及时性
- 交互性

一句话概括：

**分时系统追求的是“让多个用户都觉得系统在及时响应自己”。**

#### 实时操作系统

**实时操作系统（Real-Time Operating System, RTOS）** 的核心不只是“算对”，还要“按时算对”。

相关定义强调：

**实时系统的正确性不仅依赖逻辑结果，还依赖结果产生的时间。**

根据时限要求不同，实时系统通常分为：

- **硬实时系统**：必须在规定时刻或时间范围内完成任务，超时就可能导致严重后果。
- **软实时系统**：允许偶尔违反时限，但总体仍要求较快响应。

实时系统与批处理、分时系统相比，通常具有以下特点：

- 更强调对外部事件的快速响应
- 中断处理能力更强
- 可靠性要求更高
- 常用于专用控制场景

#### 网络操作系统

**网络操作系统（Network Operating System, NOS）** 是在常规操作系统基础上，进一步提供网络通信和网络服务功能的操作系统。

它的目标主要是：

- **相互通信**
- **资源共享**

其主要功能包括：

- 网络通信
- 网络资源管理
- 网络服务
- 网络管理

例如文件共享、打印共享、网络安全管理、性能管理等，都属于网络操作系统关心的问题。

#### 分布式操作系统

**分布式操作系统（Distributed Operating System）** 建立在计算机网络基础之上，但它并不只是“联网而已”，而是希望把整个系统组织成一个更统一的整体。

理解这一部分时要抓住几个关键词：

- **透明性**：用户感觉不到资源分布在不同机器上。
- **自治性**：多个主机处于相对平等地位。
- **自动任务分配与负载调度**：系统能在全局范围内安排任务。

分布式操作系统的优点通常包括：

- 处理能力增强
- 速度更快
- 可靠性增强

### 易混淆类型对比

#### 分布式操作系统与网络操作系统的区别

这是一个非常容易混淆的知识点。

两者都基于网络环境，但侧重点不同：

| 对比维度     | 网络操作系统                 | 分布式操作系统                   |
| ------------ | ---------------------------- | -------------------------------- |
| 系统组织     | 多台机器通过网络协作         | 追求把多台机器组织成更统一的整体 |
| 用户感受     | 用户通常知道资源在哪台机器上 | 用户通常不必关心资源具体位置     |
| 操作系统关系 | 可允许异种操作系统互联       | 往往强调系统级统一管理           |
| 任务运行     | 各机器上的进程相对独立       | 可进行更全局的任务分配和调度     |

简而言之：

- **网络操作系统** 更像“联网的多台机器”
- **分布式操作系统** 更像“由多台机器组成的一台更大的系统”

### 其他常见类型

#### 多处理机操作系统

为了进一步提高系统性能和可靠性，可以在一台系统中配置多个处理器，这时就需要 **多处理机操作系统**。

它的目标主要有两个：

- **提高吞吐量**
- **提高可靠性**

可以看到，多处理机系统大体可分为：

- **紧密耦合**：多个处理机通过高速总线或开关阵列相连，共享内存，通常由统一 OS 管理。
- **松散耦合**：各处理机带有各自的存储器、I/O 设备和操作系统，通过通信线路相连。

进一步又可按控制方式分为：

- **非对称多处理（ASMP）**：主处理器负责运行 OS 和资源管理，从处理器执行分配来的任务。
- **对称多处理（SMP）**：OS 可在多个处理器上交替运行，任务分配更均衡。

#### 嵌入式操作系统

**嵌入式操作系统（Embedded Operating System）** 面向专用设备和特定应用环境。

嵌入式系统通常具有这些特点：

- 以应用为中心
- 软硬件可裁剪
- 对功能、可靠性、成本、体积、功耗有严格要求

因此，嵌入式操作系统往往：

- 规模较小
- 资源受限
- 常具有一定实时性
- 强调稳定、低功耗、专用性

典型例子包括 `PalmOS`、`Windows CE`、`VxWorks` 等。

::: info 如何理解“分类”

这些分类并不是绝对互斥的。

例如，一个嵌入式操作系统可能同时具有实时特征；一个现代桌面操作系统也可能支持网络功能、多处理器和图形界面。

分类更多是按“主要目标和典型场景”来区分。

:::

## 1.6 操作系统的结构

前面讲的是“操作系统做什么”，这里进一步讨论“操作系统内部如何组织”。

随着功能越来越复杂，操作系统不可能随意堆砌代码，而必须考虑结构设计。

常见的操作系统结构包括以下几类：

- **整体 / 模块结构**
- **分层结构 / 虚拟机结构**
- **客户 / 服务器模型（微内核思想）**
- **对象模式**

同时还提到了 **强内核** 与 **微内核** 的对比。

### 结构设计的出发点

#### 为什么操作系统需要结构设计

操作系统是大型、复杂、长期演化的软件系统。结构设计的好坏，会直接影响：

- 开发效率
- 正确性
- 可维护性
- 可扩展性
- 性能

操作系统结构设计中，往往不存在“各方面都最优”的方案，而是要在：

- 性能
- 结构清晰度
- 可扩展性
- 可靠性

之间不断权衡。

### 传统结构

#### 整体结构：整体 / 模块结构

整体结构或模块结构（monolithic system / modular system）的特点是：

- 系统由大量服务过程或模块组成
- 模块之间可以比较直接地相互调用

它的优点是：

- 开发初期实现相对直接
- 运行效率通常较高
- 适合把系统功能集中组织起来

缺点是：

- 模块划分和接口设计不容易做到完全合理
- 模块之间依赖关系可能非常复杂
- 修改一个模块，可能牵动很多其他模块

也就是说，这类结构常见问题是：**效率高，但容易“长成一团”**。

#### 分层结构 / 虚拟机结构

分层结构（layered system）的基本思想是：

- 把操作系统划分为若干层
- 高层只能调用低层提供的服务
- 调用关系保持有序

每加一层，都可以看作在原有基础上构造出一个功能更强的“虚拟机”。

分层结构的优点包括：

- 功能边界更清楚
- 调用关系更清晰
- 有利于保证设计和实现正确性
- 高层和低层可以相对独立开发
- 高层错误通常不至于直接破坏低层

缺点则是：

- 可能降低运行效率
- 不合理分层会导致调用路径过长

##### 分层设计的基本原则

分层设计通常遵循若干重要原则：

- 每一层只使用其下层提供的功能和服务
- 经常活跃的模块应尽量靠近硬件
- 公共模块应放在较低层，供上层复用
- 用户接口通常位于较高层

这些原则背后的本质，是让系统的依赖关系尽量简单、稳定、可控。

### 现代结构

#### 强内核与微内核

先看一个很重要的对比：**强内核** 与 **微内核**。

##### 强内核

强内核可以理解为传统集中式内核思路：

- 系统调用通过陷入内核实现
- 大量系统服务都在内核内部完成
- 用户程序请求服务时，由内核统一处理后返回结果

它的特点通常是：

- 内核功能强、覆盖面广
- 运行效率可能较高
- 但内核体积较大，复杂度高

##### 微内核

微内核的基本思想是：

- 让内核尽可能小
- 只保留最基本、最公共的核心功能
- 其余功能尽量放到核外的用户级服务器中实现

这里中提到微内核通常保留的基本服务包括：

- 有限的进程管理和调度
- 进程间通信机制
- 某些存储管理功能
- 低级 I/O 操作

微内核追求的是：

- 良好的结构化
- 模块化
- 更小的核心
- 更好的扩展性和可靠性

#### 客户 / 服务器模型

微内核思想常对应一种典型实现方式：**客户 / 服务器模型（Client-Server Model）**。

它把操作系统划分为两部分：

- **服务器进程**：提供各种系统服务
- **内核**：负责客户和服务器之间的通信，以及最基本核心功能

应用程序作为客户，通过消息传递请求服务器提供服务。

例如：

- 文件服务由文件服务器进程提供
- 设备服务由设备服务器进程提供
- 这些服务不一定都放在庞大的内核内部

#### 客户 / 服务器模型的优缺点

优点：

- 扩充性好，增加服务通常只需增加新的服务器进程
- 可靠性较好，调用关系更清楚
- 更容易支持网络化与分布式处理

缺点：

- 消息传递开销通常比直接过程调用更大
- 因此性能可能不如集中式强内核结构

::: info RPC（Remote Procedure Call，远程过程调用）

- 对上层程序来说，调用远端服务尽量像调用本地过程一样
- 对下层系统来说，通信和消息传递的复杂性被封装起来

:::

#### 对象模式

对象模式是把面向对象思想引入操作系统设计中。

其基本思路是：

- 把系统资源抽象为对象
- 通过对象之间的关系和操作组织系统功能

它的优点主要包括：

- 可修改性和可扩充性较好
- 能利用继承等思想组织系统结构
- 有助于提高正确性和可靠性

例如某些现代系统会把共享资源抽象为对象来统一管理。

### 结构比较

#### 如何理解几种结构的关系

这些结构并不是简单“前者错误、后者正确”的替代关系，而更像是不同设计理念的体现：

- **整体 / 模块结构**：更强调集中组织和效率
- **分层结构**：更强调调用关系清晰、有序
- **微内核 / 客户服务器**：更强调模块化、扩展性、可靠性
- **对象模式**：更强调抽象与软件工程组织方式

实际系统中，常常会综合采用多种思想，而不是绝对只属于某一种结构。

## 本章小结

学完本章，你至少应该能比较清楚地回答下面这些问题：

1. 什么是操作系统？为什么说它既是资源管理者，也是扩展机器和用户接口？
2. 手工操作、单道批处理、多道批处理、分时系统之间的演化逻辑是什么？
3. 多道程序设计为什么能提高 CPU 利用率和系统吞吐量？
4. 操作系统的五大功能分别管理什么？
5. 并发、共享、虚拟、异步分别是什么意思？它们之间有什么联系？
6. 批处理、分时、实时、网络、分布式、多处理机、嵌入式操作系统各适合什么场景？
7. 整体结构、分层结构、微内核 / 客户服务器结构分别有什么特点？

::: details 参考答案

1. **操作系统是控制和管理计算机软硬件资源、组织系统工作流程并为用户和应用程序提供服务的程序集合**。之所以说它既是资源管理者，也是扩展机器和用户接口，是因为它**一方面统一管理 CPU、内存、设备和文件等资源**，**另一方面又向上提供文件、进程、系统调用、命令界面、图形界面等更易使用的抽象**。
2. 其演化逻辑是：**手工操作阶段人工低效、机器浪费严重**；**单道批处理通过自动成批执行减少人工切换**；**多道批处理通过让多个作业交替推进提高资源利用率**；**分时系统则进一步强调多用户交互和及时响应**。
3. 多道程序设计能提高 CPU 利用率和系统吞吐量，是因为**一个作业等待 I/O 时**，**CPU 可以切换去执行另一个作业**，从而**减少 CPU 空闲时间**，**让计算与 I/O 更充分地重叠进行**。
4. 五大功能分别是：**处理机管理负责进程控制、同步、通信和调度**；**存储管理负责内存分配、保护、地址映射和扩充**；**设备管理负责缓冲、分配和设备控制**；**文件管理负责文件存储、目录、读写和保护**；**用户接口负责命令接口、程序接口和图形接口**。
5. **并发**是**多个活动在同一时间段内推进**，**共享**是**多个进程共同使用系统资源**，**虚拟**是**把有限物理资源抽象成更灵活的逻辑资源**，**异步**是**并发活动推进的次序和速度不确定**。它们之间的关系是：**并发和共享最基础，虚拟体现系统对资源的抽象，异步则是并发环境下自然出现的运行特征**。
6. **批处理**系统适合**大批量、弱交互**的作业处理；**分时**系统适合**多用户交互式使用**；**实时**系统适合**对响应时限有严格要求**的控制场景；**网络**操作系统适合**联网通信与资源共享**；**分布式**操作系统适合**跨多台机器统一调度和透明共享资源**；**多处理机**操作系统适合**提升吞吐量和可靠性的多 CPU 环境**；**嵌入式**操作系统适合**资源受限、功能专用、常带实时要求的设备**。
7. **整体结构运行效率较高，但模块依赖复杂**；**分层结构调用关系清晰、便于设计和扩展，但可能带来效率损失**；**微内核 / 客户服务器结构把更多功能放到核外，扩展性和可靠性更好，但消息传递开销较大**。

:::

如果这些问题已经能自己解释清楚，就说明你已经具备继续学习后续章节的基础了。
