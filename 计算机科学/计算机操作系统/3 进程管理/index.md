# 第 3 章 进程管理

前两章分别回答了两个基础问题：

- 操作系统是什么，它有哪些目标、功能、特征和结构
- 用户和程序如何进入操作系统、请求操作系统服务

从这一章开始，我们进入操作系统课程中最核心、也最能体现“操作系统味道”的部分：**进程管理**。

为什么说这一章重要？因为现代操作系统几乎所有“动态运行”的问题，都和进程管理有关：

- 为什么同一台机器上可以同时运行多个程序
- 为什么多个程序一起跑时有时会出错、有时又能协同完成任务
- CPU 到底是按什么规则在不同任务之间切换的
- 线程为什么比进程更轻量
- 多个进程怎样同步、互斥和通信
- 死锁为什么会出现，又该怎样避免或处理

如果说第 1 章给你的是“静态全景图”，第 2 章给你的是“用户如何使用系统”，那么第 3 章讨论的就是：

**操作系统怎样组织和控制大量并发执行的活动。**

::: info 学习提示

本章最容易混淆的概念，主要有四组：

- **程序、进程、线程**
- **同步、互斥、通信**
- **作业调度、进程调度、对换调度**
- **死锁预防、死锁避免、死锁检测与解除**

学习时不要只背定义，更要抓住它们分别在解决什么问题。

:::

## 3.1 进程的概念

### 为什么要引入进程

#### 顺序程序的局限

在最简单的顺序执行环境中，一个程序从头到尾按既定顺序执行，通常具有以下特点：

- **顺序性**：程序中各条指令按既定次序依次执行
- **封闭性**：程序执行时与外界相对隔离，运行环境较稳定
- **可再现性**：在输入相同的情况下，多次运行通常得到相同结果

这类程序适合单任务、单道环境，但在多道程序系统中，情况就变了。

#### 多道程序系统带来的新问题

多道程序系统允许多个程序同时进入内存并交替推进，其目标是提高系统效率和资源利用率。这样一来，系统就出现了三个新特点：

- **并行性或并发性**：多个程序在时间上重叠推进
- **制约性**：程序之间会因为资源竞争或协作需求而相互影响
- **动态性**：程序在系统中的状态不断变化

这时，仅用“程序”这个静态概念已经不够了。

因为程序只是一组指令和数据的静态集合，它无法准确描述：

- 程序当前运行到哪里
- 它占有哪些资源
- 它现在是在运行、等待还是就绪
- 它和其他执行活动之间有什么关系

因此，操作系统引入了 **进程（Process）** 这个概念。

### 什么是进程

#### 进程的典型定义

教材和不同学者对进程有不同表述，但它们强调的核心意思是一致的。

常见的几种表述可以归纳为：

- 进程是 **程序在处理机上执行时所发生的活动**
- 进程是一个 **聚集相关资源的容器**
- 进程是 **具有独立功能的程序关于某个数据集合上的一次运行活动**
- 进程是系统进行 **资源分配和调度的独立单位**

因此，可以把进程理解为：

**进程不是静止的程序文件，而是“程序 + 数据 + 运行状态 + 管理信息”所形成的动态执行实体。**

#### 进程的组成

从操作系统管理角度看，一个进程通常由三部分组成：

- **程序**
- **数据**
- **进程控制块 PCB**

可以写成：

```text
进程 = 程序 + 数据 + PCB
```

其中，程序和数据说明“这个进程做什么”，PCB 说明“系统如何管理这个进程”。

### 进程的特征

这里强调了进程的几个关键特征。

#### 动态性

进程是程序的一次执行过程，它有创建、运行、等待、结束等生命周期，因此是动态变化的。

程序本身是静态的，而进程是动态的，这是两者最根本的区别之一。

#### 并发性

系统中往往存在多个已经开始但尚未结束的进程，它们在时间上交替推进，这就形成了并发。

#### 独立性

进程是资源分配和处理机调度的独立单位。操作系统可以独立地为它分配资源、保存现场、进行调度。

#### 异步性

多个并发进程按照各自独立且不可预知的速度向前推进，这种“推进速度和先后次序不确定”的特征就是异步性。

#### 交互性

很多进程并不是彼此完全无关，它们往往会：

- 共享资源
- 交换数据
- 相互等待
- 相互唤醒

这就形成了同步、互斥和通信等关系。

::: tip

“动态性、并发性、独立性、异步性”这几个词不要孤立记忆。

更好的理解方式是：

- 动态性：说明进程不是静态文件
- 独立性：说明进程是操作系统管理的基本对象
- 并发性：说明系统中会同时存在多个进程
- 异步性：说明这些进程推进的快慢和先后并不确定

:::

### 程序与进程的区别

程序和进程经常被初学者混用，但它们不是同一个概念。

| 对比维度 | 程序                 | 进程                         |
| -------- | -------------------- | ---------------------------- |
| 性质     | 静态                 | 动态                         |
| 存在形式 | 指令和数据的有序集合 | 程序的一次执行过程           |
| 生命周期 | 相对长期保存         | 有创建、运行、结束的生命周期 |
| 组成     | 程序代码和相关数据   | 程序、数据、PCB              |
| 管理对象 | 文件或程序实体       | 系统分配资源和调度的对象     |

还要注意两点：

- **一个程序可以对应多个进程**
- **一个进程在运行过程中也可能装入或调用多个程序段**

所以，“程序”更像蓝图，“进程”更像一次正在进行的施工过程。

### 进程的表示：PCB

#### 什么是 PCB

**PCB（Process Control Block，进程控制块）** 是操作系统为了管理进程而设置的一种专门数据结构。

它的作用是：

- 记录进程的外部特征
- 描述进程的动态变化过程
- 保存系统管理该进程所需要的信息

可以把 PCB 理解成“操作系统给进程建立的档案卡”。

#### 为什么说 PCB 是进程存在的唯一标志

从系统角度看，操作系统并不是通过“肉眼识别程序”来知道进程存在，而是通过 PCB 来管理进程。

因此需要强调的是：

- PCB 是进程存在的唯一标志
- 进程与 PCB 一一对应
- 系统依据 PCB 感知某个进程的存在

这句话的真正含义是：

**没有 PCB，操作系统就无法把某段代码和数据当成一个可管理的进程。**

#### PCB 中通常包含什么

PCB 的内容通常包括四大类信息。

##### 进程描述信息

例如：

- 进程标识符 `PID`
- 进程名
- 用户标识符 `UID`
- 进程组关系

##### 进程控制信息

例如：

- 当前状态
- 优先级
- 程序入口地址
- 外存地址
- 阻塞原因
- 同步和通信信息

##### 资源占有与使用情况

例如：

- 虚拟地址空间现状
- 打开文件列表
- 所拥有的资源

##### CPU 现场保护信息

例如：

- 程序计数器
- 各类寄存器值
- 栈指针

这些信息之所以重要，是因为进程会频繁被中断、挂起、恢复和切换，系统必须依赖 PCB 保存和恢复执行现场。

### PCB 的组织方式

系统中往往有很多进程，因此也会有很多 PCB。操作系统必须把它们组织起来，便于查找和管理。

#### 链表方式

把同一状态的进程 PCB 链接成表，例如：

- 就绪链表
- 阻塞链表

这种方式直观、便于按状态组织。

#### 索引表方式

也可以按状态建立索引表，由索引项指向对应 PCB。

这种方式在查找和集中管理上更方便。

这里还强调：

**PCB 表的大小决定了系统同时可存在的进程数上限，也就是系统的并发度。**

### 进程的状态与转换

#### 三种最基本的状态

最基本的进程状态有三种：

- **运行态（Running）**：进程占有 CPU，正在执行
- **就绪态（Ready）**：进程已具备运行条件，但暂时没有 CPU
- **等待态 / 阻塞态（Blocked）**：进程因等待某个事件发生而不能继续运行

其中最容易混淆的是：

- **就绪态** 不是“没准备好”，而是“已经准备好，只差 CPU”
- **阻塞态** 不是“暂时没被调度”，而是“即使给它 CPU，它也还不能运行”

#### 基本状态转换

最基本的状态转换包括：

- 就绪 -> 运行：调度程序分配 CPU
- 运行 -> 就绪：时间片用完或被抢占
- 运行 -> 阻塞：等待 I/O 或其他事件
- 阻塞 -> 就绪：等待事件发生

#### 五状态模型

在三状态基础上，通常再加入：

- **创建态（New）**
- **结束态（Exit）**

这样就得到五状态模型。

创建态表示进程刚被建立，但尚未完全具备运行条件；结束态表示进程已结束运行，系统正在回收资源。

#### 挂起与七状态模型

在更复杂的系统中，进程还可能因为内存紧张等原因被换出到外存，于是引入：

- **就绪挂起（Ready Suspend）**
- **阻塞挂起（Blocked Suspend）**

这里的“挂起”可以理解为：

**进程暂时不在主存中运行，而是被换到外存等待。**

这样系统就形成了七状态模型。

### 进程控制

进程状态不会自己变化，必须由操作系统通过专门机制来完成控制。

#### 为什么要用原语

进程创建、撤销以及状态转换要由具有特定功能的 **原语** 完成。

原语的特点是：

- 由若干条机器指令构成
- 完成一种特定功能
- 执行期间 **不可分割**

之所以要求“不可分割”，是因为进程控制本身涉及关键系统数据结构，如果中途被打断，系统状态可能被破坏。

#### 常见的进程控制原语

常见原语包括：

- **进程创建原语**
- **进程撤销原语**
- **进程状态转换原语**

### 进程的建立

#### 什么时候会创建进程

典型情形包括：

- 用户登录
- 作业调度
- 系统提供某种服务
- 应用程序主动提出请求

#### 创建进程的大致过程

创建新进程时，系统通常会：

1. 从空闲 PCB 集合中申请一个新的 PCB
2. 为其分配内部标识符
3. 填写各种控制参数
4. 建立必要的资源和地址空间
5. 把该进程置为就绪态
6. 将 PCB 插入就绪队列

#### 父子进程关系

进程创建常常形成父子关系，构成树形层次结构。

这种结构的好处是：

- 层次关系清晰
- 资源继承关系明确
- 任务分解更灵活

#### UNIX 中的典型例子

在 UNIX 中，`fork()` 用于创建新进程，`exec()` 常用于在创建后用新程序替换原地址空间。

这是经典的“先复制一个子进程，再让子进程去执行新程序”的模式。

::: details UNIX 例子想说明什么

`fork()` 和 `exec()` 的组合非常能体现进程管理思想：

- `fork()` 先创建新的执行实体
- `exec()` 再让这个执行实体去装入并运行新的程序

这说明“创建进程”和“装入程序”是两个相关但不完全相同的动作。

:::

### 进程的撤销

#### 哪些情况会导致进程终止

常见原因包括：

- 正常结束
- 异常结束
- 外界干预

#### 撤销策略

系统可以采用两种基本策略：

- 只撤销指定进程
- 同时撤销该进程及其所有子孙进程

#### 撤销过程

撤销进程时，系统通常需要：

1. 找到对应 PCB
2. 若该进程正在运行，则立即停止
3. 必要时撤销其子孙进程
4. 回收其占有的全部资源
5. 释放 PCB
6. 若需要，则重新调度

### 等待、唤醒、挂起与激活

#### 等待原语

当进程运行中因某条件不满足而不能继续推进时，就要进入等待态。

等待原语的工作通常包括：

- 修改进程状态为等待态
- 将其从运行态或就绪态转移到相应等待队列

#### 唤醒原语

当等待的事件发生后，操作系统需要把进程从等待队列中取出，改为就绪态，并重新插入就绪队列。

#### 挂起原语

挂起原语负责把活动就绪态改为静止就绪态，或把活动阻塞态改为静止阻塞态。

#### 激活原语

激活原语负责把进程从外存调回内存，并将其恢复为活动就绪态或活动阻塞态。

如果系统采用抢占式调度，那么被激活的高优先级进程还可能立即引发重新调度。

## 3.2 处理机调度

进程已经建立起来之后，接下来的核心问题就是：

**CPU 应该分给谁？什么时候分？按什么办法分？**

这就是处理机调度要解决的问题。

### 调度的基本概念

#### 调度在解决什么问题

调度的核心问题可以概括为三个角度：

- **WHAT**：按什么原则分配 CPU，也就是调度算法
- **WHEN**：何时进行 CPU 分配，也就是调度时机
- **HOW**：怎样进行 CPU 分配，也就是调度方式

#### 调度的总体目标

调度的目标通常包括：

- 提高 CPU 利用率
- 增大系统吞吐量
- 缩短响应时间

这三个目标并不总能同时最优，因此调度设计本质上是一种权衡。

### 调度的层次

处理机调度通常分为三级。

#### 高级调度

高级调度也称 **作业调度**。

它关注的是：

- 哪些作业可以从输入井或后备队列进入内存
- 哪些作业获得开始执行的机会

它更偏向从用户工作流程和系统整体负载角度分配资源，时间尺度通常较长。

#### 中级调度

中级调度主要与 **内外存交换** 有关。

它负责决定：

- 哪些进程要换出到外存
- 哪些挂起进程要重新换入内存

因此它和存储管理联系紧密。

#### 低级调度

低级调度也叫 **进程调度**，是最频繁发生的调度。

它关注的是：

**从就绪队列中选出一个进程，把 CPU 分给它运行。**

本章中大多数“调度算法”讨论的都是低级调度。

### 调度性能准则

评价一个调度算法好不好，不能只看“感觉”，而要看它在不同指标上的表现。

#### 面向用户的准则

##### 周转时间

周转时间指作业从提交到完成所经历的总时间。

它包括：

- 在后备队列中的等待
- 在就绪队列中的等待
- 在阻塞队列中的等待
- 真正在 CPU 上执行的时间

对批处理系统来说，周转时间是很重要的指标。

##### 带权周转时间

带权周转时间把周转时间和实际运行时间结合起来考察，更能反映“一个作业等待了多久相对于它本来需要执行多久”的关系。

##### 响应时间

响应时间指用户发出请求到系统给出首次响应之间的时间。

它在分时系统和交互式系统中特别重要。

##### 截止时间

截止时间在实时系统中特别关键，系统必须在规定时刻之前开始或完成某项任务。

##### 公平性与优先级

调度不能让某些进程长期得不到运行机会，同时系统又常常需要照顾更重要、更紧急的任务，因此公平与优先级之间也要平衡。

#### 面向系统的准则

##### 吞吐量

吞吐量指单位时间内完成的作业数。

它在批处理系统中通常非常重要。

##### CPU 利用率

CPU 利用率反映处理机忙碌程度，是衡量系统资源利用情况的重要指标。

##### 设备均衡利用

如果 CPU 型作业和 I/O 型作业搭配得好，系统整体资源利用会更均衡。

#### 算法本身的准则

调度算法除了结果好，还应该：

- 易于实现
- 本身开销不要过大

否则调度器本身消耗过多 CPU，也会影响系统效率。

### 进程调度的时机

系统并不是随时都在调度。常见调度时机包括：

- 当前运行进程正常结束
- 当前运行进程异常终止
- 时间片用完
- 运行进程提出 I/O 请求而阻塞
- 运行进程执行原语后进入阻塞态
- 更高优先级进程进入就绪队列

### 调度方式

#### 非抢占式调度

非抢占式调度中，一个进程一旦获得 CPU，只要它自己没有结束或阻塞，就会一直运行下去。

这种方式实现简单，但对响应速度不友好。

#### 抢占式调度

抢占式调度允许系统在必要时强行剥夺当前进程的 CPU，把处理机交给更重要或更紧急的进程。

它更适合：

- 分时系统
- 实时系统
- 优先级调度

### 常见调度算法

#### 先来先服务 FCFS

FCFS（First Come First Service）按进程进入就绪队列的先后顺序分配 CPU。

它的特点是：

- 思想最简单
- 一般为非抢占式
- 对长作业较有利
- 对短作业和 I/O 型作业不利

其典型问题是 **护航效应（Convoy Effect）**：一个长作业在前面，会导致后面一串短作业都长时间等待。

#### 最短作业优先 SJF

SJF（Shortest Job First）优先调度预计运行时间最短的作业或进程。

它的优点是：

- 在给定进程集合下，平均周转时间通常较小

但它也有明显问题：

- 对长作业不友好，可能长期得不到执行
- 需要估计运行时间，而估计往往不准确
- 无法体现任务紧迫程度

#### 最短剩余时间优先 SRTF

SRTF 可以看作 SJF 的抢占式版本。

一旦有新的更短任务到达，它可以抢占当前进程。

这种方式通常进一步改善平均等待时间，但实现复杂度更高，也更依赖准确估计。

#### 最高响应比优先 HRRN

HRRN（Highest Response Ratio Next）的响应比为：

```text
R = (等待时间 + 要求执行时间) / 要求执行时间
```

它可以看作 FCFS 和 SJF 的折中：

- 短作业更有优势
- 长时间等待的作业，其响应比会逐渐提高

因此它比纯 SJF 更不容易导致长作业长期饥饿。

#### 时间片轮转 RR

RR（Round Robin）是分时系统中非常经典的算法。

其基本思想是：

1. 把所有就绪进程按 FCFS 排成队列
2. 每次把 CPU 分给队首进程一个时间片
3. 时间片用完后，如果进程还没结束，就把它送到队尾
4. 再调度新的队首进程

它的优点是：

- 公平
- 响应快
- 适合交互式环境

但时间片长度选不好会出问题。

##### 时间片过长

如果时间片太长，RR 会逐渐退化成 FCFS，响应时间变差。

##### 时间片过短

如果时间片太短，进程切换次数会急剧增加，调度开销变大。

因此时间片的选择要综合考虑：

- 就绪进程数目
- 用户对响应时间的要求
- 系统处理能力

#### 基于优先级的调度

优先级调度为每个进程设置优先数，并总是优先选择优先级最高者。

它可以分为：

- **静态优先级**：创建时确定，运行中不变
- **动态优先级**：运行过程中根据行为动态调整

动态优先级常用于：

- 提高长期等待进程的机会
- 降低持续占用 CPU 的进程优先级

这样有助于减轻饥饿问题。

#### 多级队列算法

多级队列算法把就绪队列再划分成多个子队列，例如：

- 系统进程队列
- 交互进程队列
- 批处理进程队列

不同队列可以有不同的：

- 优先级
- 时间片长度
- 调度策略

这种做法适合对不同类型任务区别对待。

#### 多级反馈队列算法

多级反馈队列是实际系统中非常重要的一类综合算法。

它结合了：

- 时间片轮转
- 优先级调度
- 动态反馈调整

其典型思想是：

1. 设置多个优先级不同的就绪队列
2. 新进程先进入高优先级队列
3. 若在较短时间片内没执行完，就逐步下移到较低优先级队列
4. 较低优先级队列拥有更长时间片
5. I/O 型或交互型进程因为经常提前阻塞，往往留在高优先级队列

它的优点在于：

- 照顾短作业
- 照顾 I/O 型进程
- 对计算型进程给予更长时间片
- 无需事先准确估计运行时间

这也是为什么很多现代通用系统会采用类似多级反馈队列的思想。

::: tip

如果把常见算法按“更偏向谁”来粗略理解：

- `FCFS`：简单，但不够灵活
- `SJF/SRTF`：偏向短作业
- `RR`：偏向交互公平
- `优先级调度`：偏向关键任务
- `多级反馈队列`：试图综合照顾多种目标

:::

### 实时调度

实时系统不仅要求“结果对”，还要求“结果在规定时间内到达”。

#### 实时任务的分类

实时任务常分为：

- **周期性任务**
- **非周期性任务**

还可分为：

- **硬实时任务**：必须严格满足截止时间
- **软实时任务**：最好满足，但偶尔错过可容忍

#### 实现实时调度的基本条件

实现实时调度的基本条件包括：

- 提供必要的信息：就绪时间、截止时间、处理时间、资源要求、优先级
- 系统本身必须是可调度的
- 采用抢占式调度机制
- 具有快速切换能力

其中“可调度”可以理解为：

**从总处理能力上看，系统要真的来得及完成这些实时任务。**

#### 实时调度中的优先级反转

优先级反转指：

**高优先级任务反而等待低优先级任务释放资源。**

这会严重影响实时性。

对应的解决思想之一是 **优先级继承**：

低优先级进程一旦占有了高优先级进程所需的资源，就临时继承高优先级，直到释放资源为止。

#### 常见实时调度算法

##### EDF：最早截止期优先

EDF（Earliest Deadline First）总是优先调度截止时间最早的任务。

它直接体现“谁更急先做谁”的思想。

##### LLF：最低松弛度优先

LLF（Least Laxity First）根据任务的松弛度决定优先级。

松弛度可以理解为：

**距离截止时间还剩多少“可拖延空间”。**

谁的松弛度最低，谁就最紧急。

### 多处理机调度

当系统中不止一个 CPU 时，调度问题会进一步复杂化。

#### 与单处理机调度的区别

多处理机调度更强调：

- 整体运行效率
- 多处理机之间的负载均衡
- 对共享 OS 数据结构的互斥访问
- 线程作为更细粒度的调度单位

#### SMP：对称式多处理

SMP（Symmetric Multiprocessing）中，各处理机地位相对对等。

常见做法包括：

- **静态分配**：每个 CPU 拥有自己的就绪队列
- **动态分配**：多个 CPU 共享公共就绪队列
- **自调度**：各 CPU 从公共队列中自行取任务

其优点通常是：

- 利用率较高
- 负载均衡较好

但共享队列访问需要同步控制。

#### ASMP：非对称式多处理

ASMP（Asymmetric Multiprocessing）中，通常由主处理机负责管理和分派，从处理机按分配执行任务。

它实现上较集中，但扩展性和灵活性不一定如 SMP。

#### 成组调度

成组调度（Gang Scheduling）把一个并行任务中的一组线程一次性同时分派到多个处理机上运行。

这样做的好处是：

- 提高这些线程协同推进的效率
- 减少相互等待带来的阻塞

#### 专用处理机调度

专用处理机调度为某进程中的线程固定分配处理机直到完成。

其优点是切换开销小，但缺点是：

- 一旦线程阻塞，CPU 可能闲置

因此它更适用于处理器数量很多的高度并行系统。

## 3.3 线程及其管理

进程已经能描述并发执行了，为什么还要引入线程？

因为人们逐渐发现：**把“资源拥有者”和“调度执行者”完全绑在一起，代价太大。**

### 为什么要引入线程

#### 传统进程的不足

传统进程同时承担两个角色：

- 资源分配单位
- CPU 调度单位

这样做虽然统一，但缺点明显：

- 创建进程开销大
- 撤销进程开销大
- 进程切换开销大
- 并发粒度不够细

#### 线程的基本思想

线程把原来进程的两个属性分开处理：

- **进程** 主要作为资源拥有单位
- **线程** 主要作为 CPU 调度单位

这样就形成了“进程负责资源组织，线程负责执行推进”的结构。

### 什么是线程

线程可以理解为：

**进程内一个相对独立、可调度的执行单元。**

它是：

- 进程中的一个运行实体
- CPU 调度的基本单位
- 共享所属进程大部分资源的执行流

在多线程机制下：

- 一个进程可以包含多个线程
- 这些线程共享进程的地址空间、文件和数据
- 但每个线程仍有自己的运行状态、寄存器上下文和栈

### 引入线程的好处

引入线程的好处主要包括：

- 创建线程比创建进程更快
- 终止线程比终止进程更快
- 线程切换开销更小
- 同一进程内线程共享内存和文件，通信更方便
- 更适合多处理机环境

这意味着：

**线程让并发更轻量，也让程序内部并行更自然。**

### 进程与线程的比较

| 对比维度       | 进程                | 线程                   |
| -------------- | ------------------- | ---------------------- |
| 基本角色       | 资源拥有单位        | CPU 调度单位           |
| 地址空间       | 彼此独立            | 同一进程内共享         |
| 创建与撤销开销 | 较大                | 较小                   |
| 切换开销       | 较大                | 较小                   |
| 通信方式       | 通常需借助 IPC 机制 | 同进程内可直接共享数据 |
| 可见性         | 对系统是独立实体    | 对外通常依附于所属进程 |

要特别注意：

- **进程不是线程的简单“放大版”**
- **线程也不是“更小的进程”这么简单**

两者解决的是不同层次的问题。

### 多线程环境下的进程

在多线程操作系统中，进程通常具有以下属性：

- 是系统资源分配单位
- 可以包含多个线程
- 自身不一定直接作为可执行实体

线程之间虽然共享数据很方便，但也带来一个新问题：

**共享越方便，就越需要同步和互斥机制保证一致性。**

### 线程的实现方式

这里重点讲了三类实现思路：

- 用户级线程 `ULT`
- 内核级线程 `KLT`
- 轻量级进程 `LWP` 作为中间层

### 用户级线程 ULT

#### 基本思想

用户级线程完全在用户空间中实现，线程管理由线程库或运行时系统完成，内核只看见进程，不直接看见线程。

#### 优点

- 线程创建和切换不需要进入内核
- 开销小
- 可针对应用定制调度策略
- 可移植性较好

#### 缺点

- 一个线程做阻塞式系统调用时，整个进程可能都被阻塞
- 内核只把 CPU 分给进程，不能让同一进程的多个用户线程真正并行跑在多个处理器上

### 内核级线程 KLT

#### 基本思想

内核级线程由操作系统内核直接管理，内核知道线程的存在，并对线程进行调度。

#### 优点

- 一个线程阻塞，不会必然拖住同进程其他线程
- 可在多处理器上同时调度同一进程中的多个线程
- 更适合真正的内核级并发

#### 缺点

- 线程切换需要进入内核
- 开销比用户级线程大

### 轻量级进程 LWP

有些系统不直接采用“纯 ULT”或“纯 KLT”的单一路线，而是在两者之间引入轻量级进程 `LWP`。

可以把 LWP 理解为：

**连接用户线程和内核线程的中间执行载体。**

这样系统就能兼顾：

- 用户线程的灵活与低开销
- 内核线程的并行与内核支持

### 线程的状态

线程和进程一样，也会在运行过程中发生状态变化。

最基本的线程状态通常包括：

- **执行态**
- **就绪态**
- **阻塞态**

线程描述信息通常也包括：

- 线程标识符
- 寄存器状态
- 栈
- 优先级
- 线程专有存储

### 线程的创建与终止

#### 创建

一个应用启动时，通常先有一个初始化线程，它再根据需要创建其他线程。

创建完成后，系统会返回线程标识符，供后续管理使用。

#### 终止

线程终止的常见原因包括：

- 自己完成工作后主动退出
- 运行中出错
- 被其他线程或系统强行终止

### 线程的系统实例

#### Solaris

以 Solaris 为例，它同时支持：

- 用户线程
- 轻量级进程 `LWP`
- 内核线程

这种分层结构非常有代表性，有助于理解多层线程映射关系。

#### Windows NT

Windows NT 中的线程状态例如：

- 就绪态
- 备用态
- 运行态
- 等待态
- 转换态
- 终止态
- 初始化态

这说明实际系统中的线程状态通常比课堂上的“三状态”更细致。

### 作业、程序、进程和线程的对比

到这里，可以把几个常见概念放在一起区分：

| 概念 | 主要含义                     |
| ---- | ---------------------------- |
| 作业 | 用户提交给系统的一项完整任务 |
| 程序 | 一组静态指令和数据           |
| 进程 | 资源分配和调度的动态管理实体 |
| 线程 | 进程内部可调度的执行单元     |

## 3.4 进程同步、互斥与通信

多个进程并发执行以后，系统面临的关键问题不只是“谁先跑”，还包括：

- 它们之间是否要合作
- 是否会争用同一资源
- 怎样保证结果正确
- 怎样交换信息

这就引出了同步、互斥和通信。

### 进程之间为什么会有制约关系

进程间制约关系通常可分为两类。

#### 直接制约关系

直接制约关系通常来自 **协作**。

例如：

- 生产者把数据放入缓冲区
- 消费者必须等数据到来后才能取走

这类关系对应的是 **同步**。

#### 间接制约关系

间接制约关系通常来自 **竞争共享资源**。

例如：

- 两个进程都想访问打印机
- 两个线程都想改同一个共享变量

这类关系对应的是 **互斥**。

### 同步与互斥

#### 什么是同步

进程同步是指多个相互协作的进程之间，为了在某些确定点上协调工作，而对执行先后顺序进行约束的关系。

说得直白一点：

**同步强调“该等就得等，该继续时再继续”。**

#### 什么是互斥

进程互斥指多个进程不能同时使用同一临界资源，只能一个进程先用，其他进程等待。

说得直白一点：

**互斥强调“同一时刻只能一个进程进去”。**

#### 两者的关系

这里特别指出：

- 互斥实际上是同步的一种特殊情况
- 但两者关注点不同

可以这样记：

- **同步**：强调次序关系
- **互斥**：强调独占关系

### 临界资源与临界区

#### 临界资源

**临界资源（Critical Resource）** 指一次只允许一个进程访问的资源。

例如：

- 打印机
- 某共享变量
- 某共享缓冲区

#### 临界区

**临界区（Critical Section）** 指进程中访问临界资源的那段代码。

一个典型访问过程通常可分为：

- **进入区**：申请进入临界区
- **临界区**：真正访问共享资源
- **退出区**：释放临界资源
- **剩余区**：其余无关代码

#### 同步机制应遵循的准则

同步机制通常要满足四条经典准则：

- **空闲让进**
- **忙则等待**
- **有限等待**
- **让权等待**

可以分别理解为：

- 资源空闲时，应允许进程进入
- 资源忙时，其他进程应等待
- 不应让某个进程无限期等下去
- 等待时最好让出 CPU，而不是一直空转

### 软件方法解决互斥

在没有专门硬件支持时，人们曾尝试用纯软件方法解决互斥问题。

#### 单标志法

单标志法用一个变量 `turn` 指示轮到谁进入临界区。

它能保证某种程度上的互斥，但强制双方轮流进入，不满足灵活性要求。

#### 双标志先检查法

双标志先检查法让每个进程先看对方是否在临界区，再设置自己的标志。

问题是：两个进程可能几乎同时看到对方“不在”，结果同时进入临界区。

#### 双标志后检查法

双标志后检查法让每个进程先声明“我要进”，再看对方是否也想进。

问题是：双方可能都先把标志置真，结果谁都进不去。

#### Peterson 算法

Peterson 算法把“标志位”和“turn 变量”结合起来，是经典的软件互斥算法。

它通过“先表明意图，再谦让，再检查”的方式解决了前面几种简单算法的缺陷。

它的教学意义很大，因为它能帮助理解：

- 并发控制为什么困难
- 仅靠直觉写同步代码为什么容易出错

但它也有现实局限，例如忙等待问题。

### 硬件方法解决互斥

#### 关中断

一种最直接的想法是在进入临界区时关闭中断，退出时再开中断。

这样当前进程不会被切走，从而保证临界区不被打断。

但这种方法问题很多：

- 代价高
- 影响系统并发能力
- 对多处理机无效

因此它只能在很特殊的场合使用。

#### Test-and-Set 指令

硬件可以提供原子性的“测试并置位”指令，用一个指令周期同时完成检查和修改，从而避免竞态条件。

#### Swap 指令

Swap 指令通过原子交换两个变量实现互斥。

#### 锁

操作系统还可进一步封装出 `lock` / `unlock` 原语，让用户程序通过系统支持来实现互斥。

#### 硬件方法的优缺点

优点：

- 适用于单处理器和多处理器
- 原理简单
- 易于验证正确性

缺点：

- 通常仍是忙等待
- 可能造成饥饿
- 某些情况下仍可能引发死锁

### 信号量与 P、V 原语

信号量是操作系统课程中最核心的同步工具之一。

#### 什么是信号量

信号量由 Dijkstra 提出，是一种解决同步和互斥问题的有效工具。

从数据结构上看，信号量通常包括：

- 一个整型值 `value`
- 一个等待队列指针

#### 信号量值的物理意义

- `S > 0`：表示还有 `S` 个可用资源
- `S = 0`：表示没有空闲资源
- `S < 0`：表示有 `|S|` 个进程在等待该资源

#### P 原语

P 操作又叫 `wait`，本质上表示：

- 申请一个资源
- 若资源不足，则把进程阻塞并加入等待队列

#### V 原语

V 操作又叫 `signal`，本质上表示：

- 释放一个资源
- 若有进程在等待，则唤醒其中一个

#### 为什么说 P、V 是原语

因为对信号量的修改必须是不可分割的原子操作，否则多个进程同时改信号量时就会出错。

#### 使用信号量时的几个重要规则

- 信号量必须初始化，而且只初始化一次
- P 和 V 通常成对出现
- 实现互斥时，P 和 V 往往在同一进程中配对
- 实现同步时，P 和 V 往往分布在不同进程中
- 当多个 P 操作组合时，顺序可能直接决定系统是否死锁

::: warning 一个很常见的坑

P 操作的顺序常常比 V 操作更敏感。

例如在生产者消费者问题中，如果先对互斥信号量做 `P`，再对同步信号量做 `P`，就可能把别人需要的互斥资源拿住不放，从而造成死锁。

:::

### 用信号量实现互斥与同步

#### 互斥

对于互斥问题，通常把 `mutex` 初始化为 `1`。

所有要访问临界区的进程都执行：

```text
P(mutex)
临界区
V(mutex)
```

这表示：

- 进入前先申请独占权限
- 离开时再释放

#### 同步

对于同步问题，常把同步信号量初值设为 `0`。

例如进程 A 必须先于进程 B 运行完某一步，则可让：

- A 在完成后执行 `V(S)`
- B 在继续前执行 `P(S)`

这样 B 若先到达，就会在 `P(S)` 处等待，直到 A 发出信号。

### 经典进程同步问题

经典问题的价值，不只是“会做题”，更重要的是它们浓缩了各种并发控制关系。

### 生产者-消费者问题

#### 问题本质

生产者不断向有界缓冲区放产品，消费者不断从缓冲区取产品。

这里同时存在两类关系：

- **同步**：空缓冲区不能取，满缓冲区不能放
- **互斥**：对缓冲区的访问必须互斥

#### 典型信号量设置

通常设置：

- `empty`：空缓冲区数，初值为 `N`
- `full`：满缓冲区数，初值为 `0`
- `mutex`：缓冲区互斥访问，初值为 `1`

#### 关键理解

这个问题最典型地说明：

**同步和互斥往往不是二选一，而是同时存在。**

### 读者-写者问题

#### 问题本质

多个读者可以同时读共享数据，但：

- 读和写不能同时发生
- 写和写也不能同时发生

这意味着：

- `读-读` 可以并发
- `读-写` 必须互斥
- `写-写` 也必须互斥

#### 读者优先

先看第一类读者写者问题，即读者优先。

其特点是：

- 只要当前有读者在读，后来的读者通常仍可继续进入
- 这样读效率高
- 但写者可能长期等待

#### 写者优先

为了避免写者长期饥饿，可以再增加同步控制，使一旦有写者到达，后续读者必须等待。

这个变形说明：

**同步算法不只有“对不对”，还涉及“更偏向谁”。**

### 哲学家就餐问题

#### 问题本质

5 个哲学家围坐圆桌，吃饭时必须同时拿到左右两只筷子。

若每个人都先拿左边，再等右边，就可能形成环路等待，最终死锁。

#### 它想说明什么

这个问题的价值在于揭示：

- 资源分配顺序不当会导致死锁
- 单个进程代码看起来没问题，多个并发实体组合起来却会出问题

#### 常见改进思路

常见的避免死锁思路包括：

- 最多只允许 4 个哲学家同时尝试取筷子
- 仅当左右筷子都可用时才允许拿
- 规定奇偶号哲学家按不同顺序取筷子

这些方法的本质，都是在破坏死锁形成条件。

### 进程通信 IPC

同步和互斥主要解决“怎样协调推进”，而通信更强调“怎样交换较多数据”。

#### 低级通信与高级通信

低级通信通常只能传递少量状态和控制信息，速度快但表达能力有限。

高级通信则支持较高效率、较大量的数据交换，更适合实际应用。

### 进程通信的主要类型

进程通信主要可分为三类：

- **共享存储器系统**
- **消息传递系统**
- **管道通信**

### 共享存储器系统

#### 基本思想

在共享存储方式下，多个进程通过读写同一块共享数据区来交换信息。

这种方式的优点是：

- 速度快
- 数据量大

但要注意：

**共享存储只提供“能看到同一块数据”的条件，并不自动解决同步和互斥问题。**

这些问题仍要由进程自己负责。

#### UNIX 中的共享存储区

以 UNIX 为例，共享存储相关系统调用包括：

- `shmget`
- `shmat`
- `shmdt`
- `shmctl`

它们分别用于创建、连接、拆除和控制共享存储区。

### 消息传递系统

消息传递系统以 **消息** 为基本交换单位。

它主要分为两种方式。

#### 直接通信

发送者直接把消息送给接收者，通常挂到接收进程的消息队列上。

这种方式的特点是：

- 发送方和接收方需要明确彼此身份
- 通常要求通信双方都存在

#### 间接通信

间接通信通过信箱、邮箱或端口等中间实体转发消息。

这种方式更灵活，因为发送者和接收者不一定必须同时直接知道对方。

### 消息缓冲通信

消息缓冲通信是一种典型的直接通信方式。

其基本过程通常是：

1. 系统维护空白缓冲区池
2. 发送者申请缓冲区并写入消息
3. 系统把缓冲区挂到接收者消息队列
4. 接收者从消息队列取出消息

为了保证正确性，发送和接收过程通常也需要借助：

- 互斥信号量
- 消息计数信号量

### 信箱通信

#### 基本思想

信箱通信把“邮箱”作为中间站：

- 发送进程把信件送到信箱
- 接收进程从信箱取出信件

#### 典型规则

- 信箱满时，发送者等待
- 信箱空时，接收者等待

#### 信箱的分类

典型信箱有三类：

- **私用信箱**
- **公用信箱**
- **共享信箱**

它们的区别主要在于拥有权和访问权限不同。

### 管道通信

#### 什么是管道

管道是连接一个写进程和一个读进程的共享文件或内核缓冲机制。

其特点是：

- 写端写入数据流
- 读端按顺序读出数据流
- 非常适合流式数据传输

#### 管道的几个关键特点

- 需要同步：写后读、读后写
- 通常也涉及互斥访问
- 无名管道常用于有亲缘关系的进程之间
- 管道本质上通常是单向的，双向通信常需两个管道

#### 无名管道与有名管道

- **无名管道**：通常通过 `pipe()` 创建，依赖亲缘进程关系
- **有名管道（FIFO）**：在文件系统中有名字，可按名访问

#### 管道破裂

如果读端关闭，写端继续写，就可能收到 `SIGPIPE` 信号，这就是所谓的“管道破裂”。

## 3.5 死锁

当多个进程并发执行、竞争资源并相互等待时，系统可能进入一种非常危险的状态：谁都不往前走了。

这就是死锁。

### 死锁举例与系统模型

#### 从直观例子理解死锁

可以用交通路口、外设竞争、消息收发等例子来理解死锁。

这些例子虽然表面形式不同，但本质上都体现为：

- 每个参与者都已占有部分资源
- 每个参与者又都在等待别人占有的资源
- 谁也不主动让步

#### 系统模型

系统中有一定数量的资源，分布在多个竞争进程之间。

资源可分为：

- **物理资源**：CPU、内存、I/O 设备等
- **逻辑资源**：文件、信号量等

进程对资源的正常使用过程通常是：

```text
申请 -> 使用 -> 释放
```

#### 可重用资源与可消费资源

还可以把资源区分为两种：

- **可重用资源**：如处理器、设备、文件等，用完后可再次分配
- **可消费资源**：如消息、中断、缓冲区中的某些信息，被消费后就不再存在

这说明死锁不只发生在“设备占用”场景，也可能发生在通信场景。

### 什么是死锁

死锁的规范定义可以概括为：

**一组进程中，每个进程都无限等待该组中其他进程所占有的资源，从而永远无法继续推进，这种现象称为死锁。**

死锁一旦发生，后果可能包括：

- 系统资源被长期占住
- 吞吐量下降
- 某些服务彻底停顿
- 严重时导致系统崩溃

### 死锁产生的原因

可以看到，死锁的常见原因主要有：

- 资源不足引起竞争
- 并发执行顺序不当
- 请求和释放资源顺序设计不合理

其中非常重要的一点是：

**不是所有资源竞争都会死锁，但不合理的资源分配顺序非常容易把系统推向死锁。**

### 产生死锁的四个必要条件

通常采用 Coffman 的四个必要条件来描述死锁。

#### 互斥条件

资源一次只能被一个进程占有。

#### 占有且等待条件

进程已经占有至少一个资源，同时又申请新的资源并等待。

#### 不可剥夺条件

进程已获得的资源在未使用完前不能被强行剥夺。

#### 环路等待条件

存在一个进程-资源的循环等待链。

这四个条件必须同时成立，死锁才可能发生。

### 处理死锁的基本方法

处理死锁通常有四类基本策略：

- **忽略死锁**
- **死锁预防**
- **死锁避免**
- **死锁检测与恢复**

#### 忽略死锁

也就是所谓“鸵鸟策略”，认为死锁不常发生，系统不专门处理。

某些通用系统确实会在某些场景中采取接近这种思想的做法。

#### 死锁预防

通过事先限制资源申请方式，破坏四个必要条件中的至少一个，从根本上不让死锁发生。

#### 死锁避免

不预先完全禁止必要条件存在，而是在每次分配资源时动态判断系统是否会进入危险状态。

#### 死锁检测与恢复

允许系统进入死锁风险状态，但要周期性检测，一旦发现死锁，再采取措施恢复。

### 死锁预防

#### 破坏互斥条件

对某些资源可以通过共享方式来减少互斥，例如某些可共享文件场景。

但很多资源天然不可共享，因此这条通常不普适。

#### 破坏占有且等待条件

典型方法有两种：

- 进程运行前一次性申请全部资源
- 若申请新资源失败，则先释放已占有资源，稍后重新申请

优点是简单、安全；
缺点是：

- 资源浪费大
- 可能导致进程长期等待
- 容易产生饥饿

#### 破坏不可剥夺条件

一种思路是允许系统在必要时强制收回资源。

这对 CPU、内存等某些资源相对可行，但对打印机、磁带机等资源往往难以实施。

#### 破坏环路等待条件

常用方法是 **资源有序分配法**：

- 给所有资源类型编号
- 进程必须按编号递增顺序申请资源

这样可以从结构上避免形成环路。

但缺点是：

- 不够灵活
- 对资源类型变动不友好
- 可能造成资源利用下降

### 死锁避免

#### 安全状态与不安全状态

死锁避免的核心概念是 **安全状态**。

如果系统存在一个安全序列，使得所有进程按该序列依次都能最终得到所需资源并完成，那么系统就是安全的。

需要特别记住：

- **安全状态一定没有死锁**
- **不安全状态不一定已经死锁，但有可能走向死锁**

#### 死锁避免在做什么

每当进程提出资源请求时，系统不是立刻分配，而是先“试探性地假设已经分配”，然后检查：

- 分配后系统是否仍处于安全状态

如果安全，才正式分配；否则就让进程等待。

### 银行家算法

银行家算法是最经典的死锁避免算法。

#### 类比理解

操作系统像银行家，资源像周转资金，进程像借款客户。

银行家之所以不会随便把钱全借出去，是因为它必须保证：

**即使把钱借出去，未来仍有办法让所有客户最终都能还款并完成。**

#### 关键数据结构

银行家算法常用以下数据结构：

- **Available**：当前各类可用资源向量
- **Max**：各进程对各类资源的最大需求矩阵
- **Allocation**：当前已分配矩阵
- **Need**：尚需资源矩阵
- **Request**：某进程当前提出的请求向量

其中：

```text
Need = Max - Allocation
```

#### 分配检查的基本步骤

当进程 `Pi` 提出请求时，系统一般按下面思路检查：

1. 判断 `Request <= Need`
2. 判断 `Request <= Available`
3. 先试探性分配资源
4. 运行安全性算法
5. 若仍安全，则正式分配；否则回滚并让其等待

#### 安全性算法的核心思路

安全性算法通常维护：

- `Work`：当前可供继续使用的资源
- `Finish`：某进程能否顺利完成的标志

不断寻找一个满足 `Need <= Work` 的进程，假设其完成并释放资源，再更新 `Work`，看是否最终所有进程都能标记完成。

如果能，就说明存在安全序列。

#### 银行家算法的优点与局限

优点：

- 比死锁预防限制更少
- 能避免死锁

局限：

- 需要事先知道最大资源需求
- 进程数量和资源种类相对固定时更适用
- 不适合进程间有复杂同步依赖的场景

### 死锁检测

如果系统不做预防或避免，而是允许死锁发生，就必须能检测死锁。

#### 什么时候检测

常见的检测时机有：

- 进程进入等待时就检测
- 定时检测
- 资源利用率明显下降时检测

#### 资源分配图

资源分配图由两类结点构成：

- **进程结点**
- **资源结点**

两类边包括：

- **申请边**：进程请求资源
- **分配边**：资源已分配给进程

#### 资源分配图中的环

要注意一个经典结论：

- 若图中 **没有环**，则一定没有死锁
- 若图中 **有环**，则可能有死锁
- 当每类资源都只有一个实例时，**有环就是死锁成立的充分必要条件**

#### 图的化简

还可以用资源分配图的化简思想来判断死锁。

如果不断化简后：

- 所有进程结点都能变成孤立点，则图可完全化简，说明没有死锁
- 若最终仍存在不可化简部分，则说明死锁存在

### 死锁的解除

检测到死锁后，系统还要恢复运行。

典型方法包括：

- **重新启动**
- **撤销进程**
- **剥夺资源**
- **进程回退**

#### 终止进程

可选策略包括：

- 一次终止所有死锁进程
- 一次只终止一个，直到打破死锁环

若按“最小代价”原则选牺牲者，可参考：

- 已消耗 CPU 时间最少
- 已产生输出最少
- 剩余运行时间最长
- 当前占有资源最少
- 优先级最低

#### 资源抢占

系统也可从某些进程手中逐步强占资源，直到打破死锁环。

但这会带来两个难题：

- 抢谁、抢哪些资源、按什么顺序抢
- 如何避免总是抢同一个进程导致饥饿

#### 回滚

如果系统支持检查点机制，也可把进程回退到较早状态后重新执行。

这种方式思想优雅，但实现复杂度较高。

::: details 如何整体理解死锁处理四件事

- **预防**：从制度上卡死，让死锁条件根本凑不齐
- **避免**：不提前卡死，但每次分配前都要先算一遍风险
- **检测**：先不限制，出问题后再查出来
- **解除**：已经查出来了，想办法把系统救回来

这四类方法不是谁“绝对更高级”，而是取决于系统目标、代价和使用场景。

:::

## 本章小结

学完本章后，你至少应该能把下面这些问题讲清楚：

1. 为什么在多道程序系统中，仅用“程序”这个概念已经不够，必须引入“进程”？
2. 进程有哪些关键特征？程序和进程有什么本质区别？
3. PCB 为什么是进程存在的唯一标志？它通常保存哪些信息？
4. 三状态、五状态、七状态模型分别在描述什么？
5. 处理机调度要回答哪三个核心问题？高级、中级、低级调度各自负责什么？
6. FCFS、SJF、RR、优先级调度、多级反馈队列各适合什么场景？它们各自偏向什么目标？
7. 线程为什么被引入？它和进程在资源拥有与调度地位上有什么区别？
8. 用户级线程、内核级线程、轻量级进程分别有什么特点和适用性？
9. 同步、互斥、临界资源、临界区分别是什么意思？它们之间是什么关系？
10. 为什么说信号量和 P、V 原语是操作系统中非常基础的一套同步工具？
11. 生产者消费者、读者写者、哲学家就餐这三个经典问题分别在说明什么？
12. 死锁为什么会发生？预防、避免、检测与解除分别是什么思路？

::: details 参考答案提纲

1. 因为程序只是静态指令集合，不能描述资源占有、运行状态和并发关系；进程才是动态执行实体和资源分配调度单位。
2. 进程具有动态性、并发性、独立性和异步性；程序是静态的，进程是程序的一次动态执行。
3. 因为操作系统通过 PCB 感知和管理进程；PCB 中通常保存标识信息、状态信息、资源信息和 CPU 现场。
4. 三状态说明运行、就绪、阻塞；五状态再加入创建和结束；七状态再加入就绪挂起和阻塞挂起，用于描述换入换出。
5. 调度回答按什么原则分配、何时分配、怎样分配；高级调度决定作业是否进入内存，中级调度负责对换，低级调度负责从就绪队列中选进程占用 CPU。
6. FCFS 简单但偏向长作业；SJF 偏向短作业；RR 适合交互式系统；优先级调度适合区分任务重要性；多级反馈队列综合照顾交互、I/O 型和计算型进程。
7. 线程是进程内可调度的执行单元；进程主要拥有资源，线程主要作为 CPU 调度单位。
8. 用户级线程开销小但阻塞影响大；内核级线程支持真正并行和内核调度；轻量级进程常作为用户线程和内核线程之间的中间层。
9. 同步强调执行次序协调，互斥强调独占访问；临界资源是一次只允许一个进程访问的资源，临界区是访问该资源的代码段。
10. 因为信号量能统一表达资源计数、同步等待和互斥进入，而 P/V 原语又保证了对信号量操作的原子性。
11. 生产者消费者体现同步与互斥并存，读者写者体现共享读与独占写的权衡，哲学家就餐体现资源分配顺序不当会导致死锁。
12. 死锁来自资源竞争和不当的并发推进；预防是破坏必要条件，避免是在分配时保持安全状态，检测是在死锁发生后识别它，解除则通过终止进程、抢占资源、回滚等方式恢复系统。
