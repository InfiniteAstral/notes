# 4.2 并发控制

## 并发控制的必要性

多个事务并发执行时，如果不加以控制，会导致以下**四种数据不一致问题**（考试高频考点）：

::: details 1. 丢失更新（Lost Update）

两个事务同时读取并修改同一数据，后提交的事务覆盖了先提交的事务的修改。

| 时间 | 事务T1            | 事务T2            | 账户余额balx |
| ---- | ----------------- | ----------------- | ------------ |
| t1   | begin_transaction |                   | 100          |
| t2   | read(balx)        | begin_transaction | 100          |
| t3   | balx=balx-10      | read(balx)        | 100          |
| t4   | write(balx)       | balx=balx+100     | 90           |
| t5   | commit            | write(balx)       | 200          |
| t6   |                   | commit            | 200          |

**结果**：`T1` 的更新被丢失，正确结果应为 `190`，实际为 `200`。
:::

::: details 2. 脏读（Dirty Read）

一个事务读取了另一个事务**未提交**的修改数据，而该事务随后回滚，导致读取的数据无效。

| 时间 | 事务T1            | 事务T2            | 账户余额balx |
| ---- | ----------------- | ----------------- | ------------ |
| t1   |                   | begin_transaction | 100          |
| t2   |                   | read(balx)        | 100          |
| t3   |                   | balx=balx+100     | 100          |
| t4   | begin_transaction | write(balx)       | 200          |
| t5   | read(balx)        | …                 | 200          |
| t6   | balx=balx-10      | rollback          | 100          |
| t7   | write(balx)       |                   | 190          |
| t8   | commit            |                   | 190          |

**结果**：`T1` 读取了 `T2` 未提交的脏数据，导致最终结果错误。
:::

::: details 3. 不可重复读（Non-repeatable Read）

一个事务在执行过程中，两次读取同一数据，得到的结果不同，因为在两次读取之间，该数据被另一个事务修改并提交。

| 时间 | 事务T3            | 事务T4            | balx | baly | balz |
| ---- | ----------------- | ----------------- | ---- | ---- | ---- |
| t1   |                   | begin_transaction | 100  | 0    | 0    |
| t2   | begin_transaction | read(balx)        | 100  | 0    | 0    |
| t3   | read(balx)        | balx=balx+100     | 100  | 0    | 0    |
| t4   | baly=baly+balx    | write(balx)       | 200  | 0    | 0    |
| t5   | write(baly)       | commit            | 200  | 100  | 0    |
| t6   | read(balx)        |                   | 200  | 100  | 0    |
| t7   | balz=balz+balx    |                   | 200  | 100  | 0    |
| t8   | write(balz)       |                   | 200  | 100  | 200  |
| t9   | commit            |                   | 200  | 100  | 200  |

**结果**：`T3` 两次读取 `balx` 的值不同（`100` 和 `200`）。
:::

::: details 4. 不一致分析（Inconsistent Analysis）

一个事务在统计数据时，另一个事务修改了部分数据，导致统计结果不准确。
:::

## 事务的调度

**调度（Schedule）**：一组并发事务的操作按照时间顺序排列的序列。

| 调度类型             | 定义                                           | 正确性                 |
| -------------------- | ---------------------------------------------- | ---------------------- |
| **串行调度**         | 所有事务的操作连续执行，没有交叉               | 总是正确的             |
| **并发调度**         | 多个事务的操作交叉执行                         | 可能正确也可能错误     |
| **可串行化调度**     | 并发调度的执行结果与某一串行调度的结果相同     | 正确的并发调度         |
| **冲突可串行化调度** | 通过交换非冲突操作可以转换为串行调度的并发调度 | 可串行化调度的一个子集 |

## 冲突操作

对**同一数据项**的以下操作是冲突的：

1. 读 - 写（Read-Write）
2. 写 - 读（Write-Read）
3. 写 - 写（Write-Write）

::: tip 注意

对**不同数据项**的操作不冲突；对同一数据项的**读 - 读**操作不冲突。
:::

## 冲突可串行化的判定：前趋图法

**前趋图**（Precedence Graph）是一个有向图 $G(V, E)$：

- 顶点 $V$：每个顶点代表一个事务
- 有向边 $E$：如果事务 $T_i$ 和 $T_j$ 存在冲突操作，且 $T_i$ 的操作在 $T_j$ 之前，则存在一条边 $T_i \to T_j$

**判定规则**：如果前趋图**无回路**，则该调度是冲突可串行化的。

## 两段锁协议（2PL）

两段锁协议是保证并发调度**冲突可串行化**的**充分条件**（不是必要条件）。

## 基本两段锁协议

事务的执行分为**两个阶段**：

1. **增长阶段（Growing Phase）**：只能申请加锁，不能释放任何锁
2. **收缩阶段（Shrinking Phase）**：只能释放锁，不能申请任何新锁

## 锁的类型

| <span style="white-space: nowrap;">锁类型</span> | 英文                   | 作用                                      | 兼容性                         |
| ------------------------------------------------ | ---------------------- | ----------------------------------------- | ------------------------------ |
| **共享锁**                                       | Shared Lock（S 锁）    | 持有 `S` 锁的事务只能读数据，不能修改数据 | 与 `S` 锁兼容，与 `X` 锁不兼容 |
| **排他锁**                                       | Exclusive Lock（X 锁） | 持有 `X` 锁的事务既能读数据，也能修改数据 | 与任何锁都不兼容               |

## 两段锁协议的变种

1. **严格两段锁协议（Rigorous 2PL）**：事务直到提交时才释放所有锁。保证了调度的**可恢复性**，是大多数数据库系统的实现方式。
2. **次严格两段锁协议（Strict 2PL）**：事务直到提交时才释放所有**排他锁**。

## 死锁问题

**死锁**：两个或多个事务互相等待对方持有的锁，导致所有事务都无法继续执行。

::: info 死锁示例

| 时间 | 事务Ta            | 事务Tb            |
| ---- | ----------------- | ----------------- |
| t1   | begin_transaction | begin_transaction |
| t2   | Lock_x(A)         | Lock_x(B)         |
| t3   | read(A)           | read(B)           |
| t4   | A=A-10            | B=B+100           |
| t5   | write(A)          | write(B)          |
| t6   | Lock_x(B)（等待） | Lock_x(A)（等待） |
| t7   | 等待              | 等待              |

:::

## 死锁的解决方法

1. **死锁避免**：通过严格的加锁顺序避免死锁
2. **死锁检测与解除**：
   - 构建**等待图（Wait-for Graph）**，如果图中存在回路，则发生死锁
   - 解除死锁：选择一个**代价最小**的事务回滚
