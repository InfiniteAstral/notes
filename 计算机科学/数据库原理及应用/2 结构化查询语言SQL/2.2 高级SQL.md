# 2.2 高级SQL

## 索引（Index）

- 索引是建立在表的**一列或多列**上的**辅助数据结构**，用于加速元组定位与数据检索。
- 索引本身**不改变表中数据的逻辑内容**，但会额外占用存储空间，并增加插入、删除、更新时的维护开销。
- 索引**并不是越多越好**。经常作为**查询条件、连接条件、排序条件**的列更适合建立索引。
- 常见索引包括**普通索引、唯一索引、复合索引**等。具体实现方式依赖于 DBMS。

### 创建索引

```sql
CREATE [UNIQUE] INDEX 索引名
ON 表名 (
  列名 [ASC|DESC][, ...]
);
```

### 删除索引

```sql
DROP INDEX 索引名;
```

### 使用说明

- `UNIQUE`：建立**唯一索引**，要求索引列上的值不能重复。
- **复合索引**可同时包含多个列，常用于多列联合查询。
- 是否真正使用索引由**查询优化器**决定，并不是建立索引后每次查询都会强制走索引。

::: info 例

1. 为学生表的姓名字段创建普通索引

   ```sql
   CREATE INDEX sName_index
   ON Student (
     sName
   );
   ```

2. 为选课表的学号、课程号创建唯一复合索引

   ```sql
   CREATE UNIQUE INDEX sc_sNo_cNo_index
   ON SC (
     sNo,
     cNo
   );
   ```

3. 删除学生表姓名索引

   ```sql
   DROP INDEX sName_index;
   ```

:::

## 视图（View）

- 视图是从一个或多个基表导出的**虚表**，其内容由查询定义决定。
- 数据库通常只保存**视图定义**，不单独保存视图结果数据；查询视图时，系统会将其转换为对基表的查询。
- 视图可用于**简化复杂查询、隐藏部分字段、限制用户访问范围、定制化数据表示**。

### 创建视图

```sql
CREATE VIEW 视图名 [(新列名[, ...])]
AS
子查询
[WITH [CASCADED|LOCAL] CHECK OPTION];
```

### 删除视图

```sql
DROP VIEW 视图名 [RESTRICT|CASCADE];
```

### 视图可更新性

- 基表中数据发生变化时，相关视图查询结果通常会随之变化。
- **并不是所有视图都可更新**。含有**聚合、分组、去重、集合运算、多表复杂连接**等操作的视图，往往不可直接更新。
- `WITH CHECK OPTION` 用于约束通过视图执行的 `INSERT` 或 `UPDATE` 操作，要求更新后的元组**仍满足视图定义条件**。

### 视图优缺点

| 优点                   | 缺点                   |
| :--------------------- | :--------------------- |
| 简化复杂查询           | 某些视图不可更新       |
| 提高数据安全性         | 依赖基表结构稳定       |
| 支持数据逻辑独立性     | 复杂视图可能影响性能   |
| 便于按用户需求定制展示 | 过多视图会增加维护成本 |

::: info 例

1. 创建“离散数学”课程学生成绩视图

   ```sql
   CREATE VIEW view_1 AS
   SELECT
     s.sName,
     c.cName,
     sc.score
   FROM Student s, Course c, SC sc
   WHERE s.sNo = sc.sNo
     AND c.cNo = sc.cNo
     AND c.cName = '离散数学';
   ```

2. 创建按课程统计选课人数的视图

   ```sql
   CREATE VIEW view_2 (cNo, numberOf) AS
   SELECT
     cNo,
     COUNT(*)
   FROM SC
   GROUP BY cNo;
   ```

3. 创建信息学院学生视图，并启用检查选项

   ```sql
   CREATE VIEW IS_Student AS
   SELECT
     *
   FROM Student
   WHERE dNo IN (
     SELECT
       dNo
     FROM Department
     WHERE dName = '信息学院'
   )
   WITH CHECK OPTION;
   ```

4. 查询视图中不及格的学生

   ```sql
   SELECT
     sName,
     score
   FROM view_1
   WHERE score < 60;
   ```

5. 删除视图

   ```sql
   DROP VIEW view_1;
   DROP VIEW view_2;
   ```

:::

## 事务（Transaction）

- 事务是一个**逻辑工作单元**，其中的操作要么**全部执行成功**，要么**全部撤销**。
- 事务用于保证数据库操作满足 **ACID** 特性，即**原子性**（Atomicity）、**一致性**（Consistency）、**隔离性**（Isolation）、**持久性**（Durability）。
- 实际开发中，事务通常围绕一组相关的 `INSERT`、`UPDATE`、`DELETE` 语句展开。

### 事务控制语句

- 开启事务：`BEGIN TRANSACTION;` 或 `START TRANSACTION;`
- 提交事务：`COMMIT;`
- 回滚事务：`ROLLBACK;`
- 设置保存点：`SAVEPOINT 保存点名;`

::: tip 注
不同数据库对事务与 **DDL** 语句的处理**不完全相同**。有些 DBMS 中某些 DDL 语句会**隐式提交事务**，因此不能一概而论地认为 DDL 总能与 DML 一起安全放入同一事务。
:::

::: info 例

银行转账事务（1 号账户转 10 元到 2 号账户）

```sql
BEGIN TRANSACTION;

UPDATE account
SET balance = balance - 10
WHERE id = '01';

UPDATE account
SET balance = balance + 10
WHERE id = '02';

COMMIT;

-- 若执行过程中发生错误，则应回滚
-- ROLLBACK;
```

:::

## 完整性约束（Integrity Constraints）

- 完整性约束用于保证数据库中的数据满足预先定义的**语义规则**，避免出现无效数据或不一致数据。
- 常见完整性约束包括**实体完整性、参照完整性、用户定义完整性**。
- 约束可以通过 `PRIMARY KEY`、`FOREIGN KEY`、`UNIQUE`、`NOT NULL`、`CHECK` 等方式定义。

### 约束类型

1. **实体完整性**

   **主键值不能为空**，且主键必须能够**唯一标识一个元组**。

2. **参照完整性**

   **外键值**必须引用被参照关系中已存在的候选键值，或者为空值（具体是否允许为空由约束定义决定）。

3. **用户定义完整性**

   根据实际业务规则定义的**附加约束**，例如年龄范围、成绩上下限等。

### 延迟约束

- 某些 DBMS 支持将约束检查**延迟到事务提交时**再进行，这类约束称为**延迟约束**。
- 延迟约束适合处理“**中间状态暂时不满足约束，但事务结束后整体恢复一致**”的场景。
- 是否支持 `DEFERRABLE` 以及具体语法取决于数据库产品，并非所有系统都支持。

::: info 例

1. 外键约束冲突：删除仍被学生表引用的学院

   ```sql
   DELETE FROM Department
   WHERE dNo = '03';

   -- 报错：违反外键约束，因为 Student 表中仍有元组引用该学院
   ```

2. 使用延迟约束，先删学院再修改学生所属学院

   ```sql
   ALTER TABLE Student
   ALTER CONSTRAINT student_dno_fkey
   DEFERRABLE INITIALLY DEFERRED;

   BEGIN TRANSACTION;

   DELETE FROM Department
   WHERE dNo = '03';

   UPDATE Student
   SET dNo = '02'
   WHERE sNo = '008';

   COMMIT;
   ```

:::

## 访问控制（Access Control）

- 访问控制用于限制用户对数据库系统和数据库对象的**操作权限**。
- 权限管理通常由 **DBA** 负责，既可以直接授予用户，也可以通过**角色**集中管理。
- 常见权限分为**系统权限**和**对象权限**。

### 权限类型

1. **系统权限**

   控制用户是否能够登录数据库，或是否具备**建表、建用户**等系统级操作能力。

2. **对象权限**

   控制用户是否能够对某个表、视图或其他对象执行 `SELECT`、`INSERT`、`UPDATE`、`DELETE` 等操作。

### 授权（`GRANT`）

```sql
-- 系统权限授权
GRANT 系统权限[, ...]
TO 用户|角色|PUBLIC
[WITH ADMIN OPTION];

-- 对象权限授权
GRANT 对象权限[(列名[, ...])][, ...]
ON 对象
TO 用户|角色|PUBLIC
[WITH GRANT OPTION];
```

### 回收权限（`REVOKE`）

```sql
-- 回收系统权限
REVOKE 系统权限[, ...]
FROM 用户|角色|PUBLIC;

-- 回收对象权限
REVOKE 对象权限[, ...]
ON 对象
FROM 用户|角色|PUBLIC
[CASCADE CONSTRAINTS];
```

### 使用说明

- `WITH ADMIN OPTION`：允许被授权者**继续授予他人相同的系统权限**。
- `WITH GRANT OPTION`：允许被授权者**继续授予他人相同的对象权限**。
- `PUBLIC` 表示**所有用户**。
- 不同数据库对角色、列级授权、级联回收等细节支持程度不同。

::: info 例

1. 授予用户查询学生表的权限

   ```sql
   GRANT SELECT
   ON Student
   TO user01;
   ```

2. 授予用户修改学生姓名的权限，并允许继续转授

   ```sql
   GRANT UPDATE (sName)
   ON Student
   TO user01
   WITH GRANT OPTION;
   ```

3. 回收用户查询学生表的权限

   ```sql
   REVOKE SELECT
   ON Student
   FROM user01;
   ```

:::
