# 2.4 查询性能分析与优化

## 查询处理过程

SQL 查询从输入到返回结果需经过以下**标准化流程**：

1. **扫描、语法分析与有效性检查**：验证 SQL 语法正确性，检查表、列是否存在
2. **生成查询内部表示**：转换为关系代数表达式或查询树
3. **查询优化**：优化器生成多个执行计划，选择**代价最小**的方案
4. **查询代码生成**：将优化后的执行计划转换为可执行代码
5. **执行查询**：访问数据库，返回结果集

## SQL 到关系代数的转换

- 将 SQL 查询分解为**查询块（query block）**，每个查询块是一个独立的 `SELECT-FROM-WHERE` 单元
- 每个查询块转换为对应的**关系代数表达式**，作为优化的**基本单元**

::: info 例
嵌套查询的查询块分解

```sql
-- 外层查询块
SELECT sName
FROM Student
WHERE age > (
  -- 内层查询块
  SELECT MAX(age)
  FROM Student
  WHERE dNo = '01'
);
```

转换为关系代数：

- 内层：

  $$
  \gamma_{\max(\text{age})}\left(\sigma_{\text{dNo}='01'}(\text{Student})\right)
  $$

- 外层：

  $$
  \Pi_{\text{sName}}\left(\sigma_{\text{age}>c}(\text{Student})\right)
  $$

  其中 $c$ 为内层查询结果。

:::

## 查询树（Query Tree）

- 关系系统内部表示查询的**树状数据结构**，又称语法分析树、查询执行树
- **叶节点**：输入关系（表）
- **中间节点**：关系代数运算（选择、投影、连接等）
- **根节点**：最终查询结果

::: info 例
查询成绩 < 60 的学生姓名、课程名和成绩的查询树

```sql
SELECT sName, cName, score
FROM Student S, Course C, SC
WHERE S.sNo = SC.sNo
  AND C.cNo = SC.cNo
  AND score < 60;
```

对应的查询树结构：

```
Π sName,cName,score
        |
⋈ C.cNo=SC.cNo
        |
⋈ S.sNo=SC.sNo
    /       \
Course   ⋈
        /   \
    Student  σ score<60
                |
               SC
```

:::

## 基本操作的实现

### 选择运算

| 实现方法             | 适用场景                              | 特点                                      |
| :------------------- | :------------------------------------ | :---------------------------------------- |
| 线性搜索（全表扫描） | 无条件查询、小表查询                  | 简单但效率低，需扫描所有数据块            |
| 二分查找             | 排序后的表                            | 效率高于线性搜索，时间复杂度 $O(\log{N})$ |
| 索引                 | 有索引的列查询                        | 效率最高，直接定位数据位置                |
| 索引类型             | **Hash 索引、B 树/B+ 树索引、簇索引** |                                           |

### 连接运算

| 实现方法     | 适用场景           | 特点                                      |
| :----------- | :----------------- | :---------------------------------------- |
| 嵌套循环连接 | 小表作为外层表     | 简单但效率低，时间复杂度 $O(M×N)$         |
| 排序合并连接 | 连接列已排序的表   | 适合大数据量等值连接，时间复杂度 $O(M+N)$ |
| 索引连接     | 内层表连接列有索引 | 利用索引快速定位匹配行                    |
| Hash 连接    | 等值连接           | 适合大数据量连接，通过 Hash 表分桶匹配    |

::: info 例

1. 选择运算实现对比

```sql
-- C1: 无条件查询 → 全表扫描
SELECT * FROM Student;

-- C2: 主键等值查询 → 索引查找
SELECT * FROM Student WHERE sNo = '2024150101';

-- C3: 范围查询 → 索引范围扫描
SELECT * FROM Student WHERE age > 20;
```

2. 连接运算实现对比

```sql
SELECT * FROM Student S, SC WHERE S.sNo = SC.sNo;
```

- 嵌套循环：遍历 Student 每一行，再遍历 SC 所有行匹配
- 排序合并：先按 sNo 排序两个表，再顺序扫描匹配
- 索引连接：SC 表 sNo 有索引，Student 每一行通过索引查找 SC 匹配行
- Hash 连接：将两个表按 sNo 散列到 Hash 桶，再桶内匹配
  :::

## 查询优化方法

### 基于启发式规则的优化

通过预定义的**启发式规则**调整查询执行顺序，减少**中间结果数据量**：

1. **选择运算尽可能先做**：提前过滤数据，减少后续运算量
2. **投影与选择同时进行**：避免多次扫描表
3. **投影与前后双目运算结合**：减少中间结果的列数
4. **将选择与笛卡尔积合并为连接**：连接运算效率远高于笛卡尔积+选择
5. **提取公共子表达式**：避免重复计算相同子查询

### 基于代价估算的优化

- 通过**代价模型**计算不同执行策略的总代价，选择**代价最小**的方案
- 总代价组成：**I/O 代价（主要）** + CPU 代价 + 内存代价 + 通信代价（分布式数据库）
- 数据字典存储优化所需**统计信息**：
  - 基本表：元组数、元组长度、占用块数
  - 列：不同值个数、最大值、最小值、索引信息
  - 索引：B+ 树层数、叶节点数、选择基数

::: info 例
PostgreSQL 查看执行计划

```sql
EXPLAIN ANALYZE
SELECT DISTINCT course_id
FROM course
WHERE course_term = 'Fal02';
```

输出示例：

```
HashAggregate  (cost=12223.09..12339.76 rows=4667 width=4)
  Group Key: course_id
  ->  Seq Scan on course  (cost=0.00..8279.99 rows=46666 width=4)
        Filter: (course_term = 'Fal02'::bpchar)
Total runtime: 1899.24 msec
```

:::

## 关系代数等价变换规则

| 等价变换               | 内容                                                                                                                 | 作用                         |
| :--------------------- | :------------------------------------------------------------------------------------------------------------------- | :--------------------------- |
| **交换律**             | $A \times B = B \times A$<br>$A \Join B = B \Join A$<br>$A \Join_F B = B \Join_F A$                                  | 多表连接顺序优化基础         |
| **结合律**             | $(A \times B) \times C = A \times (B \times C)$<br>$(A \Join B) \Join C = A \Join (B \Join C)$                       | 多表连接顺序优化基础         |
| **分配律（条件下推）** | $\sigma_F(A \times B) = \sigma_{F_1}(A) \times \sigma_{F_2}(B)$<br>（$F = F_1 \land F_2$，$F_1 \in A$，$F_2 \in B$） | 提前过滤数据，减少中间结果   |
| **投影下推**           | $\Pi_{p,q}(A \times B) = \Pi_p(A) \times \Pi_q(B)$                                                                   | 提前减少列数，降低数据传输量 |
| **串接律**             | $\sigma_{F_1}(\sigma_{F_2}(A)) = \sigma_{F_1 \land F_2}(A)$<br>$\Pi_P(\Pi_Q(A)) = \Pi_P(A)$（$P \subseteq Q$）       | 合并多个选择/投影操作        |

## 案例分析：查询策略对比

### 需求

查询选修了 01 号课程的学生姓名

```sql
SELECT sName
FROM Student S, SC
WHERE S.sNo = SC.sNo AND SC.cNo = '01';
```

### 假设条件

- **Student 表**：1000 条记录，10 条/块 → 100 块
- **SC 表**：10000 条记录，100 条/块 → 100 块
- **01 号课程选课记录**：50 条
- **内存**：可存放 5 块 Student 元组 + 1 块 SC 元组

::: info 例
三种查询策略代价对比

#### 策略Q1：先笛卡尔积，再选择和投影

$$
\Pi_{\text{sName}}
\left(
  \sigma_{\text{S.sNo}=\text{SC.sNo} \land \text{SC.cNo}='01'}
  (\text{Student} \times \text{SC})
\right)
$$

1. **笛卡尔积**：读取 2100 块，生成 10^7 条中间记录，写出 10^6 块
2. **选择**：读取 10^6 块，过滤出 50 条记录
3. **投影**：内存中完成
   **总读写块数：2100 + 10^6 + 10^6 ≈ 200 万块**

#### 策略Q2：先自然连接，再选择和投影

$$
\Pi_{\text{sName}}
\left(
  \sigma_{\text{SC.cNo}='01'}
  (\text{Student} \Join \text{SC})
\right)
$$

1. **自然连接**：读取 2100 块，生成 10^4 条中间记录，写出 10^3 块
2. **选择**：读取 10^3 块，过滤出 50 条记录
3. **投影**：内存中完成
   **总读写块数：2100 + 10^3 + 10^3 = 4100 块**

#### 策略Q3：先选择SC，再连接和投影（最优）

$$
\Pi_{\text{sName}}
\left(
  \text{Student} \Join \sigma_{\text{SC.cNo}='01'}(\text{SC})
\right)
$$

1. **选择 SC**：读取 100 块，得到 50 条记录（内存中）
2. **连接 Student**：读取 100 块，与内存中 SC 记录连接
3. **投影**：内存中完成
   **总读写块数：100 + 100 = 200 块**

#### 索引优化后的Q3

- 若 SC 表 cNo 列**有索引**：选择 SC 仅需读取索引块 + 50 条数据块（约 2~3 块）
- 若 Student 表 sNo 列**有索引**：连接时通过索引查找，无需全表扫描
  **总读写块数可降至个位数**

:::
