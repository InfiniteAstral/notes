# 第 2 章 应用层

应用层是网络边缘的层次，直接为用户应用提供服务，核心是**应用层协议**，定义了应用进程间的通信规则。

## 2.1 应用层基础

### 应用架构

1. **客户端 / 服务器（C/S）架构**
   - **服务器**：长期在线，有固定 IP，处理客户端请求。
   - **客户端**：间歇连接，动态 IP，仅与服务器通信。

2. **P2P（对等）架构**
   - 无永久服务器，对等方直接通信，间歇连接。
   - **自扩展性**：新对等方既带来需求，也带来服务能力。

### 进程寻址

- 进程通过 **套接字（Socket）** 收发报文，套接字是应用和网络的接口。
- 进程的唯一标识：**主机 IP 地址 + 进程端口号**，端口号用于区分同一主机的不同进程。

  ::: info 知名端口
  HTTP (80)、FTP (21)、SMTP (25)、DNS (53)
  :::

### 传输层服务需求

不同应用对传输层的需求不同：

| 应用            | 可靠性         | 实时性     | 吞吐量     |
| --------------- | -------------- | ---------- | ---------- |
| 文件传输        | 要求 100% 可靠 | 不敏感     | 弹性       |
| 电子邮件        | 要求 100% 可靠 | 不敏感     | 弹性       |
| 网页浏览        | 要求 100% 可靠 | 较敏感     | 弹性       |
| 实时语音 / 视频 | 可容忍丢包     | 要求低时延 | 有最低要求 |

### 互联网传输层服务

- **TCP**：面向连接，可靠数据传输，拥塞控制，流量控制，不保证实时 / 带宽。
- **UDP**：无连接，不可靠，无拥塞 / 流量控制，开销小，适合实时应用。

## 2.2 Web 与 HTTP

HTTP（超文本传输协议）是 Web 的核心协议，基于 TCP 实现。

### 基础特性

- **客户端 / 服务器模型**：浏览器是客户端，Web 服务器是服务器。

- **无状态**：服务器不保留客户端的历史状态。

- **连接模式**：
  1. **非持续连接**：每个 TCP 连接只传输一个对象，响应时间为 `2RTT + 文件传输时间`。
  2. **持续连接**：一个 TCP 连接传输多个对象，减少连接开销。

### HTTP 报文

1. **请求报文**
   - **请求行**：方法（`GET`/`POST`/`PUT`/`DELETE`）、URL、版本。
   - **首部行**：`Host`、`User-Agent`、`Accept`、`Cookie` 等。
   - **实体主体**：`POST` 方法的提交数据。

2. **响应报文**
   - **状态行**：版本、状态码、描述。
   - **常见状态码**：
     - **200 OK**：请求成功。
     - **301 Moved Permanently**：永久重定向。
     - **404 Not Found**：资源不存在。
     - **500 Internal Server Error**：服务器错误。

### Cookie 机制

为了让 HTTP 实现“有状态”，Cookie 通过四个组件实现：

1. 响应报文的 `Set-cookie` 首部。
2. 请求报文的 `Cookie` 首部。
3. 客户端本地的 `Cookie` 文件。
4. 服务器的后端数据库。

::: info 作用
身份认证、购物车、个性化推荐、会话保持
:::

### Web 缓存（代理服务器）

- **原理**：代理服务器缓存最近访问的对象，客户端请求先访问代理，命中则直接返回，未命中则向源服务器请求。

- **作用**：降低响应时间、减少接入链路流量、低成本提升服务能力。

- **条件 GET**：通过 `If-modified-since` 首部验证缓存是否过期，避免重复传输。

## 2.3 FTP：文件传输协议

FTP 用于文件的上传下载，采用**带外控制**：

- **控制连接**：端口 21，持久连接，传输命令 / 响应。
- **数据连接**：端口 20，每个文件传输新建一个连接，传输文件数据。

::: info 作用
服务器维护用户状态，记录当前目录、授权信息。
:::

## 2.4 电子邮件

电子邮件系统由用户代理、邮件服务器、`SMTP`/`POP3`/`IMAP` 协议组成。

### SMTP：简单邮件传输协议

- 用于邮件服务器之间的邮件推送，基于 TCP，端口 25。
- **阶段**：握手、传输报文、关闭连接。
- 命令 / 响应交互，报文必须是 7 位 ASCII，支持 MIME 扩展传输非 ASCII 数据。

::: tip 与 HTTP 对比
SMTP 是推送（push），HTTP 是拉取（pull）；SMTP 用持续连接，报文有严格格式限制。
:::

### 邮件访问协议

用于用户从邮件服务器获取邮件：

- **POP3**：“**下载并删除**” 模式，用户下载邮件后，服务器删除，跨客户端无法同步。
- **IMAP**：保留邮件在服务器，支持文件夹管理、跨客户端同步，功能更完善。

## 2.5 DNS：域名系统

DNS 是**分布式**的**分层数据库**，实现**域名到 IP 地址的映射**，是互联网的核心基础设施。

::: warning 为什么不用中心式 DNS？

单点故障、通信容量瓶颈、远距离延迟、可扩展性差，因此采用分布式分层架构。

:::

### DNS 服务器层次

1. **根 DNS 服务器**：13 个根服务器，负责顶级域服务器的寻址。
2. **顶级域（TLD）服务器**：负责 `com`、`org`、`edu`、`cn` 等顶级域名。
3. **权威 DNS 服务器**：组织 / 机构的域名服务器，负责自己域名的解析。
4. **本地 DNS 服务器**：ISP 的默认 DNS，作为代理转发查询，缓存解析结果。

### 域名解析方式

- **迭代查询**：客户端依次查询根、TLD、权威服务器，自己完成迭代。

- **递归查询**：客户端只查询本地 DNS，本地 DNS 代替客户端完成所有查询，返回结果给客户端。

### DNS 资源记录（RR）

- `A`：域名到 IP 地址的映射。

- `NS`：域名到权威服务器的映射。

- `CNAME`：别名到规范域名的映射。

- `MX`：域名到邮件服务器的映射。

### 缓存

DNS 服务器会缓存解析结果，TTL 超时后失效，减少查询开销，避免频繁访问根 / TLD 服务器。

## 2.6 P2P 应用

P2P 架构下，对等方互相传输数据，充分利用用户的上行带宽。

### 文件分发的对比

- **C/S 架构**：服务器必须上传 N 份文件，分发时间随 N 线性增长。
- **P2P 架构**：服务器只需要上传 1 份，对等方互相上传，总上传能力随 N 增长，分发时间增长缓慢。

### BitTorrent（BT）

- 文件分割为 256KB 的块，对等方互相交换块。
- `Tracker`：跟踪所有对等方的列表。
- **策略**：
  - **最稀有优先**：优先下载最少的块，保证块的可用性。
  - **tit-for-tat（针锋相对）**：只向给我上传最快的 4 个对等方上传，每 30 秒随机选一个对等方乐观解禁，找到新的互惠对等方。
