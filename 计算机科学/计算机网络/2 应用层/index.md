# 第 2 章 应用层

应用层是最贴近用户的一层。你看到的网页、邮件、聊天、文件下载、音视频、域名解析，表面上是各种 App 和网站在工作，背后都离不开应用层协议。

本章重点不是死记协议字段，而是先建立一个清晰框架：

- 网络应用运行在哪里
- 应用进程如何彼此通信
- 应用需要传输层提供什么能力
- 常见应用层协议分别解决什么问题

## 2.1 应用层基础

### 什么是网络应用

网络应用运行在**端系统**上，也就是用户主机、服务器、手机、平板等设备上。它们通过网络彼此交换报文，实现网页访问、邮件收发、文件共享、实时通话等功能。

需要特别注意：

- **应用程序运行在网络边缘，而不是网络核心**
- 路由器、交换机负责转发数据，一般不运行用户应用逻辑

这也是为什么互联网中的新应用可以快速推出：开发者通常只需要更新端系统上的应用程序，不需要去改造整个网络核心。

### 常见网络应用

常见的网络应用包括：

- Web 网页服务
- 电子邮件
- 即时消息
- 远程登录
- P2P 文件共享
- 网络游戏
- 流媒体音视频
- 网络电话和视频会议
- 搜索和社交网络

虽然这些应用看起来差别很大，但从网络角度看，它们都在做一件事：**让不同主机上的进程通过协议交换报文。**

### 应用架构

网络应用通常采用两类基本架构：**客户端/服务器（C/S）**和 **P2P（对等）**。

#### 客户端/服务器架构

在这种架构中：

- **服务器**：长期在线，通常拥有固定 IP 地址，负责接收请求并提供服务
- **客户端**：按需接入网络，可能是间歇连接，IP 地址可能变化
- **客户端之间一般不直接通信**，而是通过服务器完成交互

例如浏览器访问网页、手机 App 请求后端接口，都是典型的 C/S 架构。

#### P2P 架构

在 P2P 架构中：

- 没有始终在线的中心服务器，或对中心服务器依赖较弱
- 对等方之间可以直接通信
- 一个对等方既可能请求服务，也可能向别人提供服务

P2P 的一个重要特点是**自扩展性**：

- 新用户加入时，不只是带来新的下载需求
- 同时也带来新的上传能力和服务能力

这使得 P2P 在大规模文件分发场景中很有优势，但在管理、安全、可靠性方面通常更复杂。

### 进程与通信

应用真正通信的对象不是“主机”，而是主机中的**进程（Process）**。

- 同一台主机上的进程，通常通过操作系统提供的机制通信
- 不同主机上的进程，则通过网络交换报文通信

在 C/S 应用中：

- **客户端进程**：主动发起通信
- **服务器进程**：等待连接和请求

在 P2P 应用中，一个程序往往同时扮演客户端和服务器两种角色。

### 套接字 Socket

进程通过**套接字（Socket）** 收发报文。可以把套接字理解成“应用进程通向网络的一扇门”。

- 应用把数据交给套接字
- 套接字再把数据交给传输层和网络去发送
- 接收方从自己的套接字中读取到达的数据

从职责上看：

- **应用开发者**主要控制应用层逻辑和使用套接字的方式
- **操作系统和网络协议栈**负责把数据真正送到远方

### 进程寻址

若想把数据送到正确的进程，单有主机地址还不够，因为一台主机上可能同时运行多个网络程序。

因此，一个网络进程通常通过以下信息标识：

- **IP 地址**：标识主机或网络接口
- **端口号**：标识该主机上的具体进程

也就是说，应用层常说的“找到通信对象”，本质上是找到：

**目标 IP 地址 + 目标端口号**

::: info

常见默认端口：

- HTTP：`80`
- FTP：`21`
- SMTP：`25`
- DNS：`53`

:::

### 应用层协议定义什么

一个应用层协议至少要规定以下内容：

- **报文类型**：例如请求、响应
- **报文格式**：有哪些字段、字段如何组织
- **字段语义**：每个字段分别表示什么
- **交互规则**：什么时候发、先发什么、收到后怎么处理

例如，HTTP 要定义请求行、响应状态码；SMTP 要定义 `HELO`、`MAIL FROM`、`RCPT TO` 等命令；DNS 要定义查询报文和应答报文格式。

### 开放协议与专有协议

- **开放协议**：标准公开，通常在 RFC 中定义，便于不同厂商互操作，例如 HTTP、SMTP、DNS
- **专有协议**：由特定厂商或平台自行设计，不一定公开完整细节

对学习计算机网络而言，教材更关注开放协议，因为它们代表了互联网的通用工作方式。

### 应用对传输层的需求

不同应用对底层传输服务的要求并不相同，常见维度有四个：

- **可靠性**：数据能不能丢
- **时延**：能不能等
- **吞吐量**：需要多大传输速率
- **安全性**：是否需要加密、完整性校验、身份认证

例如：

- 文件传输、电子邮件：通常要求高可靠
- 网页浏览：要求可靠，但对时延也比较敏感
- 实时语音、视频会议：可容忍少量丢包，但很在意时延
- 流媒体：往往要求最低吞吐量，否则体验明显变差

### 互联网提供的传输层服务

互联网中，应用最常用的传输层协议是 **TCP** 和 **UDP**。

#### TCP

TCP 提供的典型能力有：

- 面向连接
- 可靠传输
- 流量控制
- 拥塞控制

但 TCP **不保证**：

- 固定时延
- 最小带宽
- 实时性

#### UDP

UDP 的特点是：

- 无连接
- 尽力而为，不保证可靠送达
- 头部开销小
- 协议机制简单

UDP 也**不提供**可靠性、流量控制、拥塞控制、最小带宽、安全等保证。

### 为什么 TCP 看起来更强，却还需要 UDP

这是非常经典的问题。原因是：

- 有些应用更在意**低开销和低时延**
- 有些应用可以自己处理丢包，或能够容忍少量丢包
- 有些应用希望自行控制传输逻辑，而不想受到 TCP 机制约束

所以并不是“TCP 全面优于 UDP”，而是它们服务于不同需求。

### 安全：SSL/TLS

TCP 和 UDP 本身都不负责应用层加密。实际使用中，很多应用会在 TCP 之上再加入 **SSL/TLS**，从而提供：

- 加密
- 数据完整性校验
- 身份认证

这也是为什么今天访问网页时常看到 `HTTPS` 而不是 `HTTP`。

## 2.2 Web 与 HTTP

### Web 基本概念

一个网页（Web Page）通常不是一个单独文件，而是由多个**对象（Object）** 组成，例如：

- HTML 文件
- 图片
- CSS 文件
- JavaScript 文件
- 音频、视频等资源

通常先有一个基础 HTML 文件，里面再引用其他对象。每个对象都可以通过 **URL** 来定位。

例如：

`www.example.com:80/path/pic.jpg`

可以大致拆成：

- 主机名：`www.example.com`
- 端口号：`80`
- 路径：`/path/pic.jpg`

### HTTP 是什么

**HTTP（HyperText Transfer Protocol，超文本传输协议）** 是 Web 的核心应用层协议。

它采用 **客户端/服务器模型**：

- 浏览器是 HTTP 客户端
- Web 服务器是 HTTP 服务器

HTTP 一般运行在 TCP 之上。一个典型过程是：

1. 浏览器与服务器建立 TCP 连接
2. 浏览器发送 HTTP 请求报文
3. 服务器返回 HTTP 响应报文
4. 连接关闭或继续复用

### HTTP 的无状态特性

HTTP 是**无状态协议（Stateless Protocol）**，意思是：

- 服务器默认不会保存客户端之前请求的历史状态
- 当前请求通常独立处理，不依赖服务器记住之前发生过什么

无状态的好处是简单、易扩展；缺点是很多实际功能需要“记住用户”，例如登录态、购物车、浏览偏好等，因此后面才有 Cookie、Session 等机制。

### 非持续连接与持续连接

#### 非持续 HTTP

在非持续连接中：

- 一个 TCP 连接通常只传输一个对象
- 对象传完后，连接就关闭

如果一个网页包含很多图片，就需要反复建立多个 TCP 连接，效率较低。

对于一个对象，非持续 HTTP 的响应时间常近似写为：

`2RTT + 文件发送时间`

其中：

- 一个 RTT 用于建立 TCP 连接
- 一个 RTT 用于发送请求并收到响应的第一个字节

#### 持续 HTTP

在持续连接中：

- 客户端与服务器建立一次 TCP 连接后
- 可以在这个连接上连续传输多个对象

这样可以减少频繁建立和关闭连接的开销，因此效率更高。

### HTTP 请求报文

HTTP 报文有两类：**请求报文**和**响应报文**。

一个典型的请求报文包括：

- **请求行**：方法、URL、版本
- **首部行**：说明客户端能力和请求附加信息
- **空行**
- **实体主体**：某些方法会携带提交数据

例如常见的请求行：

```http
GET /index.html HTTP/1.1
```

### 常见请求方法

- **GET**：获取资源
- **POST**：提交数据，常用于表单提交
- **HEAD**：只要首部，不要对象本体
- **PUT**：上传并存储指定资源
- **DELETE**：删除指定资源

### GET 与 POST 的区别

这是初学者非常常见的考点。

- **GET**：提交的数据通常出现在 URL 中，更适合获取资源
- **POST**：提交的数据通常放在请求报文主体中，更适合提交表单或发送较多数据

需要注意的是，GET 和 POST 的“能否安全使用”并不由名字决定，而取决于具体业务设计和是否配合 HTTPS。

### HTTP 响应报文

响应报文通常包括：

- **状态行**：版本、状态码、原因短语
- **首部行**
- **空行**
- **实体主体**：真正返回的数据内容

例如：

```http
HTTP/1.1 200 OK
```

### 常见状态码

- **200 OK**：请求成功
- **301 Moved Permanently**：资源永久移动
- **400 Bad Request**：请求格式错误，服务器无法理解
- **404 Not Found**：请求的资源不存在
- **505 HTTP Version Not Supported**：服务器不支持该 HTTP 版本

### Cookie：让 HTTP “看起来有状态”

Cookie 不是为了替代 HTTP，而是为了弥补 HTTP 无状态带来的不便。

一个典型 Cookie 机制包含四个部分：

1. 服务器在响应报文中通过 `Set-Cookie` 设置标识
2. 浏览器把 Cookie 保存到本地
3. 浏览器后续请求时通过 `Cookie` 首部带回该标识
4. 服务器在后端数据库中根据该标识找到对应用户状态

Cookie 的常见用途：

- 用户身份认证
- 购物车
- 个性化推荐
- 登录会话保持

::: warning

Cookie 很有用，但也涉及隐私问题。站点可能借此追踪用户行为，因此在真实系统里通常要结合安全与隐私策略使用。

:::

### Web 缓存与代理服务器

**Web 缓存（Web Cache）** 或**代理服务器（Proxy Server）** 的目标是：

**尽量不访问源服务器，就满足客户端请求。**

工作方式大致是：

- 浏览器先把请求发给代理服务器
- 若代理中已有目标对象，就直接返回
- 若没有，再由代理去访问源服务器并缓存结果

代理服务器在这个过程中有双重身份：

- 对客户端来说，它像服务器
- 对源服务器来说，它像客户端

### 为什么需要 Web 缓存

它有三个核心价值：

- 降低用户访问时延
- 减少接入链路流量
- 降低源服务器负载

特别是在大量用户请求热门资源时，缓存效果非常明显。

### 条件 GET

缓存并不意味着永远不联系源服务器，因为缓存内容可能过期。

**条件 GET** 的思想是：

- 客户端或代理向服务器说明“我手上的版本更新时间是某个时刻”
- 如果服务器发现资源没变，就返回 `304 Not Modified`
- 这样就不必再次发送完整对象

这能显著减少重复传输。

## 2.3 FTP：文件传输协议

### FTP 是什么

**FTP（File Transfer Protocol）** 用于在本地主机和远程主机之间传输文件，支持上传和下载。

它采用客户端/服务器模式：

- 客户端发起文件操作
- 服务器保存远程文件系统并响应请求

### FTP 的特点：控制连接与数据连接分离

FTP 最重要的特点是：

**控制信息和文件数据走的是两条不同的 TCP 连接。**

- **控制连接**：服务器端口通常为 `21`
- **数据连接**：服务器端口通常为 `20`

控制连接用于：

- 登录认证
- 发送命令
- 浏览目录

数据连接用于：

- 真正传输文件内容

### 带外控制

由于控制命令和数据内容分开传输，所以常说 FTP 使用**带外控制（Out-of-Band Control）**。

这和 HTTP 很不一样。HTTP 通常是在同一连接中传输请求控制信息和对象内容，而 FTP 则显式区分两类连接。

### FTP 为什么是有状态协议

FTP 服务器会维护一些会话状态，例如：

- 当前目录
- 用户登录状态
- 之前是否已经完成授权

因此 FTP 与 HTTP 的“无状态”特性形成鲜明对比。

### 常见 FTP 命令

FTP 在控制连接上用 ASCII 文本发送命令。常见命令包括：

- `USER username`：提供用户名
- `PASS password`：提供密码
- `LIST`：列出当前目录文件
- `RETR filename`：获取文件
- `STOR filename`：上传并保存文件

服务器也会返回状态码和状态短语作为响应。

## 2.4 电子邮件

### 电子邮件系统的组成

电子邮件系统主要由三部分构成：

- **用户代理（User Agent）**：也叫邮件客户端，如 Outlook、Thunderbird、手机邮件 App
- **邮件服务器（Mail Server）**：负责存储、转发、投递邮件
- **邮件协议**：包括发送协议和读取协议

用户写邮件、看邮件是在用户代理里完成的；真正保存邮件和在服务器之间转发邮件的，是邮件服务器。

### 邮件服务器做什么

邮件服务器主要负责两件事：

- **邮箱（Mailbox）**：保存用户收到的邮件
- **输出报文队列**：暂存即将发送给其他服务器的邮件

所以一封邮件从发件人到收件人，通常不是“客户端直接发给对方客户端”，而是：

发件人客户端 -> 发件人邮件服务器 -> 收件人邮件服务器 -> 收件人客户端

### SMTP：发送邮件的协议

**SMTP（Simple Mail Transfer Protocol）** 是发送邮件的核心协议，主要用于：

- 用户代理向邮件服务器发送邮件
- 邮件服务器之间转发邮件

SMTP 基于 TCP，典型端口为 `25`。

### SMTP 传输的三个阶段

SMTP 的典型交互分三步：

1. **建立连接并握手**
2. **传输邮件报文**
3. **关闭连接**

它采用命令/响应方式工作，常见命令有：

- `HELO`
- `MAIL FROM`
- `RCPT TO`
- `DATA`
- `QUIT`

### SMTP 与 HTTP 的对比

两者都属于应用层协议，也都有请求/响应风格的交互，但差别很大：

- **SMTP 是 push（推送）**
- **HTTP 是 pull（拉取）**

例如：

- 邮件发送服务器主动把邮件“推”给接收服务器
- 浏览器主动去服务器“拉”网页资源

另外：

- SMTP 通常使用持续连接
- SMTP 邮件报文原始格式受 7 位 ASCII 限制

### 邮件报文格式与 MIME

邮件本身的文本格式由相关标准定义，常见首部有：

- `From:`
- `To:`
- `Subject:`

注意：

- 这些报文首部和 SMTP 会话里的 `MAIL FROM`、`RCPT TO` 不是同一个概念

由于原始 SMTP 更适合传纯文本，因此后来引入了**MIME（多用途互联网邮件扩展）**，用来支持：

- 图片
- 音频
- 视频
- 应用文件附件
- 非 ASCII 内容

MIME 常通过额外首部说明内容类型和编码方式。

### 用户如何取回邮件

用户从邮件服务器读取邮件，常见方式有三种：

- **POP3**
- **IMAP**
- **基于 Web 的邮件访问（HTTP）**

### POP3

POP3（邮局协议版本 3）比较简单，常见特点是：

- 支持认证
- 支持列出、读取、删除邮件
- 经典模式是“下载并删除”

这意味着：

- 邮件从服务器下载到本地后，服务器上可能被删掉
- 换一台设备时，可能看不到之前的邮件

虽然某些实现支持“下载并保留”，但 POP3 整体上仍偏简单。

### IMAP

IMAP 更适合现代多设备场景，它的特点是：

- 邮件通常保留在服务器上
- 支持文件夹管理
- 支持跨设备同步状态

也就是说，IMAP 不只是“把邮件取下来”，而是让服务器持续保存邮件状态和组织结构。

### POP3 与 IMAP 对比

| 对比项 | POP3 | IMAP |
| ------ | ---- | ---- |
| 邮件保存位置 | 常下载到本地 | 主要保留在服务器 |
| 多设备同步 | 较弱 | 较强 |
| 文件夹管理 | 一般较弱 | 支持较好 |
| 状态维护 | 简单 | 更丰富 |

## 2.5 DNS：域名系统

### 为什么需要 DNS

网络通信真正依赖的是 **IP 地址**，但人更擅长记忆的是**域名**。

例如：

- 记住 `www.example.com` 比记住一串数字地址容易得多

因此 DNS 的核心作用是：

**实现域名和 IP 地址之间的映射。**

但 DNS 不只做这一件事，它还支持：

- 主机别名
- 邮件服务器别名
- 负载均衡

例如，一个域名可以对应多个 IP 地址，让请求在多个服务器之间分担。

### DNS 为什么不用中心式设计

如果所有解析都依赖一个中心服务器，会有明显问题：

- 单点故障
- 访问压力过大
- 距离用户太远导致时延高
- 可扩展性差

所以 DNS 采用的是：

**分布式、分层式数据库**

### DNS 的层次结构

DNS 常见层次包括：

1. **根 DNS 服务器**
2. **顶级域（TLD）服务器**
3. **权威 DNS 服务器**
4. **本地 DNS 服务器**

各自作用如下：

- **根服务器**：告诉你该去问哪个顶级域服务器
- **TLD 服务器**：告诉你该去问哪个具体域的权威服务器
- **权威服务器**：给出目标主机名的最终映射结果
- **本地 DNS 服务器**：通常由 ISP 或学校、公司维护，离用户最近，常负责代理查询和缓存

### 一个典型解析过程

假设你要访问 `www.amazon.com`，大致过程是：

1. 主机先问本地 DNS 服务器
2. 本地 DNS 若没有缓存，就去问根服务器
3. 根服务器返回 `.com` 顶级域服务器信息
4. 本地 DNS 再去问 `.com` 顶级域服务器
5. TLD 服务器返回 `amazon.com` 的权威服务器信息
6. 本地 DNS 再去问权威服务器
7. 权威服务器返回 `www.amazon.com` 对应的 IP 地址

### 迭代查询与递归查询

这是 DNS 最容易混淆的地方之一。

#### 迭代查询

迭代查询的特点是：

- 被询问的服务器如果不知道最终答案
- 会告诉你“下一步该去问谁”

也就是：

**我不替你问到底，但我告诉你下一站。**

#### 递归查询

递归查询的特点是：

- 被询问的服务器代替你继续往下问
- 最终把完整答案返回给你

也就是：

**你只问一次，其余过程交给它处理。**

在实际互联网中，主机通常把任务交给本地 DNS；本地 DNS 再结合缓存和层次服务器完成后续解析。

### DNS 缓存

DNS 的高性能很大程度上依赖缓存。

- 域名服务器一旦得到某条映射，就可能缓存下来
- 后续相同请求可以直接返回结果

缓存的好处：

- 减少查询时延
- 降低根服务器和 TLD 服务器负担

但缓存不是永久有效的，因为记录带有 **TTL（生存时间）**。TTL 到期后，缓存可能失效，需要重新查询。

### DNS 资源记录 RR

DNS 本质上存的是各种**资源记录（Resource Record, RR）**。常见类型包括：

- **A**：主机名 -> IPv4 地址
- **NS**：域名 -> 该域的权威 DNS 服务器
- **CNAME**：别名 -> 规范主机名
- **MX**：域名 -> 邮件服务器

理解这些记录后，很多 DNS 题目就会容易很多。

### DNS 报文

DNS 使用查询报文和应答报文进行交互。两者整体格式相似，通常包括：

- 标识号
- 标志位
- 问题区
- 回答区
- 权威信息区
- 附加信息区

对于入门阶段，重点先理解：

- DNS 是应用层协议
- 它通过报文完成查询和应答
- 它的本质是“按层次查找 + 广泛缓存”

## 2.6 P2P 应用

### P2P 的基本思想

P2P（Peer-to-Peer）应用强调对等方之间直接协作：

- 没有始终在线的中心服务器，或对其依赖较弱
- 对等方可能随时加入或离开
- 对等方 IP 地址可能变化
- 每个节点既消耗资源，也贡献资源

典型例子包括：

- 文件分发
- 流媒体
- 某些 VoIP 应用

### C/S 与 P2P 的文件分发对比

假设一台服务器要把同一个文件分发给很多用户：

#### C/S 模式

在客户端/服务器模式下：

- 服务器必须向每个客户端都上传一份文件
- 用户越多，服务器压力越大
- 分发时间通常随着用户数增加而明显增长

#### P2P 模式

在 P2P 模式下：

- 服务器至少上传一份初始副本
- 后续各对等方会继续把自己拿到的块上传给别人
- 用户越多，总上传能力也可能越强

这就是为什么 P2P 在大规模文件分发中通常更具扩展性。

### BitTorrent 基本思想

BitTorrent 是最经典的 P2P 文件分发协议之一。

它的核心思想是：

- 把文件切分为很多小块
- 对等方一边下载，一边上传自己已拥有的块
- 各节点彼此协作，加快整体传播速度

常见术语：

- **Torrent**：参与同一文件块交换的一组对等方
- **Tracker**：帮助节点发现其他对等方的跟踪服务器

### BitTorrent 如何工作

一个新节点加入时，通常会：

1. 联系 Tracker
2. 获取一批对等方列表
3. 与其中部分节点建立连接，成为邻居
4. 下载自己缺少的文件块
5. 把已有文件块上传给其他节点

### 最稀有优先

BitTorrent 常采用 **rarest first（最稀有优先）** 策略：

- 节点优先请求当前网络中最少见的文件块

这样做的好处是：

- 提高稀有块的传播速度
- 避免某些块过少，导致整体下载被“卡住”

### Tit-for-Tat（针锋相对）

BitTorrent 还采用一种鼓励互惠的机制，通常称为 **tit-for-tat**：

- 节点优先向那些给自己上传速度更快的对等方上传数据
- 同时周期性地“乐观解禁”一个新对等方，尝试寻找更好的交换伙伴

这背后的核心思想是：

- 鼓励“你帮我，我也帮你”
- 抑制只下载不上传的自私行为
- 帮助节点找到更高效的互惠伙伴
