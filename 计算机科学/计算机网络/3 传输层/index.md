# 第 3 章 传输层

传输层位于应用层和网络层之间，核心任务是为**不同主机上的应用进程**提供端到端传输服务。

这一层最容易让初学者混乱，因为它既要讲“怎么把数据可靠送到对方进程”，又要讲“为什么网络会拥塞，以及 TCP 如何处理拥塞”。学习时建议始终抓住两条主线：

- **可靠传输主线**：从 `rdt1.0` 一步步演进到 TCP
- **控制主线**：流量控制解决“接收方来不及处理”，拥塞控制解决“网络本身扛不住”

::: info 学习提示

传输层这一章最容易乱，是因为它把两类问题放在了一起：

- 怎么可靠地把数据送到对方
- 怎么在网络拥塞时调整发送行为

所以学习时最好始终把“可靠传输”和“控制机制”两条线分开看。

:::

## 3.1 传输层服务

### 传输层到底做什么

传输层为运行在不同主机上的应用进程提供**逻辑通信（Logical Communication）**。

所谓“逻辑通信”，意思是：

- 从应用视角看，好像本机进程直接在和远端进程说话
- 实际上底层仍然要经过网卡、链路、路由器、IP 数据报等一整套网络机制

也就是说，传输层帮应用屏蔽了很多底层细节。

### 传输层与网络层的区别

这是本章最基础、也最重要的区分：

- **网络层**：提供**主机到主机**的逻辑通信
- **传输层**：提供**进程到进程**的逻辑通信

可以这样理解：

- IP 负责把数据送到“哪台主机”
- 传输层负责把数据交给“这台主机上的哪个进程”

::: tip 一句话区分

- 网络层：找到目标**主机**
- 传输层：找到目标主机上的目标**进程**

很多题目表面上都在讲“传数据”，但考点其实是在区分这两层各自解决的问题。

:::

### 互联网中的传输层协议

互联网中最主要的两个传输层协议是：

- **UDP**：无连接、尽力而为
- **TCP**：面向连接、可靠传输

应用程序会根据自身需求选择使用哪一种服务。

## 3.2 复用与解复用

### 什么是复用

发送端主机上往往同时运行多个应用进程，例如浏览器、聊天工具、下载器。它们都可能同时需要发数据。

**复用（Multiplexing）** 指的是：

- 发送端把来自多个套接字的数据收集起来
- 加上传输层首部后交给网络层发送

简单说，就是“多路应用数据汇入传输层”。

### 什么是解复用

接收端收到 IP 数据报后，会从中取出传输层报文段，再根据首部信息交给正确的套接字。

这就是**解复用（Demultiplexing）**：

- 根据报文段中的地址和端口信息
- 把数据交给正确的应用进程

简单说，就是“把到达的数据分发给正确程序”。

### 为什么端口号很重要

一台主机可以同时运行很多网络进程，所以只靠 IP 地址无法区分“该把数据交给哪个程序”。

因此传输层必须使用**端口号（Port Number）**。

一个传输层报文段通常至少包含：

- 源端口号
- 目的端口号

这样接收方才能把数据交给正确的套接字。

### UDP 的解复用

对于 UDP，解复用通常主要依据：

- **目的 IP 地址**
- **目的端口号**

因此：

- 如果多个 UDP 报文段的目的端口号相同
- 即使它们来自不同源 IP、不同源端口
- 也可能被交给同一个 UDP 套接字

### TCP 的解复用

TCP 比 UDP 更复杂，因为同一台服务器常常同时维护很多连接。

TCP 套接字通常由**四元组**标识：

- 源 IP 地址
- 源端口号
- 目的 IP 地址
- 目的端口号

因此，即使很多客户端都在连接服务器的同一个端口，例如 `80`，服务器也能把这些 TCP 报文段解复用到不同连接中。

## 3.3 UDP：无连接传输

### UDP 是什么

**UDP（User Datagram Protocol，用户数据报协议）** 是一种无连接的传输层协议。

它的特点非常直接：

- 发送前不建立连接
- 每个报文段独立处理
- 不保证可靠送达
- 不保证按序到达
- 不提供流量控制
- 不提供拥塞控制

### UDP 为什么还存在

很多初学者都会问：既然 UDP 这么“弱”，为什么还要有它？

原因是，UDP 提供了一种非常轻量的服务：

- 头部小，开销低
- 无需连接建立
- 不会因为可靠性机制带来额外时延
- 应用可以自己决定如何处理丢包、乱序和重传

所以 UDP 并不是“落后的 TCP”，而是为特定需求设计的另一类服务。

### UDP 的典型应用

适合 UDP 的场景通常有这些特点：

- 能容忍少量丢包
- 更看重实时性和低时延
- 或者应用本身就比较简单，没必要引入 TCP 的完整机制

常见例子：

- DNS
- 实时音视频
- VoIP
- 一些在线游戏
- SNMP 等简单查询控制协议

### UDP 首部

UDP 首部非常短，只有 8 字节，主要包括：

- 源端口号
- 目的端口号
- 长度
- 校验和

### UDP 校验和

UDP 使用校验和来检测传输过程中是否出现比特错误。

需要注意：

- 校验和主要是**检测错误**
- 不是**纠正错误**

如果接收方发现报文出错，通常不会尝试修复，而是根据实现策略丢弃或交给上层处理。

## 3.4 可靠数据传输原理

### 为什么要研究可靠传输

网络层 IP 提供的是“尽力而为”服务，它不保证：

- 一定送达
- 按序送达
- 数据完整无误

因此，如果应用或传输层需要“可靠”效果，就必须额外设计机制，在不可靠信道之上实现可靠传输。

所谓**可靠数据传输（Reliable Data Transfer, RDT）**，通常意味着：

- 不丢失
- 不重复
- 按序交付
- 交给上层的数据与发送时一致

### 从理想到现实：rdt 协议演进

教材通常通过 `rdt1.0 -> rdt2.x -> rdt3.0` 的演进来说明可靠传输机制是如何一步步加出来的。

### rdt1.0：信道完全可靠

在最理想的情况下：

- 信道不会出错
- 不会丢包

那就几乎不需要额外机制，发送方直接发，接收方直接收。

显然，这只是教学起点，现实网络远没有这么理想。

### rdt2.0：信道会出现比特错误

如果信道可能发生比特错误，就需要：

- **校验和**：发现报文是否损坏
- **ACK**：正确收到后确认
- **NAK**：收到错误后请求重传

这时发送方就不再是“发完就算”，而要等待接收方反馈。

### rdt2.0 的问题

如果 ACK 或 NAK 自己也损坏了怎么办？

发送方就会面临困境：

- 我到底该不该重传？
- 如果误判，可能造成重复交付

这说明，仅靠 ACK/NAK 还不够。

### rdt2.1 / rdt2.2：引入序号

为了解决确认报文出错带来的歧义，需要引入**序号（Sequence Number）**。

这样即使某个分组被重传，接收方也能根据序号判断：

- 这是新分组
- 还是之前已经收过的重复分组

其中：

- `rdt2.1` 仍使用 ACK/NAK
- `rdt2.2` 只使用 ACK，通过 ACK 的内容来表达“下一个期望收到什么”

这已经非常接近现代 TCP 中“累计确认”的思想了。

### rdt3.0：既可能出错，也可能丢包

如果信道不只是出错，还可能直接把分组丢掉，那么仅靠 ACK/NAK 还不够，因为：

- 分组可能丢了
- ACK 也可能丢了

发送方不能无限等待，因此必须加入：

- **定时器**
- **超时重传**

这就是 `rdt3.0` 的核心思想。

### 停止-等待协议

`rdt3.0` 的基本工作方式通常是**停止-等待（Stop-and-Wait）**：

- 发送方一次只发送一个分组
- 必须等这个分组被确认后，才能继续发下一个

它的优点是：

- 逻辑简单
- 正确性容易保证

缺点是：

- 链路利用率可能非常低

尤其在“高带宽、长 RTT”环境下，发送方大部分时间都在等待确认。

::: warning

停止-等待协议能保证正确，但不一定高效。它解决的是“能不能可靠传”，并没有很好解决“传得快不快”。

:::

### 为什么需要流水线

为了提高利用率，发送方不能总是“发一个等一个”，而应该允许多个未确认分组同时在路上飞行。

这就是 **流水线（Pipelining）** 思想：

- 在前面的分组还没确认时
- 继续发送后续分组

这样就能显著提升吞吐量。

### 两种经典流水线协议

可靠传输中，两种最经典的流水线协议是：

- **GBN（Go-Back-N，后退 N 帧）**
- **SR（Selective Repeat，选择重传）**

### GBN：后退 N 帧

GBN 的核心思想是：

- 发送方维护一个窗口，允许连续发送多个分组
- 接收方只接受按序到达的分组
- 使用**累计确认**

例如：

- `ACK(5)` 表示 `0~4` 或直到 `4` 的数据都已经按序收到，接下来期望的是 `5`

如果中间某个分组丢了：

- 接收方会继续重复确认最后一个按序收到的位置
- 发送方一旦超时，就把该分组及其后面所有未确认分组全部重传

GBN 的优点：

- 发送方和接收方逻辑相对简单

缺点：

- 可能重传很多其实已经到达过的数据，效率不高

### SR：选择重传

SR 比 GBN 更“精细”：

- 每个分组单独确认
- 每个分组可单独计时
- 接收方可以缓存失序到达的分组
- 只重传真正丢失的那个分组

SR 的优点：

- 重传更有针对性，效率更高

缺点：

- 发送方和接收方都更复杂
- 接收方需要缓存乱序分组

### SR 为什么要求窗口不能太大

SR 中，序号空间必须足够区分“旧分组”和“新分组”。

因此通常要求：

- **窗口大小不超过序号空间的一半**

否则就可能因为序号回绕而把旧数据误当成新数据。

## 3.5 TCP：面向连接的可靠传输

### TCP 的总体特征

**TCP（Transmission Control Protocol，传输控制协议）** 是互联网中最重要的传输层协议之一，主要特点包括：

- 面向连接
- 提供可靠传输
- 提供全双工通信
- 提供字节流服务
- 支持流量控制
- 支持拥塞控制

### TCP 为什么说是“字节流”

TCP 并不关心应用层消息边界，而是把数据看作一串连续的字节流。

因此：

- 应用写入多少次数据，不一定对应接收方读出多少次
- TCP 关注的是“字节序号”，而不是“应用消息编号”

### MSS 是什么

**MSS（Maximum Segment Size）** 表示一个 TCP 报文段中**数据部分**允许的最大长度。

它受链路 MTU 等因素影响。

可以简单记为：

- MTU 偏向链路层能装多大
- MSS 偏向 TCP 一次能装多少应用数据

### TCP 报文段结构

TCP 报文段首部中，最重要的字段包括：

- **源端口号 / 目的端口号**
- **序号 `seq`**
- **确认号 `ack`**
- **接收窗口 `rwnd`**
- **校验和**
- **首部长度**
- **标志位**

常见标志位：

- `ACK`：确认号字段有效
- `SYN`：建立连接
- `FIN`：关闭连接
- `RST`：复位连接
- `URG`、`PSH`：在现代基础学习中通常不是重点

### TCP 的序号与确认号

这是 TCP 最核心的概念之一。

#### 序号

TCP 的序号按**字节**计数，而不是按报文段计数。

- 一个报文段的序号，表示这个报文段中**第一个字节**的编号

#### 确认号

TCP 的确认号表示：

- **接收方期望收到的下一个字节序号**

因此 TCP 使用的是**累计确认**思想。

例如：

- 如果接收方返回 `ACK = 100`
- 表示它已经按序收到了 `0~99` 这些字节
- 下一个希望收到的是字节 `100`

### RTT 估计与超时重传

TCP 需要设置合理的超时时间：

- 太短：会发生很多不必要重传
- 太长：真正丢包时反应太慢

因此 TCP 不会使用固定超时值，而是动态估计 RTT。

### SampleRTT 与 EstimatedRTT

- **SampleRTT**：某次报文段发送到收到对应 ACK 的测量时间
- **EstimatedRTT**：对多个 SampleRTT 做平滑后的估计值

常见估计公式为：

$$
\text{EstimatedRTT} = (1-\alpha)\times \text{EstimatedRTT} + \alpha\times \text{SampleRTT}, \quad \alpha = 0.125
$$

这是一种指数加权移动平均（EWMA）思想：

- 最近样本影响更大
- 过去样本影响逐渐减弱

### 超时时间

TCP 的超时值通常不是简单等于 `EstimatedRTT`，而是还要加上安全裕度。

常见写法为：

$$
\text{TimeoutInterval} = \text{EstimatedRTT} + 4\times \text{DevRTT}
$$

其中 `DevRTT` 用于刻画 RTT 波动程度。

若发生超时，TCP 通常还会采用**指数退避**，把超时值适当加大。

### TCP 的可靠传输机制

TCP 在 IP 的不可靠服务之上，综合使用多种机制实现可靠传输：

- 校验和
- 序号
- 确认号
- 定时器
- 超时重传
- 累计确认
- 流水线发送

### TCP 发送方的基本思路

TCP 发送方通常维护两个关键概念：

- **已经发送但还没确认的数据**
- **还可以继续发送的数据范围**

发送方会：

1. 从应用接收数据
2. 加上序号形成报文段
3. 发送给 IP 层
4. 为最早未确认的数据启动计时器
5. 收到 ACK 后滑动发送窗口
6. 超时后重传

::: info 超时重传与快重传

这两者都在做“丢了就补发”，但触发依据不同：

- **超时重传**：等了太久还没确认
- **快重传**：连续收到 `3` 个冗余 `ACK`，提前推测某段丢失

可以把快重传理解成：还没等正式“超时报警”，就先根据迹象抢先处理。

:::

### TCP 的 ACK 特性

TCP 的确认机制有几个初学者容易忽视的点：

- TCP 通常使用**累计确认**
- 接收方可能使用**延迟确认**
- 乱序到达时，接收方可能发送**冗余 ACK**

### 快重传

如果发送方连续收到 **3 个冗余 ACK**，通常说明：

- 某个报文段很可能丢了
- 但后面的某些报文段已经到达接收方

这意味着网络并没有完全断掉，所以 TCP 不必死等超时，而是可以：

- **立即重传最早未确认的报文段**

这就是**快重传（Fast Retransmit）**。

它能显著减少丢包恢复时延。

### 流量控制

流量控制解决的问题是：

**发送方发得太快，接收方来不及收，怎么办？**

这和拥塞控制完全不同。流量控制关心的是**接收方缓存是否吃得消**。

### rwnd：接收窗口

TCP 使用**接收窗口 `rwnd`** 来做流量控制。

- 接收方会在 TCP 首部中通告自己的空闲缓存大小
- 发送方必须保证未确认的在途数据量不要超过这个值

因此可以理解为：

- `rwnd` 小：接收方比较“忙”，你慢点发
- `rwnd` 大：接收方还有较多缓存空间，可以继续发

### 连接管理

TCP 是面向连接的，所以通信前要建立连接，通信结束后还要拆除连接。

建立连接的目的不仅是“打招呼”，还包括：

- 确认双方都愿意通信
- 协商连接参数
- 初始化双方状态变量和缓存

### 为什么两次握手不够

如果只做两次握手，可能出现旧的、延迟到达的连接请求引发错误连接，造成**半打开连接**等问题。

因此 TCP 采用三次握手，而不是两次。

### TCP 三次握手

建立连接的过程如下：

1. **客户端 -> 服务器：`SYN`**
   - 发送连接请求
   - 指定初始序号 `x`

2. **服务器 -> 客户端：`SYN + ACK`**
   - 同意建立连接
   - 指定自己的初始序号 `y`
   - 确认客户端序号，即 `ACK = x + 1`

3. **客户端 -> 服务器：`ACK`**
   - 确认服务器的序号，即 `ACK = y + 1`

完成后三次握手，两端都知道：

- 对方收到了自己的建立连接意图
- 双方都已经准备好收发数据

### TCP 四次挥手

TCP 连接关闭时，双方的发送方向是**分别关闭**的，所以通常需要四次挥手。

典型过程如下：

1. 主动关闭方发送 `FIN`
2. 被动关闭方返回 `ACK`
3. 被动关闭方准备好后发送自己的 `FIN`
4. 主动关闭方返回最后的 `ACK`

### TIME_WAIT 与 2MSL

主动关闭方在发送最后一个 ACK 后，不会立刻彻底消失，而会进入 `TIME_WAIT` 状态并等待一段时间，常写为 `2MSL`。

这样做主要有两个原因：

- 防止最后一个 ACK 丢失时，对方重发 `FIN` 却得不到响应
- 让旧连接中的残留报文在网络中自然消失，避免影响后续新连接

## 3.6 拥塞控制原理

### 什么是拥塞

拥塞不是“某一台接收方太慢”，而是：

- 网络中进入的分组太多
- 路由器来不及转发
- 导致排队、时延增大、丢包增加

所以：

- **流量控制**面向接收方
- **拥塞控制**面向整个网络

### 拥塞的表现

拥塞最典型的表现有两个：

- **排队时延越来越大**
- **分组被丢弃**

### 拥塞的代价

教材中通常强调拥塞带来的几类代价：

- 大量排队导致高时延
- 丢包后重传，带来额外工作量
- 超时过早会触发冗余重传，进一步浪费带宽
- 某些分组即使最终被丢弃，也已经消耗了上游链路资源

这说明拥塞不仅让“速度变慢”，还会让网络资源被无效消耗。

### 拥塞控制的两种思路

#### 端到端拥塞控制

端系统不依赖网络层明确告知是否拥塞，而是通过现象去推断，例如：

- 丢包
- RTT 增大
- 冗余 ACK

TCP 就属于这种典型思路。

#### 网络辅助拥塞控制

网络中的路由器或交换机显式反馈拥塞信息给发送方，例如：

- 某些显式拥塞位
- 显式速率建议

这类方法在一些专用网络中出现过，在互联网教材中更多作为“对比思路”理解。

## 3.7 TCP 的拥塞控制

### 两个窗口不要混

学习 TCP 时，很容易把 `rwnd` 和 `cwnd` 混在一起：

- **`rwnd`**：接收窗口，解决流量控制问题
- **`cwnd`**：拥塞窗口，解决拥塞控制问题

实际发送方能发送的数据量，通常受两者共同限制：

$$
\text{发送窗口} = \min(\text{rwnd}, \text{cwnd})
$$

::: warning `rwnd` 和 `cwnd` 不要混

这是 TCP 中最经典的易错点：

- `rwnd` 小，说明接收方缓存紧张
- `cwnd` 小，说明发送方判断网络可能拥塞

一个是“对方吃不消”，一个是“网络扛不住”，含义完全不同。

:::

### cwnd 的基本含义

`cwnd` 是发送方根据自己对网络状况的判断，给自己设定的发送上限。

通俗地说：

- 网络看起来畅通，就逐渐放大发送量
- 网络看起来拥塞，就收缩发送量

### AIMD：加性增、乘性减

TCP 拥塞控制的核心思想是 **AIMD（Additive Increase, Multiplicative Decrease）**：

- **加性增加**：网络看起来没问题时，缓慢增加窗口
- **乘性减小**：一旦检测到拥塞，快速把窗口减小

这种机制会形成经典的“锯齿状”窗口变化曲线。

### 慢启动

TCP 刚开始发送时，并不知道当前网络能承受多少负载，因此采用**慢启动（Slow Start）**。

名称叫“慢启动”，但增长其实并不慢：

- 初始 `cwnd` 较小，常记为 `1 MSS`
- 每收到一个 ACK，`cwnd` 就增加
- 从整体上看，大致表现为每个 RTT 近似翻倍，属于**指数增长**

慢启动的目标是：

- 尽快探测出一个大致可用的发送速率

::: info 为什么叫“慢启动”却增长很快

这个名字是相对于“不要一开始就把网络打满”而言的，不是说它增长速度真的慢。

实际上在慢启动阶段：

- 每收到一个 `ACK` 都会增大窗口
- 从整体看接近指数增长

所以它的本质是“谨慎起步，但探测很积极”。

:::

### ssthresh：慢启动阈值

TCP 使用 `ssthresh` 来区分两个阶段：

- `cwnd < ssthresh`：通常处于慢启动阶段
- `cwnd >= ssthresh`：转入拥塞避免阶段

### 拥塞避免

进入拥塞避免后，TCP 不再像慢启动那样快速增长，而是改为更稳妥的线性增长：

- 大致每经过一个 RTT，`cwnd` 增加 `1 MSS`

这是为了更平滑地探测可用带宽，避免过快冲进拥塞。

### 拥塞是如何被检测到的

TCP 通常把以下事件视为拥塞信号：

- **超时**
- **3 个冗余 ACK**

但这两种信号的严重程度通常不一样。

### 超时：更严重的拥塞信号

如果发生超时，TCP 通常认为：

- 丢包比较严重
- 当前发送速率可能明显过高

典型处理方式是：

- `ssthresh = cwnd / 2`
- `cwnd = 1 MSS`
- 重新进入慢启动

### 3 个冗余 ACK：网络还没完全堵死

如果收到 3 个冗余 ACK，说明：

- 某个报文段可能丢失了
- 但后续一些报文段还在继续到达

因此 TCP 通常不必像超时那样把 `cwnd` 降到最小。

### Tahoe 与 Reno 的区别

教材中常用两个经典版本说明拥塞后的处理差异：

#### TCP Tahoe

无论是超时还是 3 个冗余 ACK，都采取较激进的回退：

- `ssthresh = cwnd / 2`
- `cwnd = 1 MSS`
- 回到慢启动

#### TCP Reno

当出现 **3 个冗余 ACK** 时，Reno 采用更温和的策略：

- 先快速重传丢失段
- 将 `ssthresh` 设为原窗口的一半
- `cwnd` 不必降到 `1 MSS`
- 进入 **快恢复（Fast Recovery）**

这使得 Reno 在很多情况下恢复更快。

### 快恢复

快恢复的核心思想是：

- 既然还能收到冗余 ACK，说明网络并没有完全停摆
- 那就没必要从极小窗口重新慢启动

因此它会：

- 在减小窗口后继续维持较快恢复速度
- 收到新的确认后再回到拥塞避免阶段

### TCP 平均吞吐量

如果忽略慢启动等细节，并假设 TCP 总有数据可发，则一个经典近似结论是：

$$
\text{Throughput} \approx 0.75 \times \frac{W}{RTT}
$$

其中：

- `W` 是发生丢包时的窗口大小
- `RTT` 是往返时延

这个式子表达的核心直觉是：

- 窗口越大，吞吐量越高
- RTT 越大，吞吐量越低

### TCP 公平性

TCP 希望在多个连接共享同一瓶颈链路时，达到一种相对公平的结果：

- 若 `K` 个 TCP 会话共享带宽为 `R` 的瓶颈链路
- 理想情况下，每个连接平均获得 `R/K`

这与 AIMD 机制密切相关：

- 大家都逐步增加
- 一旦拥塞一起缩小

但要注意，现实中公平并不总是完美成立。例如：

- UDP 流不一定遵守 TCP 的拥塞控制礼让
- 某些应用可以开启多个并行 TCP 连接，从而获得更多带宽份额

## 本章小结

学完传输层后，建议你至少能回答这些问题：

1. 传输层和网络层分别解决什么问题
2. 复用和解复用为什么需要端口号
3. 为什么 UDP 明明不可靠却仍然很重要
4. 可靠数据传输为什么要经历 `校验和 -> ACK/NAK -> 序号 -> 超时重传 -> 流水线`
5. GBN 和 SR 的核心差别是什么
6. TCP 为什么按字节编号，而不是按报文段编号
7. 三次握手为什么不能少成两次
8. 流量控制和拥塞控制到底有什么本质区别
9. `rwnd`、`cwnd`、`ssthresh` 分别表示什么
10. 为什么 3 个冗余 ACK 和超时会触发不同的拥塞应对策略

::: details 参考答案

1. 网络层解决主机到主机的转发与路由，

   传输层解决进程到进程的端到端通信。

2. 复用和解复用需要端口号，是因为一台主机上会同时运行多个网络进程，必须区分数据该交给谁。
3. UDP 虽然不可靠，但它开销小、时延低、机制简单，适合实时应用和希望自行控制传输逻辑的应用。
4. 可靠数据传输之所以一步步演进，是因为真实信道可能出现比特错误、确认报文出错、丢包、重复等问题，因此必须依次加入校验和、确认、序号、超时重传和流水线机制。
5. `GBN` 使用累计确认、接收方通常丢弃失序分组，超时后会重传一段连续分组；

   `SR` 对每个分组单独确认和重传，接收方可缓存失序分组。

6. TCP 按字节编号，是因为它提供的是字节流服务，而不是“按报文段编号”的消息服务。
7. 三次握手不能少成两次，是为了确认双方都具备收发能力，并避免旧的失效连接请求导致错误连接。
8. 流量控制关注“接收方是否来得及接收”，

   拥塞控制关注“网络是否已经过载”。

9. `rwnd` 是接收窗口，反映接收方还能接收多少；

   `cwnd` 是拥塞窗口，反映发送方认为网络当前能承受多少；

   `ssthresh` 是慢启动阈值，用于区分慢启动和拥塞避免阶段。

10. 3 个冗余 ACK 说明网络还能继续送到后续数据，通常意味着局部丢失；

    超时则说明情况更严重，因此 TCP 会对两者采取不同强度的回退策略。

:::

如果这些问题已经能顺着逻辑讲清楚，说明你对传输层已经建立起比较扎实的基础框架。
