# 第 3 章 传输层

传输层的核心是**进程间的逻辑通信**，在网络层的基础上，为应用提供端到端的传输服务。

## 3.1 传输层基础

### 复用与解复用

- **复用**：发送端将多个进程的报文段封装到同一个 IP 数据报中。
- **解复用**：接收端根据端口号，将报文段交付给正确的进程。
  - UDP：仅用目的端口号解复用。
  - TCP：用 **四元组（源 IP、源端口、目的 IP、目的端口）** 解复用，支持同一端口的多个连接。

## 3.2 UDP：用户数据报协议

UDP 是无连接的不可靠传输协议，开销极小。

- **首部**：源端口、目的端口、长度、校验和，共 8 字节。
- **校验和**：检测传输中的比特错误，采用 16 位补码和。
- **特点**：无连接、无可靠传输、无拥塞 / 流量控制，适合实时应用（流媒体、VoIP）、DNS、SNMP 等。

## 3.3 可靠数据传输原理

可靠数据传输是在不可靠信道上，实现数据的有序、无丢失、无重复交付，是传输层、链路层的核心机制。

### 协议演进

1. **rdt1.0**：完全可靠信道，直接收发数据，无需额外处理。
2. **rdt2.0**：信道有比特错误，通过校验和检测错误，ACK/NAK 反馈，出错重传。
3. **rdt2.1**：处理 ACK/NAK 出错，引入序号，冗余分组丢弃，停止等待。
4. **rdt3.0**：信道有丢包，引入超时计时器，超时重传。

::: warning 停止等待的性能问题

**停止等待协议**每次发一个分组等 ACK，链路利用率极低，例如 1Gbps 链路、30ms RTT，利用率仅 0.00027。

:::

### 流水线技术

允许发送方同时发送多个未确认的分组，提升链路利用率，有两种经典协议：

1. **后退 N 帧（`GBN`）**
   - **滑动窗口**：最多 `N` 个未确认的分组。
   - **累积确认**：`ACK(n)` 确认 `n` 及之前的所有分组。
   - **超时重传**：超时后重传所有未确认的分组，接收方丢弃失序分组。

2. **选择重传（`SR`）**
   - 每个分组单独确认，单独计时。
   - 接收方缓存失序分组，只重传丢失的分组。
   - 窗口大小必须不超过序号空间的一半，避免序号混淆。

## 3.4 TCP：传输控制协议

TCP 是面向连接的可靠传输协议，是互联网的核心传输协议。

### TCP 报文段结构

- **序号**：报文段第一个字节的编号，按字节计数，初始随机。
- **确认号**：期望收到的下一个字节的序号，累积确认。
- **接收窗口**：接收方的空闲缓存大小，用于流量控制。
- **标志位**：`URG`（紧急数据）、`ACK`（确认有效）、`PSH`（推送）、`RST`/`SYN`/`FIN`（连接管理）。

### 往返时间与超时

- 动态估计 `RTT`：采用指数加权移动平均（`EWMA`）：

  $$\text{EstimatedRTT} = (1-\alpha)\times \text{EstimatedRTT} + \alpha\times \text{SampleRTT},\quad \alpha=0.125$$

- 超时时间：$\text{TimeoutInterval} = \text{EstimatedRTT} + 4\times \text{DevRTT}$，超时后加倍。

### TCP 的可靠传输

TCP 在 [rdt](#_3-3-可靠数据传输原理) 的基础上，实现了流水线的可靠传输：

- **超时重传**：超时后重传最早的未确认分组。
- **快重传**：收到 3 个冗余 ACK 后，立即重传丢失的分组，不用等超时，减少重传时延。

### 流量控制

TCP 通过接收窗口（`rwnd`）实现流量控制，避免接收方缓冲区溢出：

- 接收方在报文段中通告 `rwnd`（空闲缓存大小）。
- 发送方限制未确认的数据量不超过 `rwnd`。

### 连接管理：三次握手与四次挥手

TCP 是面向连接的，通信前需要建立连接，结束后关闭连接。

1. **三次握手（建立连接）**
   - 客户端发 `SYN` 报文，序号 `x`。
   - 服务器发 `SYN + ACK` 报文，序号 `y`，`ACK = x + 1`。
   - 客户端发 `ACK` 报文，`ACK = y + 1`，连接建立。

   ::: tip 为什么三次？
   避免失效的连接请求报文导致半打开连接。
   :::

2. **四次挥手（关闭连接）**
   ::: info `MSL`（Maximum Segment Lifetime）
   表示报文段在网络中的最大生存时间，`2MSL` 是它的两倍，对应主动关闭方在 `TIME_WAIT` 状态的等待时长。
   :::
   - 主动方发 `FIN` 报文，关闭发送方向。
   - 被动方发 `ACK` 确认。
   - 被动方发 `FIN` 报文，关闭自己的发送方向。
   - 主动方发 `ACK` 确认，等待 `2MSL` 后关闭，以便在最后一个 `ACK` 丢失时重发，并让旧报文从网络中消失。

## 3.5 拥塞控制

拥塞是指网络中分组过多，导致路由器排队、丢包，TCP 通过端到端的拥塞控制，避免网络拥塞崩溃。

- **拥塞窗口 `cwnd`**：发送方根据网络拥塞情况维护的窗口大小，用于限制自己一次能发送多少未确认数据。
- **[接收窗口 `rwnd`](#流量控制)** 用于流量控制，防止接收方缓冲区溢出；`cwnd` 用于拥塞控制，防止网络过载。
- 发送方实际可发送的数据量通常受两者共同限制，可理解为 `发送窗口 = min(rwnd, cwnd)`。
- **`MSS`（Maximum Segment Size）**：TCP 报文段中数据部分的最大长度；`1MSS` 就表示一个最大报文段的数据量。

### 核心机制：AIMD（加性增、乘性减）

- **加性增**：拥塞避免阶段，每个 RTT 拥塞窗口 `cwnd` 增加 `1MSS`，线性探测可用带宽。
- **乘性减**：发生丢包后，`cwnd` 减半，降低发送速率。
- **呈现锯齿状的窗口变化**：`cwnd` 增加 `1MSS`，`cwnd` 减半，重复。

### 慢启动

连接初始阶段，`cwnd` 从 `1MSS` 开始，每收到一个 `ACK` 就加 1，指数增长，直到达到 `ssthresh`（慢启动阈值），或者发生丢包。

### 丢包的处理

- 超时：`cwnd` 重置为 1，`ssthresh` 设为原来 `cwnd` 的一半，进入慢启动。

- 3 个冗余 `ACK`：`cwnd` 减半，`ssthresh` 设为原来 `cwnd` 的一半，进入快恢复，线性增长。

### TCP 吞吐量

TCP 的平均吞吐量为：$\text{Throughput} = 0.75 \times \frac{\text{W}}{\text{RTT}}$，$\text{W}$ 是丢包时的窗口大小。
